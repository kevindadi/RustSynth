\begin{abstract}
% Rust 库越来越多地暴露有状态、类型导向的 API，其有效测试需要生成长且类型明确的调用序列，而不是修改非结构化的字节输入。然而，Rust 的仿射所有权、借用排他性和生命周期约束构成了编译障碍：由于使用后移动、&mut 下的非法别名或未满足的泛型边界，简单生成的序列经常无法通过类型检查，从而阻止任何运行时反馈。本文通过将 API 调用建模为类型化值和权限上的资源流问题，解决了 Rust crate 的签名级测试生成问题。
Rust libraries increasingly expose stateful, type-directed APIs whose effective testing requires generating long, well-typed call sequences rather than mutating unstructured byte inputs. However, Rust’s affine ownership, borrow exclusivity, and lifetime constraints impose a compilation barrier: naively generated sequences frequently fail to type-check due to move-after-use, illegal aliasing under \texttt{\&mut}, or unsatisfied generic bounds, preventing any runtime feedback. This paper addresses signature-level test generation for Rust crates by modeling API invocations as a resource-flow problem over typed values and permissions.

% 我们提出了一种下推着色 Petri 网，它由 crate 提取的函数签名构建而成。其中，Place 表示 Rust 类型表达式和相关的编译时义务（包括泛型占位符、trait 满足性、关联类型和字段投影），而 Transition 则捕获签名驱动的函数调用和一小部分结构转换（例如，字段的借用、借用终止、投影和重新借用）。令牌流代表所有权转移和借用引起的权限变更，而下推栈则强制执行正确嵌套的借用范围，将生成的引用绑定到其来源，并在借用结束时恢复权限。为了支持泛型较多的 API，我们定义了一个基于偏序关系的满足性关系，该关系在转换启用期间检查具体实例是否满足 trait 边界和关联类型约束。复合类型通过形式化的投影和（重新）借用转换来处理，从而在生成的框架代码层面保留 Rust 的别名限制。在此模型之上，我们将生成过程建模为一个可达性问题，并显式地设定了转换（以及可选的签名项）的覆盖率目标，同时开发了一种有界可达性分析方法，用于在保护约束下搜索高覆盖率的触发轨迹。
We propose a Pushdown Colored Petri Net that is constructed from a crate’s extracted function signatures. Places denote Rust type expressions and related compile-time obligations (including generic placeholders, trait satisfaction, associated types, and field projections), while transitions capture signature-driven function calls and a small set of structural transformations (e.g., borrowing, borrow termination, projection and reborrowing of fields). Token flow represents ownership transfer and borrow-induced capability changes, and a pushdown stack enforces properly nested borrow extents, binding produced references to their provenance and restoring permissions when borrows end. To support generic-heavy APIs, we define a partial-order-based satisfaction relation that checks whether a concrete instantiation meets trait bounds and associated-type constraints during transition enabling. Composite types are handled via formal projection and (re)borrow transitions that preserve Rust’s aliasing restrictions at the level of generated harness code. On top of the model, we formulate generation as a reachability problem with explicit coverage objectives over transitions (and, optionally, over signature items) and develop a bounded reachability analysis that searches for high-coverage firing traces under guard constraints.

% 每个轨迹都被确定性地转换为 Rust 框架，插入 \texttt{drop} 或作用域边界来实现栈弹出事件，从而生成在模型保守的借用规则下可编译的程序。我们实现了一个原型工具，该工具与 Rust 工具链集成，用于提取签名，并生成与标准覆盖率引导模糊测试工作流程兼容的工具。
Each trace is deterministically translated into a Rust harness, inserting \texttt{drop} or scope boundaries to realize stack-pop events, yielding programs that are compilable under the model’s conservative borrowing discipline. We implement a prototype tool that integrates with the Rust toolchain for signature extraction and produces harnesses compatible with standard coverage-guided fuzzing workflows.

\end{abstract}

\section{Introduction}
% Rust + 测试用例生成/模糊测试来写动机、挑战与贡献，先讲 Rust 特性与挑战，再讲现有方法不足，最后提出一个“模型（Petri 网）+ 工程实现 + 评估/验证 + 工具集成”的贡献闭环。
% Rust 已成为构建可靠系统软件和性能关键型库的主流选择。它兼具底层性能和丰富的类型系统，消除了多种内存安全漏洞，同时仍能对资源进行细粒度控制。因此，越来越多的安全性和正确性关键功能以可重用 Rust crate 的形式交付。对于这类库，自动化测试生成，尤其是覆盖率引导的模糊测试，是发现传统单元测试无法捕获的 panic、逻辑错误和意外边界情况的必备技术。
Rust has become a mainstream choice for building reliable systems software and performance-critical libraries. Its combination of low-level performance and a rich type system eliminates broad classes of memory-safety bugs, while still exposing fine-grained control over resources. As a result, an increasing amount of security and correctness-critical functionality is delivered as reusable Rust crates. For such libraries, automated test generation especially coverage-guided fuzzing, is an indispensable technique for discovering panics, logic bugs, and unexpected corner cases that escape conventional unit tests.

% 引出核心矛盾：要想覆盖更高，往往需要“长序列 + 状态交互 + 结构化输入”；而 Rust 的所有权/借用/生命周期/泛型让“生成可编译、可执行”的序列本身就很难。
% 尽管 Rust 库取得了成功，但为其生成有效的测试用例仍然比许多其他语言要困难得多。现代模糊测试工具擅长修改字节数组并驱动单个入口函数，但库 API 通常是有状态的且类型导向的：覆盖率通常依赖于构建一个交互对象网络，并调用一系列遵循资源协议的 API 调用。在 Rust 中，这些协议不仅通过运行时检查来强制执行，还通过编译时关于所有权、借用和生命周期的规则来强制执行。因此，很大一部分简单生成的调用序列永远不会执行：它们由于移动错误、别名冲突或未满足的泛型约束而编译失败。这种编译障碍会转化为覆盖率障碍。
Despite its success, effective test-case generation for Rust libraries remains substantially harder than for many other languages. Modern fuzzers excel at mutating byte arrays and driving a single entry function, but library APIs are often stateful and type-directed:
coverage typically depends on constructing a web of interacting objects and invoking a long sequence of API calls that respects resource protocols. In Rust, such protocols are enforced not only by runtime checks, but also by compile-time rules on ownership, borrowing, and lifetimes. Consequently, a large fraction of na\"{i}vely generated call sequences never reach execution: they fail to compile due to move errors, aliasing violations, or unsatisfied generic constraints. This compilation barrier turns into a coverage barrier.

\paragraph{A motivating example.}
Consider a simplified crate API in Listing~\ref{lst:api}, which resembles many real-world libraries (parsers, databases, protocol stacks, etc.).
It features (i) composite types (\texttt{Result}, \texttt{Option}, \texttt{Vec}), (ii) lifetime-carrying handles that borrow from a parent object, and (iii) a generic entry point constrained by trait bounds.


\begin{lstlisting}[language=Rust,caption={A small Rust API exhibiting ownership transfer, borrowing, lifetimes, and generics.},label={lst:api}]
pub struct Store { /* ... */ }
pub struct Tx<'s> { store: &'s mut Store /* ... */ }
pub trait KeyLike {
    fn as_bytes(&self) -> &[u8];
}

impl Store {
    pub fn open(path: &str) -> Result<Self, ()> { /* ... */ }
    // Borrowing & lifetime: a transaction borrows the store mutably.
    pub fn begin<'s>(&'s mut self) -> Tx<'s> { /* ... */ }
}

impl<'s> Tx<'s> {
    pub fn put<K: KeyLike>(&mut self, k: K, v: Vec<u8>) { /* ... */ }
    // Immutable borrow from &self, returns a reference tied to 's.
    pub fn get<'a, K: KeyLike>(&'a self, k: K) -> Option<&'a [u8]> { /* ... */ }
    // Ownership consumption: committing consumes the Tx value.
    pub fn commit(self) -> Result<(), ()> { /* ... */ }
}
\end{lstlisting}

% 忽略 Rust 所有权规则的测试生成器可能会尝试以 Rust 中无效的方式交错调用。例如，在创建了一个持有 &mut store 的事务 tx 之后，生成器可能会尝试再次调用 store.begin()，或者直接修改 store，而此时 tx 仍然处于活动状态——这两种操作都会被借用检查器拒绝。类似地，commit(self) 会消耗 tx，因此任何后续对 tx 的使用都会触发移动错误。最后，诸如 K: KeyLike 之类的泛型参数要求生成器生成满足边界的具体类型，而不是任意字节块。这些问题并非仅仅是语法问题。在 Rust 中，类型正确性编码了 API 协议的很大一部分。因此，Rust 库能否实现高覆盖率取决于能否生成可编译且类型正确的调用序列，这些调用序列必须遵循所有权、借用、生命周期和泛型约束。
A test generator that ignores Rust's ownership discipline might attempt to interleave calls in ways that are invalid in Rust.
For instance, after creating a transaction \texttt{tx} that holds \texttt{\&mut store}, a generator may try to call \texttt{store.begin()} again, or mutate \texttt{store} directly, while \texttt{tx} is still live---both are rejected by the borrow checker. Similarly, \texttt{commit(self)} consumes \texttt{tx}, so any subsequent use of \texttt{tx} triggers a move error. Finally, generic parameters such as \texttt{K: KeyLike} require the generator to produce a concrete type that satisfies the bound, rather than an arbitrary byte blob. These issues are not merely syntactic. In Rust, well-typedness encodes a large portion of the API protocol. Therefore, achieving high coverage for Rust libraries hinges on the ability to generate compilable and type-correct call sequences that respect ownership, borrowing, lifetimes, and generic constraints.

% 自动化测试技术种类繁多，包括覆盖率引导模糊测试、基于属性的测试、基于语法的生成、符号执行和基于模型的测试。然而，当应用于 Rust 库 API 时，它们通常会面临以下至少一个限制：
A wide range of automated testing techniques exist, including coverage-guided fuzzing, property-based testing, grammar-based generation, symbolic execution, and model-based testing. However, when applied to Rust library APIs, they commonly face at least one of the following limitations:
\begin{enumerate}
  % 面向字节的输入，没有类型化的对象构造 
  \item \textbf{Byte-oriented inputs without typed object construction.}
  % 许多模糊测试设置都基于字节数组，并依赖于手动编写的框架将字节转换为结构化对象。对于有状态的 API，该框架还必须管理长调用序列、中间值和资源协议。这种手动操作容易出错，并且常常成为主要的瓶颈。
  Many fuzzing setups operate on byte arrays and rely on a hand-written harness to translate bytes into structured objects.
  For stateful APIs, the harness must additionally manage long call sequences, intermediate values, and resource protocols.
  This manual effort is error-prone and often becomes the dominant bottleneck.

  % 对 Rust 特有的资源约束支持有限
  \item \textbf{Limited support for Rust-specific resource constraints.}
  % 将值视为可自由复制的生成器难以应对 Rust 的仿射所有权：值可以被移动、部分移动、不可变借用或可变借用，并且具有严格的别名规则。此外，Rust 的生命周期通常是推断出来的（例如，非词法生命周期），这意味着借用操作的确切结束时间并没有在源代码中显式地写明——然而，这对于判断后续修改何时合法至关重要。
  Generators that treat values as freely copyable struggle with Rust's affine ownership: values can be moved, partially moved, borrowed immutably, or borrowed mutably, with strict aliasing rules.
  Moreover, Rust lifetimes are typically inferred (e.g., non-lexical lifetimes), which means that the precise end of a borrow is not explicitly written in the source code---yet it is essential for deciding when subsequent mutations become legal.

  % 对复合型和泛型 API 的处理能力较弱
  \item \textbf{Weak handling of composite and generic APIs.}
  % 真正的 crate 会公开基于嵌套代数数据类型、切片、智能指针和丰富的 trait 绑定泛型的 API。如果没有一个规范的泛型实例化机制（并确保候选类型满足 trait 的约束），生成器要么 (i) 会避免使用这类函数，要么 (ii) 会生成类型错误且永远无法编译的代码。
  Real crates expose APIs over nested algebraic data types, slices, smart pointers, and rich trait-bound generics. Without a principled mechanism for instantiating generics (and ensuring that candidate types satisfy trait bounds), generators either (i) avoid such functions, or (ii) produce ill-typed code that never compiles.

  % 内部状态过度建模时，可扩展性较差
  \item \textbf{Poor scalability when internal state is over-modeled.}
  % 一些分析驱动的方法试图对内部程序状态进行建模来驱动代码生成。虽然这些方法功能强大，但对于大型 crate 来说可能成本高昂且脆弱，而且当目标是生成 API 级测试而不是完整的程序验证时，这些方法通常是不必要的。
  Some analysis-driven approaches attempt to model the state of the internal program to drive generation. Although powerful, these approaches can be expensive and brittle for large crates and are often unnecessary when the goal is the generation of API-level tests rather than the verification of the full program.
\end{enumerate}

% 指出“现在 Petri 网生成代码序列不满足 Rust 的特性”：传统 Petri 网多用于协议/并发/事件序列，token 通常缺少“类型 + 借用 + 生命周期 + move”的语义。
% 另一项平行研究探索了用于生成事件序列和测试协议的 Petri 网（及相关自动机）。然而，经典的 Petri 网编码通常是无类型化的，或者只是弱类型化的：标记代表抽象资源，而不是具有所有权和借用能力的 Rust 值。因此，直接使用 Petri 网生成序列往往会在三个方面违反 Rust 的编译时规则：(i) 它无法强制执行亲和性（移动的标记不得被重用），(ii) 它无法表达借用排他性（不允许可变别名，不允许在未完成的不可变借用下进行修改）以及 (iii) 它缺乏作用域和生命周期限定的可用性概念（引用的生命周期不得超过被借用的所有者，并且借用权限最终必须归还）。此外，通用约束需要类似求解器的步骤（选择可接受的实例化），而传统的 Petri 网模型无法提供这一步骤。
A parallel line of work explores Petri nets (and related automata) for generating event sequences and testing protocols. Classic Petri-net encodings, however, are typically \emph{untyped} or only weakly typed: tokens represent abstract resources, but not Rust values with ownership and borrow capabilities. As a result, straightforward Petri-net-driven sequence generation tends to violate Rust's compile-time rules in three recurring ways: (i) it fails to enforce \emph{affinity} (a moved token must not be reused), (ii) it cannot express \emph{borrow exclusivity} (no mutable aliasing, no mutation under an outstanding immutable borrow), and (iii) it lacks a notion of \emph{scope- and lifetime-bounded availability} (references must not outlive the borrowed owner, and borrow permissions must eventually return). Furthermore, generic constraints require a solver-like step (selecting admissible instantiations), which traditional Petri-net models do not provide.

\paragraph{Our perspective: signature-driven generation without heavy internal modeling.}
% 本文针对一个常见且实际应用重要的场景：给定一个 Rust 代码库，我们的目标是生成调用其公共（或选定）函数和方法的测试框架。关键在于，我们假设编译器工具链提供了函数签名——包括输入输出类型、引用可变性、生命周期和泛型边界。我们并不试图精确地建模函数体的内部控制流或内存状态。相反，我们将每个函数视为一个原子转换，其类型签名指定了值（及其功能）如何在调用边界之间流动。这种签名级别的处理方式使得该方法轻量且可扩展，同时仍然能够捕捉到影响生成测试编译成功的约束条件。
This paper targets a common and practically important setting:
we are given a Rust codebase and we aim to generate test harnesses that invoke its public (or selected) functions and methods.
Crucially, we assume that function signatures including input/output types, reference mutability, lifetimes, and generic bounds are available from the cargo doc. We do not attempt to precisely model the internal control flow or memory state of function bodies.
Instead, we treat each function as an atomic transition whose type signature specifies how values (and their capabilities) can flow across call boundaries. This signature-level stance makes the approach lightweight and scalable, while still capturing the constraints that dominate compilation success for generated tests.

% 核心技术：下推着色 Petri 网。要把“类型、所有权、借用、生命周期、泛型”映射到 Petri 网的 token/颜色/栈结构上。
\paragraph{Approach overview: Pushdown Colored Petri Nets for Rust API fuzzing.}
We introduce a new model for Rust test-case generation based on \emph{Pushdown Colored Petri Nets} (PCPNs).
At a high level:
\begin{itemize}
  \item Places represent pools of available \emph{typed values} together with \emph{capabilities} describing how they may be used (owned, immutably borrowed, mutably borrowed).
  \item Transitions represent \emph{function calls} extracted from the target crate's signatures.
  A transition consumes tokens matching the function's input types/capabilities and produces tokens matching its output types/capabilities.
  \item Colors encode Rust types (including composites) and capability metadata (e.g., \texttt{owned T}, \texttt{\&T}, \texttt{\&mut T}).
  \item A pushdown stack models \emph{nested scopes} and \emph{borrow extents}.
  When a reference is created, the model pushes the borrowed provenance; when the reference becomes dead (e.g., explicitly \texttt{drop}'d or out of scope), the model pops the corresponding frame and returns permissions to the original owner.
\end{itemize}

% 这种下推结构对于 Rust 风格的生命周期推理至关重要：借用关系可以嵌套（重新借用），而较长的 API 序列可能会反复创建和释放借用。有限状态抽象通常要么过于保守（阻塞合法程序），要么过于宽松（生成类型错误的程序）。PCPN 提供了一种紧凑、可组合的方式来表示可能无界的、正确嵌套的借用模式。
This pushdown structure is essential for Rust-style lifetime reasoning: borrow relationships can nest (reborrowing), and long API sequences may create and discharge borrows repeatedly. A finite-state abstraction often either becomes overly conservative (blocking legal programs) or overly permissive (generating ill-typed programs).
PCPNs provide a compact, compositional way to represent potentially unbounded, properly nested borrow patterns.

\paragraph{Generics via a satisfiability partial order.}
To support generic APIs, we equip the model with a \emph{type satisfaction} relation written as a partial order $\preceq$.
Intuitively, $A \preceq \mathcal{R}$ means that the concrete type $A$ satisfies the generic requirement $\mathcal{R}$ (e.g., a set of trait bounds, lifetime constraints, and where-clauses).
During transition firing, generic parameters are instantiated by selecting concrete types that are minimal (or otherwise prioritized) under $\preceq$ while remaining consistent with the surrounding token colors.
This design cleanly separates:
(i) the Petri-net execution semantics for sequencing, and
(ii) the constraint reasoning needed to produce admissible generic instantiations.
In practice, the partial order enables both sound pruning (reject impossible instantiations) and guided exploration (prefer simpler types early, diversify later).

\paragraph{From model to executable fuzz harnesses.}
% 给一个“生成出来的 harness 长什么样”的代码例子，并点出我们不需要建模函数体内部状态。
From a PCPN execution trace, we synthesize a Rust harness that composes API calls and feeds the resulting program to a standard coverage-guided fuzzer.
Listing~\ref{lst:harness} illustrates a typical generated skeleton for the API in Listing~\ref{lst:api}.
The harness is compilable by construction: every use respects move semantics, borrow exclusivity, and lifetime boundaries.

\begin{lstlisting}[language=Rust,caption={A compilable, signature-driven harness sketch generated from a PCPN trace.},label={lst:harness}]
fn harness(bytes: &[u8]) {
    // Construct base values (from bytes or small generators).
    let path = "tmp.db"; // tmp.db from Constant
    let mut store = match Store::open(path) { Ok(s) => s, Err(_) => return };
    // Begin a transaction: tx borrows store mutably.
    let mut tx = store.begin();
    // Choose a concrete key type K that satisfies K: KeyLike.
    // Here we pick Vec<u8> (assume the crate provides impl KeyLike for Vec<u8>).
    let k: Vec<u8> = bytes.to_vec();
    // Use tx while it is live; store cannot be used mutably here.
    tx.put(k.clone(), bytes.to_vec());
    let _v = tx.get(k);
    // Explicitly end the borrow extent and consume tx.
    let _ = tx.commit();
    // After commit, permissions return: store is usable again.
    let _tx2 = store.begin();
}
\end{lstlisting}

\noindent
Importantly, the generator does not need to model what \texttt{Store::open}, \texttt{Tx::put}, or \texttt{Tx::get} do internally.
% 区别于 PCG，我们不追踪“函数体内每个程序点”的能力变化；只在“函数边界”做资源流动建模，足以让生成序列可编译并能跑起来，从而提升覆盖率探索效率。
The signature alone determines how ownership and borrow capabilities move across call boundaries, which is exactly the information required to produce executable tests.

\section{Background}
\label{sec:background}
% 本章给出最小但足够的背景：Rust 的所有权/借用/生命周期（理解“为什么生成的序列很难编译”），
% 以及 Petri 网（理解“我们用什么模型来生成序列”）。内容保持简短、面向后续模型定义。

\subsection{Rust Ownership, Borrowing, and Lifetimes}
\label{sec:background:rust}
% 这一小节只讲与“生成可编译的 API 调用序列”直接相关的 Rust 规则：
% ownership（移动/析构）、borrowing（& / &mut 的互斥）、lifetimes（引用不能超过被借用者）。
% Rust 主要通过所有权和借用机制来确保内存安全。每个非 `\Copy` 值在同一时间都只有一个所有者；将值传递给另一个变量（或按值传递）通常会转移所有权。当所有者超出作用域时，该值将被丢弃，其资源将被释放。
Rust enforces memory safety primarily through a static discipline of ownership and borrowing. Every non-Copy value has exactly one owner at a time; transferring a value to another variable typically moves ownership. When the owner goes out of scope, the value is dropped and its resources are released.

% 借用允许临时访问值，而无需转移所有权。Rust 区分：(i) 共享借用 `\texttt{\&T}`，允许只读访问，并且可以任意别名；以及(ii) 可变借用 `\texttt{\&mut T}`，允许修改，但必须是互斥的。核心规则通常可以概括为“别名异或可变性”：在任何时刻，要么存在多个共享借用，要么存在一个可变借用。
Borrowing allows temporary access to a value without transferring ownership.
Rust distinguishes:
(i) shared borrows \texttt{\&T}, which permit read-only access and can be aliased arbitrarily, and
(ii) mutable borrows \texttt{\&mut T}, which permit mutation but must be exclusive.
The central rule is often summarized as \emph{aliasing XOR mutability}:
at any point, either many shared borrows exist, or exactly one mutable borrow exists.

% 生命周期是一个编译时区域，用于限制引用的使用时长。引用的生命周期不能超过它所引用的值，借用权限会在借用结束时归还给所有者。在现代 Rust 中，借用范围通常是推断出来的（例如，通过非词法生命周期），这意味着借用的确切结束点不一定能从语法表面看出，但对于判断后续修改是否合法仍然至关重要。
A lifetime is a compile-time region that bounds how long a reference may be used. A reference cannot outlive the value it refers to, and borrow permissions return to the owner when the borrow ends.
In modern Rust, borrow extents are often inferred (e.g., via non-lexical lifetimes), meaning the precise end point of a borrow is not necessarily obvious from surface syntax, but is still crucial for deciding whether later mutations are legal.

% 对于测试用例生成，这些规则对调用序列施加了约束：生成器必须跟踪一个值是否仍然属于某个所有者、是否已被移动、当前是否处于共享借用状态或是否处于可变借用状态，并且必须确保引用的生命周期不会超过其来源。
For test-case generation, these rules impose constraints on call sequences: a generator must track whether a value is still owned, has been moved, is currently shared-borrowed, or is mutably borrowed, and it must ensure that references do not outlive their provenance.

\subsection{Generics and Trait Bounds in Signatures}
\label{sec:background:generics}
% Rust API 通常是泛型的。函数可以量化类型参数，并使用 trait 边界（可能在 `where` 子句中）对其进行约束。例如，签名 `fn f<T: Trait>(x: T) -> U` 将 `T` 的可实例化限制为实现了 `Trait` 的类型。从自动化测试生成的角度来看，这意味着生成一个函数调用需要两个步骤：(1) 为泛型参数选择具体的实例化，以及(2) 提供类型与实例化签名匹配的值。如果步骤 (1) 失败，则生成的程序将无法通过类型检查，无论运行时行为如何。
Rust APIs are frequently generic.
A function may quantify over type parameters and constrain them with trait bounds possibly in \texttt{where} clauses.
For example, a signature \texttt{fn f<T: Trait>(x: T) -> U} restricts admissible instantiations of \texttt{T} to types implementing \texttt{Trait}.
From the perspective of automated test generation, this means that producing a call is a two-step task:
(1) choose concrete instantiations for generic parameters, and
(2) supply values whose types match the instantiated signature.
If step (1) fails, the generated program will not type-check, regardless of runtime behavior.

\section{Method: Pushdown Colored Petri Nets for Rust API Test Generation}
\label{sec:method}
% 1) 形式化定义下推着色 Petri 网（PCPN）；
% 2) 形式化给出 Rust（以函数签名为中心）到 PCPN 的映射；
% 3) 覆盖 move/borrow/mut/指针、生命周期（栈绑定）、泛型偏序、复合类型字段(重)借用、Trait/关联类型等。

\subsection{Problem Setting}
\label{sec:method:setting}
% 设\(\mathcal{C}\)为一个 Rust 包。我们可以从编译器前端提取出一个有限的可调用项集合\(\mathcal{F}\)（包含自由函数与方法），并附带它们的签名信息，具体包括：(i) 输入 / 输出类型；(ii) 引用可变性；(iii) 若存在显式生命周期参数，则包含该参数及其生存期约束；(iv) 带有 trait 约束与关联类型约束的泛型参数。
Let $\mathcal{C}$ be a Rust crate. We can extract, from the compiler front-end, a finite set of \emph{callable items}
$\mathcal{F}$ (free functions and methods) together with their signatures, including (i) input/output types, (ii) reference mutability, (iii) explicit lifetime parameters and outlives constraints when present, and (iv) generic parameters with trait bounds and associated-type constraints.

% 我们的目标是生成可编译的 Rust测试桩，这类测试桩能够构造带类型的值，并执行针对\(\mathcal{F}\)中各项的调用序列。我们刻意避免对函数体的内部状态进行建模；相反，我们将每一次调用视为一种原子变换，其编译期合法性由签名层面的所有权、借用、生命周期与泛型约束共同决定。
Our goal is to generate compilable Rust \emph{harnesses} that construct typed values and execute sequences of calls to items in $\mathcal{F}$. We deliberately avoid modeling internal states of function bodies; instead, each call is treated as an atomic transformation whose compile-time validity is determined by the signature-level ownership/borrowing/lifetime/generic constraints.

\subsection{Petri Net Preliminaries}
\label{sec:method:pn}
\begin{definition}[Net and Petri Net]
\label{def:pn}
A net is a $3$-tuple $N=(P,T,F)$, where $P$ is a finite set of places, $T$ is a finite set of transitions,
$F\subseteq (P\times T) \cup (T\times P)$ is a finite set of arcs, and $P\cap T=\emptyset$.
A net can be viewed as a directed bipartite graph where places are represented by circles and transitions by rectangles.
The pre-set and post-set of a node $x \in\{P \cup T\}$ are defined as
$^{\bullet}x=\{y\in P\cup T\mid (y,x)\in F\}$ and $x^{\bullet}=\{y\in P\cup T\mid (x,y)\in F\}$, respectively.
In this paper, it is assumed that the weight of each directed arc is $1$.

A marking of $N=(P,T,F)$ is a mapping $M:P\rightarrow \mathbb{N}$ where $M(p)$ is the number of tokens in place $p$.
A net $N$ with an initial marking $M_{0}$ is called a Petri Net and denoted as $(N,M_{0})$.
Transition $t$ is enabled at $M$ if $\forall p\in {}^{\bullet}t: M(p)>0$, denoted as $M[t\rangle$.
A marking $M$ is a deadlock marking if any transition is disabled at $M$.
Firing an enabled transition $t$ yields a new marking $M^{\prime}$:
$M^{\prime}(p)=M(p)-1$ if $p\in {}^{\bullet}t\backslash t^{\bullet}$;
$M^{\prime}(p)=M(p)+1$ if $p\in t^{\bullet}\backslash {}^{\bullet}t$;
and otherwise $M^{\prime}(p)=M(p)$.
This is denoted as $M[t\rangle M^{\prime}$.
A marking $M_{k}$ is reachable from $M$ if $M_{k}=M$ or there exists a non-empty transition sequence
$\sigma=t_{1}t_{2}\dots t_{k}$ such that
$M[t_{1}\rangle M_{1}[t_{2}\rangle \dots M_{k-1}[t_{k}\rangle M_{k}$ (abbreviated as $M[\sigma \rangle M_{k}$).
All markings reachable from $M_{0}$ in a Petri net is written as $\mathcal{M}$.
This paper only considers bounded Petri nets, i.e.,
$\exists k\in \mathbb{N},\forall{M}\in \mathcal{M}:M(p)\leq k$.
\end{definition}

\subsection{Pushdown Colored Petri Nets (PCPN)}
\label{sec:method:pcpn}
% PCPN 是对上面 PN 的扩展：
% (i) token 带颜色(携带变量 id、泛型/生命周期实例化等)；
% (ii) 引入全局栈用于建模 borrow/lifetime 的嵌套与归还；
% (iii) 为了表达同一 place 多重输入（同型多个参数），使用标准的“加权弧/平行弧”扩展。

\paragraph{Type expressions.}
% 设GTy为从代码集合中提取的单态 Rust 类型集合。设TVar为泛型类型变量集合，LVar为生命周期变量集合（如'a 'static）。设Sigma{C}表示从C中提取的签名 / 类型环境（注：存储函数签名、结构体定义等全局类型信息，如fn f<'a>(x: &'a i32) -> &'a i32）。
Let $\mathsf{GTy}$ be the set of monomorphic Rust types extracted from $\mathcal{C}$. Let $\mathsf{TVar}$ be a set of generic type variables and $\mathsf{LVar}$ a set of lifetime variables. Let $\Sigma(\mathcal{C})$ denote the extracted signature/type environment.

We use the following type-expression language:
\begin{align*}
\tau \in \mathsf{Ty} ::=~&
g \mid \alpha \mid \tau_0\langle \tau_1,\dots,\tau_n\rangle \mid (\tau_1,\dots,\tau_n) \mid [\tau] \mid \\
& \mathsf{Ref}^q_{\ell}(\tau) \mid \mathsf{Assoc}(\tau,\mathsf{Tr},A) \mid \mathsf{Field}(\tau,f) \mid \mathsf{Impl}(\tau,\mathsf{Tr}),
\end{align*}
where $g\in\mathsf{GTy}$, $\alpha\in\mathsf{TVar}$, $q\in\{\mathsf{shr},\mathsf{mut}\}$,
$\ell\in\mathsf{LVar}$, $\mathsf{Tr}$ ranges over traits, $A$ over associated type names, and $f$ over fields/projection paths.

% 解决 Rust 隐式借用的生命周期追踪问题 —— 比如闭包捕获变量时生成的隐式借用、编译器自动插入的临时借用，这些借用没有用户显式声明的生命周期，需要一个专用变量承载其运行时区域标签，确保借用规则的完整性。
Let $\ell_{\bullet}\in \mathsf{LVar}$ be a reserved lifetime variable that does not appear in extracted signatures.
All internally created references are typed with $\ell_{\bullet}$ and carry the actual region label via $\lambda(\ell_{\bullet})$.

\paragraph{Capability modes and places.}
Let $\mathsf{Cap}=\{\mathsf{own},\mathsf{frz},\mathsf{blk},\mathsf{val}\}$.
We define the place set:
\[
P ~=~ \{\, p_{\kappa,\tau} \mid \kappa\in\mathsf{Cap},~ \tau\in\mathsf{Ty} \,\}.
\]
Intuitively, $p_{\mathsf{own},\tau}$ stores owned values;
% p_frz存储被共享借用冻结的所有权值（如 let r = &x后，x被冻结，不能再取可变引用）
$p_{\mathsf{frz},\tau}$ stores owned values frozen by shared borrows;
% p_blk存储被可变借用阻塞的所有权值（如 let mr = &mut x后，x被阻塞，不能再取任何引用）
$p_{\mathsf{blk},\tau}$ stores owned values blocked by a mutable borrow;
% p_val存储非所有权值（如引用&x、trait 实现证据等，本身不拥有数据）
$p_{\mathsf{val},\tau}$ stores non-owning values (references, witnesses, etc.).

\paragraph{Colors and colored markings.}
% 设V 为可数无穷的值标识符集合（即新鲜的 Rust 变量名，如x y temp_1）；
Let $\mathbb{V}$ be a countably infinite set of value identifiers (fresh Rust variable names),
% L为可数无穷的 区域标签集合（运行时真实的生命周期区域，如函数调用栈帧对应的区域）。
and $\mathbb{L}$ a countably infinite set of region labels.
% 设Theta为有限类型替换集合：theta是从泛型变量TVar到单态类型GTy的部分映射（如\theta(T)=i32，表示将泛型T实例化为i32）；设Lambda为有限生命周期替换集合：lambda是从生命周期变量LVar到区域标签L的部分映射（如lambda('a)=L_1，表示将生命周期'a绑定到运行时区域L_1。
Let $\Theta$ be finite type substitutions $\theta:\mathsf{TVar}\rightharpoonup \mathsf{GTy}$,
and $\Lambda$ finite lifetime substitutions $\lambda:\mathsf{LVar}\rightharpoonup \mathbb{L}$.

A token color is
\begin{equation*}
    c ~=~ \langle v,\theta,\lambda\rangle \in \mathsf{Col} := \mathbb{V}\times\Theta\times\Lambda.
\end{equation*}

% 通过 带颜色的令牌 追踪每个变量的动态信息（谁、类型是什么、生命周期绑定到哪个运行时区域），解决 Petri 网 无法区分不同变量 的问题 —— 比如两个i32变量，通过颜色中的v和lambda区分，确保借用检查时能精准匹配 变量 - 生命周期 - 区域。
A colored marking is
\begin{equation*}
    M: P \to \mathbb{N}^{(\mathsf{Col})},
\end{equation*}
i.e., each place carries a finite multiset of colored tokens.

\paragraph{Pushdown stack.}
Let $\Gamma$ be a stack alphabet:
\begin{equation*}
    \gamma ::= \mathsf{Freeze}(v)\mid \mathsf{Shr}(v_o,v_r,\ell)\mid \mathsf{Mut}(v_o,v_r,\ell),
\end{equation*}
where $\ell\in\mathbb{L}$ is a region label.
A configuration is $\mathsf{Cfg}=\langle M,S\rangle$ where $S\in \Gamma^*$ (top at the left).

\paragraph{Weighted arcs (or parallel arcs).}
We model arc multiplicities by a weight function
\begin{equation*}
    W: (P\times T)\cup (T\times P)\to \mathbb{N}
\end{equation*}
with finite support, and let $F=\{(x,y)\mid W(x,y)>0\}$.
For a transition $t$, we write $W(p,t)$ and $W(t,p)$ for input/output multiplicities.

\paragraph{Certificates, guards, and firing semantics.}
A \emph{choice} for $t$ at marking $M$ is a multiset-valued function
\begin{equation*}
    \chi:P\to \mathbb{N}^{(\mathsf{Col})}
    \quad\text{s.t.}\quad
    \forall p\in P:\ \chi(p)\subseteq M(p)\ \wedge\ |\chi(p)| = W(p,t).
\end{equation*}

% 实例化信息绑定
Let $\iota$ range over instantiations, containing at least:
\begin{equation*}
    \iota=(\theta^{*},\lambda^{*},\nu,\mu,\dots)
\end{equation*}
where $\theta^{*}\in\Theta$ and $\lambda^{*}\in\Lambda$ are the unified substitutions, and $\nu$ and $\mu$ provide any fresh value ids / fresh region labels needed by $\mathsf{Out}$. Other resolved information such as associated-type resolution may also be stored.

Each transition $t$ carries:
\begin{align*}
    & \mathsf{Guard}(t)(\langle M,S\rangle,\chi,\iota),\qquad \\
    & \mathsf{Out}(t)(\langle M,S\rangle,\chi,\iota):P\to \mathbb{N}^{(\mathsf{Col})},\qquad \\
    & \mathsf{Act}(t)\in\{\epsilon,\mathsf{push}(\bar\gamma),\mathsf{pop}(\bar\gamma)\}.
\end{align*}

\begin{definition}[Enabledness and firing]
\label{def:pcpn:firing}
Let $\mathsf{Cfg}=\langle M,S\rangle$. A transition $t$ is enabled at $\mathsf{Cfg}$ if there exist
$\chi$ and $\iota$ such that $\mathsf{Guard}(t)(\mathsf{Cfg},\chi,\iota)$ holds and the pop-pattern (if any) matches the stack top.
A firing rule is $\kappa=\langle t,\chi,\iota\rangle$.

Given $\kappa$, firing yields $\mathsf{Cfg}'=\langle M',S'\rangle$ where:
\begin{itemize}
\item Consume: $M^{-}(p)=M(p)\ominus \chi(p)$ for all $p$;
\item Produce: let $\pi=\mathsf{Out}(t)(\mathsf{Cfg},\chi,\iota)$ with $|\pi(p)|=W(t,p)$, then $M'=M^{-}\oplus \pi$;
\item Stack: $S'$ is obtained by applying $\mathsf{Act}(t)$ (push/pop/$\epsilon$) to $S$.
\end{itemize}
\end{definition}

\subsection{Constructing a Rust PCPN from $\mathcal{C}$}
\label{sec:method:construct}
We construct a crate-specific PCPN
\begin{equation*}
    \mathcal{N}_{\mathcal{C}}=(P,T,W,\mathsf{Col},\Gamma,M_0,\mathsf{Guard},\mathsf{Out},\mathsf{Act}),
\end{equation*}
where $M_0$ seeds a finite set of constructors (literals, empty containers, etc.).

\paragraph{Copy predicate.}
% Rust 的 Copy 是 trait：用环境判定 $\Sigma(\mathcal{C})\models \tau:\mathsf{Copy}$
We write $\mathsf{IsCopy}(\tau)$ iff $\Sigma(\mathcal{C})\models \tau:\mathsf{Copy}$.

\paragraph{Signature-induced transitions.}
For each callable item $f\in\mathcal{F}$ with signature
\begin{equation*}
    \mathsf{sig}(f)=\langle \bar\alpha;\bar\ell;\ \tau_1,\dots,\tau_n \to \tau_o;\ \rho\rangle,
\end{equation*}
we add a transition $t_f$ whose arc weights are determined by the parameter passing mode:
\begin{itemize}
\item[1.] If parameter $i$ is by-value $\tau_i$, then $W(p_{\mathsf{own},\tau_i},t_f){+}{=}1$.
      Moreover, if $\mathsf{IsCopy}(\tau_i)$, then $W(t_f,p_{\mathsf{own},\tau_i}){+}{=}1$ (self-loop, no move).
\item[2.] If parameter $i$ is shared reference $\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)$, then \\
      $W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)},t_f){+}{=}1$ and also
      $W(t_f,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)}){+}{=}1$ since $\&T$ is Copy.
\item[3.] If parameter $i$ is mutable reference $\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)$, then \\
      $W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)},t_f){+}{=}1$. No self-loop by default: $\&mut \ T$ is not Copy.
\end{itemize}
If $\tau_o\neq ()$, we add one output arc:
\begin{equation*}
    W(t_f,\ p_{\kappa_o,\tau_o})=1
\end{equation*}
where $\kappa_o=\mathsf{own}$ if the return is owned, and $\kappa_o=\mathsf{val}$ if it returns a reference.

The guard $\mathsf{Guard}(t_f)$ solves:
(i) type/lifetime unification across arguments, (ii) trait/assoc constraints in $\rho$,
and (iii) capability feasibility (e.g., no direct use of blocked owners).
$\mathsf{Out}(t_f)$ allocates fresh ids for returned values and reproduces Copy-arguments on self-loop arcs.
$\mathsf{Act}(t_f)=\epsilon$ unless the signature semantically begins/ends a borrow-carrying owned value.

\subsection{Ownership and Copy as Token Flow}
\label{sec:method:ownership}

% 为了简洁，定义 color 的投影函数
Let $\pi_V(\langle v,\theta,\lambda\rangle)=v$, $\pi_\Theta(\cdot)=\theta$, and $\pi_\Lambda(\cdot)=\lambda$.
For a singleton multiset $\{c\}$, write $\mathsf{one}(\{c\})=c$.

\paragraph{(1) Move for non-Copy owned values.}
For each type $\tau$, add a transition $\mathsf{Move}_\tau$ with:
\[
W(p_{\mathsf{own},\tau},\mathsf{Move}_\tau)=1,
\]
and no outputs. Define:
\begin{align*}
\mathsf{Guard}(\mathsf{Move}_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ \neg \mathsf{IsCopy}(\tau),\\
\mathsf{Out}(\mathsf{Move}_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ \emptyset,\\
\mathsf{Act}(\mathsf{Move}_\tau)
&~=~ \epsilon.
\end{align*}

\paragraph{(2) By-value use for Copy owned values (self-loop).}
For each $\tau$, add $\mathsf{CopyUse}_\tau$:
\begin{equation*}
    W(p_{\mathsf{own},\tau},\mathsf{CopyUse}_\tau)=1,\qquad
    W(\mathsf{CopyUse}_\tau,p_{\mathsf{own},\tau})=1.
\end{equation*}
Let $c=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$. Define:
\begin{align*}
\mathsf{Guard}(\mathsf{CopyUse}_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ \mathsf{IsCopy}(\tau),\\
\mathsf{Out}(\mathsf{CopyUse}_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau}\mapsto \{c\}\,],\\
\mathsf{Act}(\mathsf{CopyUse}_\tau)
&~=~ \epsilon.
\end{align*}
% 解释：Copy 类型按值使用不 move，token 仍留在 own place。

\paragraph{Copy assignment to create a fresh variable id.}
% 用于建模 `let y = x;`，会增长 token 数，可能破坏 bounded；需要用预算/上界控制。
Fix a per-type copy budget $B_{\mathsf{copy}}(\tau)\in\mathbb{N}$.
Add $\mathsf{DupCopy}_\tau$:
\begin{equation*}
    W(p_{\mathsf{own},\tau},\mathsf{DupCopy}_\tau)=1,\qquad
    W(\mathsf{DupCopy}_\tau,p_{\mathsf{own},\tau})=2.
\end{equation*}
Let $c=\langle v,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$ and let $\nu(\mathsf{new})=v'$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(\mathsf{DupCopy}_\tau)(\langle M,S\rangle,\chi,\iota)
&\equiv \mathsf{IsCopy}(\tau) \\
&\quad \wedge\ |M(p_{\mathsf{own},\tau})| < B_{\mathsf{copy}}(\tau)\\ &\quad \wedge\ v'\notin \mathsf{Ids}(M,S),\\
\mathsf{Out}(\mathsf{DupCopy}_\tau)(\langle M,S\rangle,\chi,\iota)
&= [\,p_{\mathsf{own},\tau}\mapsto \{\langle v,\theta,\lambda\rangle,\ \langle v',\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(\mathsf{DupCopy}_\tau)
&= \epsilon.
\end{align*}
% 其中 $\mathsf{Ids}(M,S)$ 表示 marking 和 stack 中出现过的 value ids。

\paragraph{(3) Drop.}
For each place $p_{\kappa,\tau}$, add $\mathsf{Drop}_{\kappa,\tau}$:
\[
W(p_{\kappa,\tau},\mathsf{Drop}_{\kappa,\tau})=1,
\]
and no outputs. Define:
\begin{align*}
\mathsf{Guard}(\mathsf{Drop}_{\kappa,\tau})(\langle M,S\rangle,\chi,\iota) &~\equiv~ \top,\\
\mathsf{Out}(\mathsf{Drop}_{\kappa,\tau})(\langle M,S\rangle,\chi,\iota) &~=~ \emptyset,\\
\mathsf{Act}(\mathsf{Drop}_{\kappa,\tau}) &~=~ \epsilon.
\end{align*}

\paragraph{(4) First shared borrow from an owned value.}
Add $\mathsf{BorrowShrOwn}_\tau$ as $T_\tau$: $W(p_{\mathsf{own},\tau},T_\tau)$=$W(T_\tau,p_{\mathsf{frz},\tau})$=$W(T_\tau,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)})$=1.

Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$ and let $\nu(\mathsf{ref})=v_r$, $\mu(\mathsf{lab})=\ell$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_r\notin \mathsf{Ids}(M,S)\ \wedge\ \ell\notin \mathsf{Labs}(M,S),\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{frz},\tau}\mapsto \{c_o\},\\
&\qquad\ \ p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)}\mapsto \{\langle v_r,\theta,\lambda[\ell_{\bullet}\mapsto \ell]\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{push}(\mathsf{Freeze}(v_o)\cdot \mathsf{Shr}(v_o,v_r,\ell)).
\end{align*}

\paragraph{(5) Additional shared borrow while already frozen.}
Add $\mathsf{BorrowShrFrz}_\tau$ as $T_\tau$: 
$W(p_{\mathsf{frz},\tau},T_\tau)$=$W(T_\tau,p_{\mathsf{frz},\tau})$=$W(T_\tau,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)})$=1.
Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{frz},\tau}))$ and let $\nu(\mathsf{ref})=v_r$, $\mu(\mathsf{lab})=\ell$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_r\notin \mathsf{Ids}(M,S)\ \wedge\ \ell\notin \mathsf{Labs}(M,S),\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{frz},\tau}\mapsto \{c_o\},\\
&\qquad\ \ p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)}\mapsto \{\langle v_r,\theta,\lambda[\ell_{\bullet}\mapsto \ell]\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{push}(\mathsf{Shr}(v_o,v_r,\ell)).
\end{align*}

\paragraph{(6) Mutable borrow from an owned value.}
Add $\mathsf{BorrowMut}_\tau$ as $T_\tau$: $W(p_{\mathsf{own},\tau},T_\tau)$=$W(T_\tau,p_{\mathsf{blk},\tau})$=$W(T_\tau,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau)})$=1.
Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$ and let $\nu(\mathsf{ref})=v_r$, $\mu(\mathsf{lab})=\ell$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_r\notin \mathsf{Ids}(M,S)\ \wedge\ \ell\notin \mathsf{Labs}(M,S),\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{blk},\tau}\mapsto \{c_o\},\\
&\qquad\ \ p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau)}\mapsto \{\langle v_r,\theta,\lambda[\ell_{\bullet}\mapsto \ell]\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{push}(\mathsf{Mut}(v_o,v_r,\ell)).
\end{align*}

\label{sec:method:lifetime}

\paragraph{(7) Ending a mutable borrow (restore owner).}
Add $\mathsf{EndMut}_\tau$ as $T_\tau$:
$W(p_{\mathsf{blk},\tau},T_\tau)$=$W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau)},T_\tau)$=$W(T_\tau,p_{\mathsf{own},\tau})$=1.

Let $c_o=\langle v_o,\theta_o,\lambda_o\rangle=\mathsf{one}(\chi(p_{\mathsf{blk},\tau}))$ and
$c_r=\langle v_r,\theta_r,\lambda_r\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau)}))$.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ S = \mathsf{Mut}(v_o,v_r,\ell)\cdot S' \\
&\quad \wedge\ \theta_o=\theta_r\ \wedge\ \lambda_o=\lambda_r,\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau}\mapsto \{\langle v_o,\theta_o,\lambda_o\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{pop}(\mathsf{Mut}(v_o,v_r,\ell)).
\end{align*}

\paragraph{(8) Ending a shared borrow (non-last).}
Add $\mathsf{EndShrKeep}_\tau$ as $T_\tau$:
$W(p_{\mathsf{frz},\tau},T_\tau)$=$W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)},T_\tau)$=$W(T_\tau,p_{\mathsf{frz},\tau})$=1.

Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{frz},\tau}))$ and
$c_r=\langle v_r,\theta',\lambda'\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)}))$.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ S=\mathsf{Shr}(v_o,v_r,\ell)\cdot S' \\
&\quad \wedge\ \theta=\theta' \ \wedge\ \lambda=\lambda' \\ 
&\quad \wedge \big(S'=\epsilon\ \vee\ \mathsf{top}(S')\neq \mathsf{Freeze}(v_o)\big),\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{frz},\tau}\mapsto \{\langle v_o,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{pop}(\mathsf{Shr}(v_o,v_r,\ell)).
\end{align*}

\paragraph{(9) Ending a shared borrow (last, restore owner).}
Add $\mathsf{EndShrLast}_\tau$ as $T_\tau$:
$W(p_{\mathsf{frz},\tau},T_\tau)$=$W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)},T_\tau)$=$W(T_\tau,p_{\mathsf{own},\tau})$=1.

Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{frz},\tau}))$ and
$c_r=\langle v_r,\theta',\lambda'\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)}))$.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ S=\mathsf{Shr}(v_o,v_r,\ell)\cdot \mathsf{Freeze}(v_o)\cdot S' \\
&\quad \wedge\ \theta=\theta' \ \wedge\ \lambda=\lambda',\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau}\mapsto \{\langle v_o,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{pop}(\mathsf{Shr}(v_o,v_r,\ell)\cdot \mathsf{Freeze}(v_o)).
\end{align*}

\subsection{Composite Types and Field (Re)Borrowing}
\label{sec:method:composite}

\paragraph{(10) Field projection by move (conservative).}
% 保守：move 出字段视为消耗整个 owner（避免 partial move 复杂性）
Let $\tau$ be a struct type with field $f:\tau_f$.
Add $\mathsf{ProjMove}_{\tau,f}$ as $T_{\tau,f}$: 
\begin{equation*}
W(p_{\mathsf{own},\tau},T_{\tau,f})=W(T_{\tau,f},p_{\mathsf{own},\tau_f})=1.
\end{equation*}

Let $c=\langle v,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$ and let $\nu(\mathsf{fld})=v_f$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_f\notin \mathsf{Ids}(M,S),\\
\mathsf{Out}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau_f}\mapsto \{\langle v_f,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_{\tau,f})
&~=~ \epsilon.
\end{align*}

\paragraph{(11) Field reborrow from a shared reference (Copy, keep parent).}
% &S -> &S.f：shared ref 是 Copy，因此 parent token 不应被消耗
Add $\mathsf{ProjShr}_{\tau,f}$ as $T_\tau$:
\begin{equation*}
    W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)},T_{\tau,f})=W(T_{\tau,f},p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)})=W(T_{\tau,f},p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau_f)})=1.
\end{equation*}

Let $c_p=\langle v_p,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)}))$
and let $\nu(\mathsf{child})=v_c$ be fresh. Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_c\notin \mathsf{Ids}(M,S),\\
\mathsf{Out}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)}\mapsto \{c_p\},\\
&\qquad\ \ p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau_f)}\mapsto \{\langle v_c,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_{\tau,f})
&~=~ \epsilon.
\end{align*}

\paragraph{(12) Field reborrow from a mutable reference (reborrowing, block parent ref).}
% &mut S -> &mut S.f：子借用活着时父 &mut 不可用，因此消耗 parent ref token，并用栈 frame 恢复
Add $\mathsf{ProjMut}_{\tau,f}$ as $T_\tau$: 
\begin{equation*}
    W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)},T_{\tau,f})=W(T_{\tau,f},p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau_f)})=1.
\end{equation*}

Let $c_p=\langle v_p,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)}))$
and let $\nu(\mathsf{child})=v_c$, $\mu(\mathsf{lab})=\ell_c$ be fresh. Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_c\notin \mathsf{Ids}(M,S)\ \wedge\ \ell_c\notin \mathsf{Labs}(M,S),\\
\mathsf{Out}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau_f)}\mapsto
      \{\langle v_c,\theta,\lambda[\ell_{\bullet}\mapsto \ell_c]\rangle\}\,],\\
\mathsf{Act}(T_{\tau,f})
&~=~ \mathsf{push}(\mathsf{Mut}(v_p,v_c,\ell_c)).
\end{align*}

\paragraph{(13) Ending a mutable reborrow (restore parent ref).}
% pop Mut(parent,child,ell_c) 并恢复 parent &mut token
Add $\mathsf{EndProjMut}_{\tau,f}$ as $T_\tau$:
\begin{equation*}
    W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau_f)},\mathsf{EndProjMut}_{\tau,f})=W(\mathsf{EndProjMut}_{\tau,f},p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)})=1.
\end{equation*}

Let $c_c=\langle v_c,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau_f)}))$.
Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ S=\mathsf{Mut}(v_p,v_c,\ell_c)\cdot S',\\
\mathsf{Out}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)}\mapsto \{\langle v_p,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_{\tau,f})
&~=~ \mathsf{pop}(\mathsf{Mut}(v_p,v_c,\ell_c)).
\end{align*}

\subsection{Soundness Intuition}
\label{sec:method:soundness}
% 根据构造方式，Nc中的任意变迁序列均对应一个满足以下条件的测试套：(i) 被移动的自有值不会被重复使用（它们会从库所\(p_{\mathsf{own},\tau}\)中被消耗）；(ii) 可变性别名的产生被严格禁止（在\(\mathsf{Mut}\)帧的作用下，自有值的所属权会被转移至库所\(p_{\mathsf{blk},\tau}\)）；(iii) 共享别名场景下的突变操作被严格禁止（在\(\mathsf{Freeze}/\mathsf{Shr}\)帧的作用下，自有值的所属权会被托管至库所\(p_{\mathsf{frz},\tau}\)）；(iv) 生命周期具备合法的嵌套结构，并通过栈弹出变迁完成生命周期的终结。泛型实例化过程由可满足性关系\(\Vdash\)关联类型解析机制共同约束。因此，在该保守式结构化借用规则的约束下，所生成的测试套能够被 Rust 编译器正常接受。
By construction, any transition sequence in $\mathcal{N}_{\mathcal{C}}$ corresponds to a harness in which:
(i) moved owned values are not reused (they are consumed from $p_{\mathsf{own},\tau}$),
(ii) mutable aliasing is prevented (owners are moved to $p_{\mathsf{blk},\tau}$ under $\mathsf{Mut}$ frames),
(iii) mutation under shared aliasing is prevented (owners reside in $p_{\mathsf{frz},\tau}$ under $\mathsf{Freeze}/\mathsf{Shr}$ frames), and
(iv) lifetimes are properly nested and ended through stack-pop transitions.
Generic instantiation is guarded by the satisfiability relation $\Vdash$ and associated-type resolution.
Therefore, generated harnesses are expected to be accepted by the Rust compiler under this conservative, structured borrowing discipline.

\section{Formal Analysis for Sequence Generation and Coverage}
\label{sec:analysis}
\subsection{Executions, Traces, and Snippet Well-Formedness}
\label{sec:analysis:executions}

% firing 序列与Rust 可编译 snippet之间的关系说：
% - trace 是 certificate 序列；
% - Emit(trace) 输出代码；
% - 为保证 snippet 自包含，通常需要 trace 在末尾关闭借用/清空栈。

A \emph{trace} is a finite sequence of firing certificates
$\sigma=\kappa_1\cdots\kappa_n$ such that
$\mathsf{Cfg}_0[\sigma\rangle \mathsf{Cfg}_n$ where
$\mathsf{Cfg}_0=\langle M_0,\epsilon\rangle$ and the step relation
$\mathsf{Cfg}[\kappa\rangle \mathsf{Cfg}'$ follows Definition~\ref{def:pcpn:firing}.

We write $\mathrm{Emit}(\sigma)$ for the Rust snippet synthesized from $\sigma$
using the API transitions emit calls; structural transitions emit borrows, drops, projections, and reborrow scopes.

\paragraph{Closed configurations.}
% 为了让生成的 snippet不依赖外部作用域，terimal makring通常不能悬挂 borrow；
% 在 PCPN 中最直观的条件是栈为空，也可以额外要求 frz/blk 为空。
Define:
\begin{equation*}
    \mathsf{Closed}(\langle M,S\rangle)\ \triangleq\ (S=\epsilon).
\end{equation*}

Optionally, one may strengthen the notion by additionally requiring that no owner remains frozen/blocked:
$\forall\tau:\ M(p_{\mathsf{frz},\tau})=\emptyset\ \wedge\ M(p_{\mathsf{blk},\tau})=\emptyset$.

\paragraph{Well-formed snippet traces.}
% 最终生成的是一组 snippets；每条 snippet 最好对应 closed trace。
A trace $\sigma$ is \emph{snippet-well-formed} if it can be extended to a closed configuration:
\[
\exists\sigma':\ \mathsf{Cfg}_0[\sigma\cdot\sigma'\rangle \mathsf{Cfg}\ \wedge\ \mathsf{Closed}(\mathsf{Cfg}).
\]
In practice, we build $\sigma'$ by deterministically closing the pushdown stack (Section~\ref{sec:analysis:closing}).

\subsection{Reachability Graph vs. Coverability Tree}
\label{sec:analysis:rg-vs-ct}
% 可达图给精确可达状态空间（bounded 前提下）；能抽取具体 witness trace；覆盖树 (Karp--Miller) 是 unbounded 的过近似，有 \omega 不对应具体 token 数；
% 由于我们要输出具体 Rust 序列，因此主分析基于规范化可达图。

In Petri-net analysis, \emph{reachability graphs} provide an exact state-space for bounded systems,
whereas \emph{coverability trees} (e.g., Karp--Miller) over-approximate unbounded systems and may
introduce $\omega$-markings.
Since our goal is to synthesize \emph{concrete} firing sequences that translate to \emph{concrete} Rust snippets,
we base our analysis on a canonical reachability graph (CRG). For trace extraction we additionally maintain
a \emph{witness tree} as a by-product of reachability saturation.

\subsection{Guard Solving and Certificate Enumeration}
\label{sec:analysis:guards}
For each transition $t\in T$, we write the guard as a conjunction:
\begin{align*}
    \mathsf{Guard}(t)(\mathsf{Cfg},\chi,\iota)\ &\equiv \quad
G_t^{\mathsf{uni}}(\chi,\iota)\ \wedge\
G_t^{\mathsf{life}}(\chi,\mathsf{Cfg}.S,\iota)\ \\
&\quad \quad \wedge 
G_t^{\mathsf{trait}}(\chi,\iota)\ \wedge\
G_t^{\mathsf{cap}}(\chi,\mathsf{Cfg}.S,\iota).
\end{align*}

% 该约束强制要求，泛型类型变量在所有被消耗的令牌颜色与签名所规定的等式上的实例化保持一致。具体而言，\iota包含一个代换\theta，G_t^{\mathsf{uni}}要求\theta与令牌携带的代换保持一致，且在应用代换\theta后，签名 /where 从句中的所有类型等式均成立。
\paragraph{(1) Type unification $G_t^{\mathsf{uni}}$.}
It enforces consistent instantiation of generic type variables across all consumed token colors and signature-imposed equalities. Concretely, $\iota$ contains a substitution $\theta$ and $G_t^{\mathsf{uni}}$ requires that $\theta$ agrees with token-carried substitutions and that all type equalities in the signature/where-clause hold after applying $\theta$.

% 该约束强制保障生命周期代换\lambda的一致性与栈的兼容性，具体要求如下：(i) 若某一签名在多个参数中使用了同一个生命周期变量，则这些参数对应的选定令牌必须在代换\lambda上保持一致；(ii) 若变迁t执行弹帧操作，则所需的模式必须与栈顶元素相匹配（遵循后进先出LIFO原则）。
\paragraph{(2) Lifetime/stack constraint $G_t^{\mathsf{life}}$.}
It enforces lifetime substitution $\lambda$ consistency and stack compatibility: (i) if a signature uses the same lifetime variable across multiple parameters, their chosen tokens must agree on $\lambda$, and (ii) if $t$ pops frames, the required pattern must match the stack top.

% 该约束在代换\theta的作用下，对 where 从句中的所有约束条件进行校验：对于每一项约束要求\rho_i，需满足\theta(\tau)\Vdash \rho_i，且关联类型等式的解析结果保持一致。
\paragraph{(3) Trait/associated-type constraint $G_t^{\mathsf{trait}}$.}
It checks all bounds in the where-clause under $\theta$: for each requirement $\rho_i$, require $\theta(\tau)\Vdash \rho_i$ and that associated-type equalities resolve consistently.

% 该约束确保被消耗/生成的库所符合 Rust 兼容性要求（遵循所有权 / 借用规则），例如：借用操作会将所属权转移至\mathsf{frz}/\mathsf{blk}库所；可变操作的影响无法直接消耗处于阻塞状态的所属权等。
\paragraph{(4) Capability constraint $G_t^{\mathsf{cap}}$.}
It enforces Rust-compatibility of consumed/produced places including ownership/borrow discipline such as borrows move owners to $\mathsf{frz}/\mathsf{blk}$, mutable effects cannot directly consume from blocked owners, etc.

\begin{algorithm}[t]
\caption{$\mathsf{SolveGuard}(t,\mathsf{Cfg},\chi)$}
\label{alg:solve-guard}
% SolveGuard 是 reachability 算法的核心子过程：给定 (t, 当前栈, 选中的输入 token multiset)，
% 产出所有可行 instantiations（有限集合）。
\begin{algorithmic}[1]
\REQUIRE transition $t$, configuration $\mathsf{Cfg}=\langle M,S\rangle$, choice $\chi$
\ENSURE finite set of admissible instantiations $\mathsf{Inst}(t,\mathsf{Cfg},\chi)$
\STATE $\Theta \gets \mathsf{UnifyTypes}(t,\chi)$
\STATE $\Lambda \gets \mathsf{UnifyLifetimes}(t,\chi,S)$
\STATE $\mathsf{Inst}\gets \emptyset$
\FORALL{$(\theta,\lambda)\in \Theta\times \Lambda$}
  \IF{$\mathsf{CheckTraitAssoc}(t,\theta)$ \AND $\mathsf{CheckCap}(t,\mathsf{Cfg},\chi,\theta,\lambda)$}
     \STATE $\mathsf{Inst}\gets \mathsf{Inst}\cup\{(\theta,\lambda,\dots)\}$
  \ENDIF
\ENDFOR
\RETURN $\mathsf{Inst}$
\end{algorithmic}
\end{algorithm}

\subsection{Boundedness and Finiteness of the Analysis Domain}
\label{sec:analysis:boundedness}

% PCPN 本质上是 pushdown + colored（id/label 新鲜性），理论上容易变成无限状态。所以为了做可达图 + 覆盖保证，必须把分析域变成有限：token 数有界（通过预算/guard 或不变量证明）；栈深有界（D）；泛型候选集有限，也就是基本类型和自定义复合类型；新的 id/label 通过 alpha-renaming 规范化（Canon）quotient。

\paragraph{Budgeted boundedness (constructive).}
Let $B:P\to\mathbb{N}$ be a \emph{place budget}. We enforce $B$ by requiring that any transition that
would increase tokens in $p$ satisfies $|M(p)| < B(p)$ in its guard (e.g., constructors, copy-duplication,
API calls that return fresh owned values).
We similarly enforce a \emph{stack-depth budget} $D\in\mathbb{N}$ by requiring $|S|<D$ before any push action.

\paragraph{Finite instantiation domain.}
For each generic type variable $\alpha$, let $\mathsf{Cand}(\alpha)\subseteq \mathsf{GTy}$ be a finite candidate set
computed from the crate (e.g., monomorphic types appearing in $\Sigma(\mathcal{C})$). Then $\mathsf{UnifyTypes}$
returns a finite set of $\theta$.

\paragraph{Canonicalization (quotient by $\alpha$-renaming).}
% token 颜色里有新的 value id/region label，若不 quotient，就会无限多等价状态。Canon 的做法：按 首次出现顺序 把所有 v/l 重新命名成 v_1,v_2,... 和 \ell_1,\ell_2,...，并对每个 place 的 multiset 做排序/规范表示。
Let $\mathsf{Canon}(\langle M,S\rangle)$ rename all value identifiers and region labels to a canonical sequence
based on first occurrence (and canonically order multisets), yielding a representative of the $\alpha$-equivalence class.

\begin{assumption}[Finiteness assumptions]
\label{ass:finite}
We assume:
\begin{enumerate}
\item[(B1)] token counts are bounded by a budget $B$ (or equivalently, the net is $k$-bounded);
\item[(B2)] stack depth is bounded by $D$;
\item[(B3)] each generic variable ranges over a finite candidate set $\mathsf{Cand}(\alpha)$, hence $\mathsf{SolveGuard}$ is finite;
\item[(B4)] configurations are quotiented by canonicalization $\mathsf{Canon}$.
\end{enumerate}
\end{assumption}

\begin{theorem}[Finite canonical state space]
\label{thm:finite-state}
Under Assumption~\ref{ass:finite}, there are only finitely many canonical configurations
$\widehat{\mathsf{Cfg}}=\mathsf{Canon}(\mathsf{Cfg})$ reachable from $\mathsf{Canon}(\langle M_0,\epsilon\rangle)$.
\end{theorem}
\noindent\textbf{Proof sketch.}
(B1) bounds token multiplicities per place; since $P$ is finite for a fixed crate, the multiset carrier is bounded.
(B2) bounds stack words over a finite alphabet.
(B3) bounds instantiation choices per step.
(B4) quotients away unboundedly many fresh names. Thus the canonical reachable set is finite. \hfill $\square$

\subsection{Canonical Reachability Graph and Witness Tree}
\label{sec:analysis:crg}

\begin{definition}[Annotated Canonical Reachability Graph]
\label{def:crg}
Let $\widehat{\mathsf{Cfg}}_0=\mathsf{Canon}(\langle M_0,\epsilon\rangle)$.
The canonical reachability graph is $\widehat{G}=(\widehat{V},\widehat{E})$ where $\widehat{V}$ is the set of
canonical configurations reachable from $\widehat{\mathsf{Cfg}}_0$, and
\begin{align*}
    (\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\in \widehat{E}\ \quad &\text{iff} \quad 
\exists \mathsf{Cfg},\\
&\quad \mathsf{Cfg}':\
\mathsf{Canon}(\mathsf{Cfg})=\widehat{\mathsf{Cfg}},\\
&\quad \mathsf{Canon}(\mathsf{Cfg}')=\widehat{\mathsf{Cfg}}',\
\mathsf{Cfg}[\kappa\rangle \mathsf{Cfg}'.
\end{align*}

\end{definition}

\begin{definition}[Witness tree]
\label{def:witness-tree}
A witness tree $\mathcal{T}$ is a rooted spanning tree over $\widehat{V}$ that stores,
for each discovered node $\widehat{\mathsf{Cfg}}\neq \widehat{\mathsf{Cfg}}_0$, one parent edge
$(\widehat{\mathsf{Par}},\kappa,\widehat{\mathsf{Cfg}})\in \widehat{E}$.
Thus each node has a unique root-to-node witness trace.
\end{definition}

\begin{algorithm}[t]
\caption{$\mathsf{SaturateWithWitness}(\mathcal{N}_{\mathcal{C}})$}
\label{alg:saturate-wit}
% 核心枚举：transition t × choices chi × instantiations iota。
\begin{algorithmic}[1]
\STATE $\widehat{V}\gets \emptyset,\ \widehat{E}\gets \emptyset$
\STATE $\mathsf{parent}(\cdot)\gets \bot$
\STATE $\widehat{\mathsf{Cfg}}_0\gets \mathsf{Canon}(\langle M_0,\epsilon\rangle)$
\STATE $\mathsf{W}\gets \{\widehat{\mathsf{Cfg}}_0\}$
\WHILE{$\mathsf{W}\neq \emptyset$}
  \STATE remove some $\widehat{\mathsf{Cfg}}=\langle \widehat{M},\widehat{S}\rangle$ from $\mathsf{W}$
  \IF{$\widehat{\mathsf{Cfg}}\in \widehat{V}$} \textbf{continue} \ENDIF
  \STATE $\widehat{V}\gets \widehat{V}\cup\{\widehat{\mathsf{Cfg}}\}$
  \FORALL{$t\in T$}
    \FORALL{choices $\chi$ for ${}^{\bullet}t$ at $\widehat{M}$}
      \FORALL{$\iota\in \mathsf{SolveGuard}(t,\widehat{\mathsf{Cfg}},\chi)$}
        \STATE $\kappa \gets \langle t,\chi,\iota\rangle$
        \STATE $\widehat{\mathsf{Cfg}}'\gets \mathsf{Canon}(\mathsf{Fire}(\widehat{\mathsf{Cfg}},\kappa))$
        \STATE $\widehat{E}\gets \widehat{E}\cup\{(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\}$
        \IF{$\widehat{\mathsf{Cfg}}'\notin \widehat{V}$ \AND $\mathsf{parent}(\widehat{\mathsf{Cfg}}')=\bot$}
          \STATE $\mathsf{parent}(\widehat{\mathsf{Cfg}}')\gets (\widehat{\mathsf{Cfg}},\kappa)$
        \ENDIF
        \STATE $\mathsf{W}\gets \mathsf{W}\cup\{\widehat{\mathsf{Cfg}}'\}$
      \ENDFOR
    \ENDFOR
  \ENDFOR
\ENDWHILE
\RETURN $(\widehat{V},\widehat{E},\mathsf{parent})$
\end{algorithmic}
\end{algorithm}

\begin{theorem}[Termination and completeness of saturation]
\label{thm:term-comp}
Under Assumption~\ref{ass:finite}, Algorithm~\ref{alg:saturate-wit} terminates and returns the complete
canonical reachability graph $\widehat{G}$ and a witness tree spanning $\widehat{V}$.
\end{theorem}
\noindent\textbf{Proof sketch.}
By Theorem~\ref{thm:finite-state}, there are finitely many canonical configurations. For each configuration,
the number of (choices $\chi$) is finite because token multiplicities are bounded, and the number of instantiations is finite
by (B3). Thus saturation terminates. Completeness follows from enumerating all enabled certificates. \hfill $\square$

\paragraph{Complexity note (optional).}
% 看看可以放 appendix 里，这里给出粗略复杂度：O(|V| * |T| * #choices * #instantiations)；
Let $\mathsf{Ch}(\widehat{M},t)$ be the number of valid choices for $t$ at marking $\widehat{M}$,
and $\mathsf{Inst}(\widehat{\mathsf{Cfg}},t,\chi)$ the number of admissible instantiations.
The worst-case saturation cost is
$O\!\left(\sum_{\widehat{\mathsf{Cfg}}\in\widehat{V}}\sum_{t\in T}\sum_{\chi}\ |\mathsf{Inst}(\widehat{\mathsf{Cfg}},t,\chi)|\right)$.

\subsection{Deadness and Liveness Analysis on $\widehat{G}$}
\label{sec:analysis:liveness}

% 活性分析：
% - dead transition：在可达图里根本没有出现（不可生成）；
% - quasi-live：至少在某个可达配置里可触发（可生成一次）；
% - live：从任意可达配置出发，总能走到一个可触发它的配置（永远不会“彻底失去机会”）。
% 对生成很重要：dead 的 API 从分母剔除；live/quasi-live 用于调度与避免死胡同。

\paragraph{Enabledness sets.}
For a transition $t$, define the set of nodes where $t$ is enabled:
\[
\mathsf{En}(t)=\{\ \widehat{\mathsf{Cfg}}\in \widehat{V}\mid \exists \kappa,\widehat{\mathsf{Cfg}}':\
(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\in \widehat{E}\ \wedge\ \kappa.\mathsf{tr}=t\ \}.
\]

\begin{definition}[Dead, quasi-live, live]
\label{def:liveness}
Let $t\in T$.
\begin{itemize}
\item $t$ is \emph{dead} iff $\mathsf{En}(t)=\emptyset$ (no reachable firing instance).
\item $t$ is \emph{quasi-live} iff $\mathsf{En}(t)\neq\emptyset$ (fireable at least once along some execution).
\item $t$ is \emph{live} iff for every $\widehat{\mathsf{Cfg}}\in\widehat{V}$ there exists a path in $\widehat{G}$
from $\widehat{\mathsf{Cfg}}$ to some node in $\mathsf{En}(t)$.
\end{itemize}
\end{definition}

\begin{algorithm}[t]
\caption{$\mathsf{AnalyzeDeadQuasiLive}(\widehat{G})$}
\label{alg:dead}
% dead/quasi-live 用一遍扫边即可（看每条边上出现了哪些 t）。
\begin{algorithmic}[1]
\REQUIRE $\widehat{G}=(\widehat{V},\widehat{E})$
\ENSURE $\mathsf{Dead},\mathsf{QuasiLive}\subseteq T$
\STATE $\mathsf{Seen}\gets \emptyset$
\FORALL{$(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\in \widehat{E}$}
  \STATE $\mathsf{Seen}\gets \mathsf{Seen}\cup\{\kappa.\mathsf{tr}\}$
\ENDFOR
\STATE $\mathsf{QuasiLive}\gets \mathsf{Seen}$
\STATE $\mathsf{Dead}\gets T\setminus \mathsf{Seen}$
\RETURN $(\mathsf{Dead},\mathsf{QuasiLive})$
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[t]
\caption{$\mathsf{AnalyzeLiveness}(\widehat{G},t)$}
\label{alg:live}
% live 判定可以用“反向可达”做：从 En(t) 反向 BFS 得到可达它的节点集合 Pred*(En(t))；
% 如果覆盖全体 V，则 t live。
\begin{algorithmic}[1]
\REQUIRE $\widehat{G}=(\widehat{V},\widehat{E})$, transition $t$
\ENSURE boolean $\mathsf{IsLive}(t)$
\STATE $\mathsf{Targets}\gets \mathsf{En}(t)$
\IF{$\mathsf{Targets}=\emptyset$} \RETURN $\mathsf{false}$ \ENDIF
\STATE build reverse adjacency $\widehat{E}^{-1}$
\STATE $\mathsf{R}\gets \mathsf{Targets}$ \COMMENT{nodes that can reach an enabling node}
\STATE $\mathsf{Q}\gets \mathsf{Targets}$
\WHILE{$\mathsf{Q}\neq\emptyset$}
  \STATE pop $\widehat{\mathsf{Cfg}}$ from $\mathsf{Q}$
  \FORALL{$(\widehat{\mathsf{Cfg}}_{\mathrm{pre}},\_,\widehat{\mathsf{Cfg}})\in \widehat{E}^{-1}$}
     \IF{$\widehat{\mathsf{Cfg}}_{\mathrm{pre}}\notin \mathsf{R}$}
        \STATE $\mathsf{R}\gets \mathsf{R}\cup\{\widehat{\mathsf{Cfg}}_{\mathrm{pre}}\}$
        \STATE $\mathsf{Q}\gets \mathsf{Q}\cup\{\widehat{\mathsf{Cfg}}_{\mathrm{pre}}\}$
     \ENDIF
  \ENDFOR
\ENDWHILE
\RETURN $(\mathsf{R}=\widehat{V})$
\end{algorithmic}
\end{algorithm}

\paragraph{Deadlocks and termination sinks.}
% 识别 deadlock 配置，避免把 trace 走到“栈无法闭合/owner 永远 blk”之类。
A canonical configuration $\widehat{\mathsf{Cfg}}$ is a \emph{deadlock} iff it has no outgoing edges in $\widehat{E}$.
Such nodes can be detected by scanning out-degrees in $\widehat{G}$.

\subsection{Closing Borrows and Making Snippets Self-Contained}
\label{sec:analysis:closing}

\begin{algorithm}[t]
\caption{$\mathsf{CloseStack}(\mathsf{Cfg})$}
\label{alg:closestack}
% 按栈顶符号决定用哪个 End* transition，直到 S=\epsilon。
% 每个 frame 都有唯一对应的 ending transition（EndShrKeep/EndShrLast/EndMut/EndProjMut...）。
\begin{algorithmic}[1]
\REQUIRE reachable configuration $\mathsf{Cfg}=\langle M,S\rangle$
\ENSURE a closing trace $\sigma_{\mathrm{close}}$ such that $\mathsf{Cfg}[\sigma_{\mathrm{close}}\rangle \langle M',\epsilon\rangle$
\STATE $\sigma_{\mathrm{close}}\gets \epsilon$
\WHILE{$S\neq \epsilon$}
  \STATE let $\gamma=\mathsf{top}(S)$
  \STATE choose the unique applicable ending transition $t_{\mathrm{end}}$ for $\gamma$ (by pattern)
  \STATE choose $\chi$ matching the required input tokens for $t_{\mathrm{end}}$
  \STATE pick some $\iota\in \mathsf{SolveGuard}(t_{\mathrm{end}},\langle M,S\rangle,\chi)$
  \STATE fire $\kappa=\langle t_{\mathrm{end}},\chi,\iota\rangle$ to update $(M,S)$
  \STATE $\sigma_{\mathrm{close}}\gets \sigma_{\mathrm{close}}\cdot \kappa$
\ENDWHILE
\RETURN $\sigma_{\mathrm{close}}$
\end{algorithmic}
\end{algorithm}

\begin{lemma}[Closing terminates]
\label{lem:close-terminates}
If $\mathsf{CloseStack}$ does not get stuck, it terminates in at most $|S|$ iterations and returns a trace that empties the stack.
\end{lemma}
\noindent\textbf{Proof sketch.}
Each step pops at least one stack symbol; hence the stack length strictly decreases. \hfill $\square$

\paragraph{Pragmatic note.}
% 是否不会卡住取决于 method 里 End* transition 的 guard/out 设计；
% 一般保证 push 的时候一定同时生成未来 pop 所需的 token或者把 pop 设计成不依赖已被 move 的 token。
% TODO：写成一个 lamda 表达式
In our construction, each push action is paired with a corresponding ending transition that restores owners/references,
so reachable configurations admit stack closing under the intended usage discipline (structured borrows).

\subsection{Coverage Metrics over Snippet Sets}
\label{sec:analysis:coverage}

% 覆盖率对生成的一组 snippets/trace 集合定义；
% 分母是可达目标集合，否则不可达 API 永远导致覆盖<1。
Let $T_{\mathrm{API}}\subseteq T$ be transitions corresponding to crate functions/methods.
Define the \emph{reachable} API transition set:
\[
T_{\mathrm{API}}^{\mathrm{reach}}
=\{\, t\in T_{\mathrm{API}}\mid \mathsf{En}(t)\neq \emptyset \,\}.
\]
For type forms, let $P_{\mathrm{form}}\subseteq P$ be places representing type forms of interest
(owned/ref/assoc/field/witness, etc.), and define:
\[
P_{\mathrm{form}}^{\mathrm{reach}}
=\{\, p\in P_{\mathrm{form}}\mid \exists \widehat{\mathsf{Cfg}}=\langle \widehat{M},\widehat{S}\rangle\in \widehat{V}:\ \widehat{M}(p)\neq \emptyset \,\}.
\]
For borrow shapes, define:
\[
\mathcal{B}^{\mathrm{reach}}=\{\, \mathsf{Shape}(\widehat{S}) \mid \langle \widehat{M},\widehat{S}\rangle\in \widehat{V}\,\}.
\]

\paragraph{Coverage of a finite trace set.}
Let $\Sigma$ be a finite set of traces (equivalently, a set of generated snippets $\{\mathrm{Emit}(\sigma)\mid \sigma\in\Sigma\}$).
Define:
\[
\mathsf{API}(\Sigma)=\bigcup_{\sigma\in\Sigma}\{\, \kappa.\mathsf{tr}\mid \kappa\in\sigma,\ \kappa.\mathsf{tr}\in T_{\mathrm{API}}\,\}.
\]
Let $\sigma$ induce intermediate configurations $\mathsf{Cfg}_i=\langle M_i,S_i\rangle$.
Define:
\[
\mathsf{Vis}_P(\Sigma)=\bigcup_{\sigma\in\Sigma}\bigcup_{i}\{\, p\in P_{\mathrm{form}}\mid M_i(p)\neq \emptyset \,\},
\quad
\mathsf{Vis}_B(\Sigma)=\bigcup_{\sigma\in\Sigma}\bigcup_{i}\{\mathsf{Shape}(S_i)\}.
\]
Then:
\begin{align*}
\mathrm{Cov}_{\mathrm{API}}(\Sigma) &= \frac{|\mathsf{API}(\Sigma)|}{|T_{\mathrm{API}}^{\mathrm{reach}}|}, \\
\mathrm{Cov}_{\mathrm{Type}}(\Sigma) &= \frac{|\mathsf{Vis}_P(\Sigma)\cap P_{\mathrm{form}}^{\mathrm{reach}}|}{|P_{\mathrm{form}}^{\mathrm{reach}}|}, \\
\mathrm{Cov}_{\mathrm{Borrow}}(\Sigma) &= \frac{|\mathsf{Vis}_B(\Sigma)|}{|\mathcal{B}^{\mathrm{reach}}|}.
\end{align*}
We combine them as:
\[
\mathrm{GCov}(\Sigma)=w_1\cdot \mathrm{Cov}_{\mathrm{API}}(\Sigma)+w_2\cdot \mathrm{Cov}_{\mathrm{Type}}(\Sigma)+w_3\cdot \mathrm{Cov}_{\mathrm{Borrow}}(\Sigma),
\quad w_i\ge 0.
\]

\subsection{Coverage-Guaranteed Snippet Generation from $\widehat{G}$}
\label{sec:analysis:guarantee}

\paragraph{Witness extraction.}
Using the witness-tree parent map, each reachable node $\widehat{\mathsf{Cfg}}$ has a unique witness trace
$\mathsf{WitNode}(\widehat{\mathsf{Cfg}})$ obtained by backtracking $\mathsf{parent}$ pointers to the root.
For each edge $(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\in \widehat{E}$, define:
\[
\mathsf{WitEdge}(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')=
\mathsf{WitNode}(\widehat{\mathsf{Cfg}})\cdot \kappa.
\]

\begin{theorem}[Full model-level coverage within bounds]
\label{thm:full-coverage}
Let $(\widehat{V},\widehat{E},\mathsf{parent})$ be the output of Algorithm~\ref{alg:saturate-wit}.
Define:
\[
\Sigma_{\mathrm{API}}=\{\ \mathsf{WitEdge}(e)\mid e\in \widehat{E},\ e.\kappa.\mathsf{tr}\in T_{\mathrm{API}}\ \}.
\]
Then $\mathrm{Cov}_{\mathrm{API}}(\Sigma_{\mathrm{API}})=1$.
Moreover, by adding witnesses for first occurrences of each $p\in P_{\mathrm{form}}^{\mathrm{reach}}$
and each $b\in \mathcal{B}^{\mathrm{reach}}$, we obtain a finite set $\Sigma^\star$ such that
\[
\mathrm{Cov}_{\mathrm{API}}(\Sigma^\star)=\mathrm{Cov}_{\mathrm{Type}}(\Sigma^\star)=\mathrm{Cov}_{\mathrm{Borrow}}(\Sigma^\star)=1.
\]
\end{theorem}
\noindent\textbf{Proof sketch.}
For any $t\in T_{\mathrm{API}}^{\mathrm{reach}}$, there exists an edge in $\widehat{E}$ whose certificate transition is $t$.
Its witness edge trace contains $t$. The place/shape statements follow by selecting a node/edge where the objective appears. \hfill $\square$

\paragraph{Making witnesses self-contained.}
% 把每条 witness trace 通过 CloseStack 变成 closed trace，不影响覆盖集合。
For each witness trace $\sigma$, we post-process it as
$\sigma^{\downarrow} = \sigma \cdot \mathsf{CloseStack}(\mathsf{Cfg}_\sigma)$
where $\mathsf{Cfg}_\sigma$ is the reached configuration after firing $\sigma$.
Then $\mathrm{Emit}(\sigma^{\downarrow})$ yields a self-contained snippet.

\subsection{Budgeted High-Coverage Selection}
\label{sec:analysis:budgeted}

% 全覆盖会产生很多 snippets；实际上只需要少量的，只要达到高覆盖就行。
% 这可形式化为 maximum coverage / (weighted) set cover，用贪心有 (1-1/e) 近似保证。

Let $\mathcal{S}$ be a pool of candidate traces (e.g., witnesses for edges or for objectives).
Each trace $\sigma$ covers objectives:
\[
\mathsf{Obj}(\sigma)=
\big(\mathsf{API}(\{\sigma\})\big)\ \cup\
\big(\mathsf{Vis}_P(\{\sigma\})\cap P_{\mathrm{form}}^{\mathrm{reach}}\big)\ \cup\
\mathsf{Vis}_B(\{\sigma\}).
\]
Given a budget $m$, select $\Sigma\subseteq \mathcal{S}$ with $|\Sigma|\le m$ maximizing
$\left|\bigcup_{\sigma\in \Sigma}\mathsf{Obj}(\sigma)\right|$.

\begin{algorithm}[t]
\caption{$\mathsf{GreedyCover}(\mathcal{S},m)$}
\label{alg:greedycover}
\begin{algorithmic}[1]
\STATE $\Sigma\gets \emptyset,\ \mathsf{U}\gets$ all objectives initially uncovered
\FOR{$i=1$ to $m$}
  \STATE choose $\sigma^\star\in \mathcal{S}\setminus \Sigma$ maximizing $|\mathsf{Obj}(\sigma^\star)\cap \mathsf{U}|$
  \STATE $\Sigma\gets \Sigma\cup\{\sigma^\star\}$
  \STATE $\mathsf{U}\gets \mathsf{U}\setminus \mathsf{Obj}(\sigma^\star)$
\ENDFOR
\RETURN $\Sigma$
\end{algorithmic}
\end{algorithm}

\paragraph{Optional weighted variant.}
If each objective $o$ has weight $w(o)$ and each trace has cost $\mathsf{cost}(\sigma)$ (e.g., length),
we can use a standard greedy ratio rule maximizing
$\frac{w(\mathsf{Obj}(\sigma)\cap \mathsf{U})}{\mathsf{cost}(\sigma)}$.

We structure the evaluation around the following research questions:

\begin{description}

  % 证明生成的 Rust 代码能过编译/能跑

  \item[\textbf{RQ2 (Target Code Coverage).}]
  % 在相同 fuzzing 预算下，目标 crate 的覆盖率更高。
  Under the same fuzzing budget, do PCPN-generated harnesses achieve higher \emph{target-library} coverage
  (line/branch/function coverage of the crate-under-test) than competing approaches?

  \item[\textbf{RQ3 (Bug-Finding Effectiveness).}]
  % 这个待定，可能只做生成，不检查 bug。
  Does our approach find more unique failures (e.g., panics, assertion violations, sanitizer crashes) and/or find them faster
  (time-to-first-crash) than baselines?

  \item[\textbf{RQ4 (Feature Support and Stress Cases).}]
  % 我们支持 Rust 特性：复合类型、借用/生命周期、泛型、trait/关联类型、字段重借用。
  How robust is the approach on crates that heavily use (i) borrow-carrying types and explicit lifetimes in signatures,
  (ii) generic APIs with trait bounds and associated types, and (iii) composite types requiring field projection/reborrowing?

  \item[\textbf{RQ5 (Ablation: Which Components Matter?).}]
  % 证明不是随便一个搜索/生成就行，而是 PCPN 增强了这一能力。
  What is the contribution of each major component (pushdown lifetime stack, generic partial-order instantiation, field (re)borrowing rules,
  POR/heuristics if any) to compilability and coverage?

  \item[\textbf{RQ6 (Model Coverage $\rightarrow$ Code Coverage).}]
  % 在 Petri 网层面追求 transition coverage/marking diversity，这是否与目标代码覆盖相关？
  To what extent does increasing \emph{net-level coverage} (e.g., transition coverage over API transitions) correlate with
  increased \emph{target-code} coverage, and can it serve as a reliable surrogate objective?
\end{description}

\subsection{Benchmarks and Subjects}
\label{sec:evaluation:benchmarks}
\paragraph{Real-world crates.}
We evaluate on a curated set of Rust crates drawn from crates.io.
We select crates that (i) expose non-trivial public APIs (multiple modules/types),
(ii) include stateful protocols (resources/handles/transactions/builders), and
(iii) collectively cover diverse domains (parsers, codecs, data structures, storage, networking, etc.).
% 这里给出选择标准即可，具体名单放表格；不要在正文里硬编百分比。
For each crate, we extract callable items $\mathcal{F}$ (public functions/methods) and their signatures via the compiler front-end.

\paragraph{Feature-stratified subsets.}
% 为了回答 RQ4，把 crate 按特性分组（用静态特征统计）：
We partition crates into groups based on signature-level features:
\begin{itemize}
  \item \textbf{Lifetime-heavy}: explicit lifetimes and borrow-carrying return types in signatures;
  \item \textbf{Generic-heavy}: frequent type parameters with trait bounds / where-clauses / associated types;
  \item \textbf{Composite-heavy}: deep nested ADTs and frequent field projections (struct/tuple) in APIs.
\end{itemize}


\subsection{Baselines}
\label{sec:evaluation:baselines}
% (B0) 传统 fuzz（字节输入 + 手写/半手写 harness）；
% (B1) 只用签名做 typed 生成但忽略 borrow/lifetime （RULF 和 RPG 所定义的 api graph，这个要说比他们强）；
% (B2) Petri 网但不含 pushdown/不含泛型求解（和 POPL 2017 论文比较一下，sypet）。
We compare against the following baselines:

\begin{description}
  \item[\textbf{B0: Byte-oriented fuzzing harness}.]
  A conventional setup where the fuzzer mutates a byte array and a harness decodes it into inputs for a fixed entry point.
  % 尽量找已经写好测试用里的库

  \item[\textbf{B1: Signature-driven random sequences (no borrow/lifetime model).}]
  A type-directed generator that samples call sequences using signatures, but treats values as freely duplicable or ends borrows heuristically
  without a formal stack discipline.

  \item[\textbf{B2: Petri net without pushdown (finite-state approximation).}]
  A Petri-net-based generator that uses places/transitions derived from signatures but replaces the pushdown lifetime mechanism with a finite-state
  approximation (e.g., lexical-only or bounded non-nested borrow tracking).

  \item[\textbf{B3 (optional): Property-based testing}.]
  A property-based generator (e.g., arbitrary structured values) for selected APIs, without stateful call-sequence generation.
\end{description}

\subsection{Metrics}
\label{sec:evaluation:metrics}
\paragraph{Compilability and executability.}
Let $\mathcal{H}$ be the set of synthesized harnesses.
We define:
\begin{align}
\textbf{CHR} &= \frac{|\{h\in\mathcal{H} \mid \texttt{rustc}(h)~\text{accepts}\}|}{|\mathcal{H}|},
\label{eq:chr}\\
\textbf{EHR} &= \frac{|\{h\in\mathcal{H} \mid h~\text{executes and completes initialization}\}|}{|\mathcal{H}|}.
\label{eq:ehr}
\end{align}
% CHR=编译通过率；EHR=能跑起来（至少跑过初始化/跑若干轮 fuzz iteration）的比例。

\paragraph{Target-library coverage.}
% 覆盖率针对目标 crate，而不是 harness 本身。
For each crate-under-test, we measure standard code coverage (e.g., line and branch coverage):
\begin{align}
\textbf{TLC} &= \frac{\text{covered lines in target crate}}{\text{total instrumented lines in target crate}},
\label{eq:tlc}\\
\textbf{TBC} &= \frac{\text{covered branches in target crate}}{\text{total instrumented branches in target crate}}.
\label{eq:tbc}
\end{align}

\paragraph{Model (net-level) coverage.}
Let $T_{\mathsf{api}}\subseteq T$ be transitions corresponding to extracted API calls.
For a trace multiset $\Sigma$ (generated during exploration/fuzzing), define:
\begin{align}
\textbf{NTC} &= \frac{|\{t\in T_{\mathsf{api}} \mid \exists \sigma\in\Sigma: t\in\sigma\}|}{|T_{\mathsf{api}}|}.
\label{eq:ntc}
\end{align}
% NTC=网层面 API transition 覆盖率（至少被触发一次的比例）。

% \paragraph{Bug-finding.}
We record:
(i) the number of unique failures (deduplicated by stack trace hash or sanitizer report signature),
and (ii) time-to-first-crash (TTFC).


\paragraph{Cost and scalability.}
We measure:
net construction time, exploration time per trace, harness compilation time, and fuzzing throughput (exec/s).

\subsection{Experimental Protocol}
\label{sec:evaluation:protocol}
For each crate and each approach (ours and baselines), we perform the following steps:

\begin{enumerate}
  \item \textbf{Model extraction (ours / B2).} Extract signatures $\mathcal{F}$ and build the Petri-net instance.
  \item \textbf{Harness generation.} Generate $K$ harnesses (or $K$ traces), each bounded by max trace length $L$ and stack depth $D$.
  \item \textbf{Compilation filter.} Compile each harness; record CHR and compile-time.
  \item \textbf{Fuzzing run.} Run each compiled harness for a fixed time budget $B_{\mathsf{time}}$ (or fixed total executions),
        using the same fuzzer backend and identical resource limits.
  \item \textbf{Measurement.} Collect target coverage (TLC/TBC), net-level coverage (NTC), and failures.
\end{enumerate}

\paragraph{Repetitions and statistics.}
Each configuration is repeated $R$ times with independent random seeds.
We report mean and standard deviation (or median and IQR) and use non-parametric tests when appropriate.
% 可以只写置信区间

\subsection{Ablation and Sensitivity Studies}
\label{sec:evaluation:ablation}
% 为回答 RQ5/RQ4，做消融与敏感性：
\paragraph{Ablations.}
We evaluate the following variants:
\begin{itemize}
  \item \textbf{NoStack}: remove the pushdown lifetime stack and replace with a finite approximation;
  \item \textbf{NoPO}: replace the partial-order-based generic instantiation with random instantiation from the same candidate pool;
  \item \textbf{NoFieldReborrow}: disable field projection/reborrow transitions;
  \item \textbf{NoAssoc}: disable associated-type rewriting transitions.
\end{itemize}
We compare each variant against the full system in CHR/EHR, TLC/TBC, and TTFC.

\paragraph{Sensitivity.}
We vary $L$ (trace length bound), $D$ (stack depth bound), and $|\mathsf{Cand}(\alpha)|$ (generic candidate pool size)
to quantify trade-offs between exploration power, compilation success, and runtime coverage.
