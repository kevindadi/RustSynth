\begin{abstract}
% Rust 库越来越多地暴露有状态、类型导向的 API，其有效测试需要生成长且类型明确的调用序列，而不是修改非结构化的字节输入。然而，Rust 的仿射所有权、借用排他性和生命周期约束构成了编译障碍：由于使用后移动、&mut 下的非法别名或未满足的泛型边界，简单生成的序列经常无法通过类型检查，从而阻止任何运行时反馈。本文通过将 API 调用建模为类型化值和权限上的资源流问题，解决了 Rust crate 的签名级测试生成问题。
Rust libraries increasingly expose stateful, type-directed APIs whose correct usage is enforced not only by runtime checks but also by compile-time rules on ownership, borrowing, and lifetimes. As a result, automatically synthesizing API-using Rust code is often blocked by a compilation barrier: generated call sequences fail to type-check due to move-after-use, aliasing violations under $\&mut$, or unsatisfied generic bounds, and thus never execute. This barrier limits not only coverage-guided fuzzing harnesses, but more broadly any signature-level code generation task that requires producing well-typed, protocol-respecting API sequences.

We present a Pushdown Colored Petri Net for correct-by-construction Rust snippet synthesis from crate signatures. In our encoding, places represent pools of typed values annotated with capabilities, transitions represent signature-driven API calls and a small set of structural operations, and a pushdown stack enforces properly nested borrow extents with permission return. Generic parameters are instantiated at firing time via unification and a trait/associated-type satisfiability check over the crate environment.

On top of the model, we develop a bounded reachability analysis that constructs a finite monomorphized reachability graph, extracts witness firing sequences for user-specified synthesis objectives, and deterministically closes open scopes to emit self-contained, compilable Rust snippets. In a downstream case study, wrapping synthesized snippets as fuzz harnesses yields a compilation success rate of 11\% for baselines and improves branch coverage by 13 points on 8 real-world crates .

\section{Introduction}
Rust has become a mainstream choice for building reliable systems software and performance-critical libraries.
Its combination of low-level performance and a rich type system eliminates broad classes of memory-safety bugs,
while still exposing fine-grained control over resources.
As a result, an increasing amount of security and correctness critical functionality is delivered as reusable Rust crates. This paper studies a problem that sits one level above traditional input fuzzing: how to automatically generate compilable Rust code that composes a crate's public APIs in long, stateful sequences.
Such snippets are useful in multiple downstream settings:
coverage-guided fuzzing, property-based testing, regression reproduction, and more generally, signature-level program synthesis.

In many languages, an automated generator can emit a syntactically plausible API sequence and rely on runtime feedback to guide search. In Rust, however, a large portion of API protocols is enforced at compile time: ownership is affine, borrows are exclusive under \texttt{\&mut}, and references must respect lifetimes. Moreover, real-world crates often expose generic APIs constrained by trait bounds and associated types. Consequently, generated sequences frequently fail to type-check due to move-after-use, illegal aliasing, or unsatisfied generic constraints, and thus never execute. This compilation barrier turns into a coverage barrier for fuzzing, but more fundamentally it blocks any form of automated API-level code generation.

Consider a simplified crate API in Listing~\ref{lst:api}, which resembles many real-world libraries (parsers, databases, protocol stacks, etc.). It features (i) composite types (\texttt{Result}, \texttt{Option}, \texttt{Vec}), (ii) lifetime-carrying handles that borrow from a parent object, and (iii) a generic entry point constrained by trait bounds.

\begin{lstlisting}[language=Rust,caption={A small Rust API exhibiting ownership transfer, borrowing, lifetimes, and generics.},label={lst:api}]
pub struct Store { /* ... */ }
pub struct Tx<'s> { store: &'s mut Store /* ... */ }
pub trait KeyLike {
    fn as_bytes(&self) -> &[u8];
}

impl Store {
    pub fn open(path: &str) -> Result<Self, ()> { /* ... */ }
    // Borrowing & lifetime: a transaction borrows the store mutably.
    pub fn begin<'s>(&'s mut self) -> Tx<'s> { /* ... */ }
}

impl<'s> Tx<'s> {
    pub fn put<K: KeyLike>(&mut self, k: K, v: Vec<u8>) { /* ... */ }
    // Immutable borrow from &self, returns a reference tied to 's.
    pub fn get<'a, K: KeyLike>(&'a self, k: K) -> Option<&'a [u8]> { /* ... */ }
    // Ownership consumption: committing consumes the Tx value.
    pub fn commit(self) -> Result<(), ()> { /* ... */ }
}
\end{lstlisting}

A generator that ignores Rust's ownership discipline might attempt to interleave calls in ways that are invalid in Rust. For instance, after creating a transaction \texttt{tx} that holds \texttt{\&mut store}, a generator may try to call \texttt{store.begin()} again or mutate \texttt{store} directly while \texttt{tx} is still live---both are rejected by the borrow checker. Similarly, \texttt{commit(self)} consumes \texttt{tx}, so any subsequent use triggers a move error. Finally, generic parameters such as \texttt{K: KeyLike} require the generator to produce a concrete type satisfying the bound, rather than an arbitrary byte blob. These issues are not merely syntactic.
In Rust, well-typedness encodes a large portion of the API protocol. Therefore, generating executable sequences for Rust libraries hinges on generating compilable code that respects ownership, borrowing, lifetimes, and generic constraints.

A wide range of automated techniques exist, including coverage-guided fuzzing, property-based testing, grammar-based generation,
symbolic execution, and model-based testing. However, when applied to Rust crate APIs, they commonly face at least one of the following limitations:
\begin{enumerate}
  \item[1.] Byte-oriented inputs without typed object construction.
  Many fuzzing setups operate on byte arrays and rely on a hand-written harness to translate bytes into structured objects.
  For stateful APIs, the harness must additionally manage long call sequences, intermediate values, and resource protocols.
  This manual effort is error-prone and often becomes the dominant bottleneck.

  \item[2.] Limited support for Rust-specific resource constraints.
  Generators that treat values as freely copyable struggle with Rust's affine ownership: values can be moved, borrowed immutably, or borrowed mutably, with strict aliasing rules. Moreover, Rust borrow extents are often inferred, so the end of a borrow is not obvious from surface syntax but is crucial for enabling subsequent mutations.

  \item[3.] Weak handling of composite and generic APIs.
  Real crates expose APIs over nested algebraic data types, slices, smart pointers, and rich trait-bound generics.
  Without a principled mechanism for instantiating generics and checking trait/associated-type constraints, generators either avoid such functions or produce ill-typed code that never compiles.

  \item[4.] Poor scalability when internal state is over-modeled.
  Some analysis-driven approaches attempt to model internal program state to drive generation. Although powerful, these approaches can be expensive and brittle for large crates,
  and are often unnecessary when the objective is signature-level code synthesis rather than whole-program verification.
\end{enumerate}

This paper targets a common and practically important setting:
we are given a Rust codebase, and we want to synthesize self-contained Rust snippets that invoke its public functions and methods. Crucially, we assume that function signatures, including input/output types, reference mutability, lifetimes, and generic bounds, are available from the Rust toolchain. We do not attempt to model the internal control flow or memory state of function bodies. Instead, we treat each function as an atomic transformer whose signature specifies how typed values and permissions may flow across call boundaries. This signature-level stance makes the approach lightweight and scalable, while still capturing the constraints that dominate compilation success.

We introduce a new model for Rust code generation based on Pushdown Colored Petri Nets (PCPNs). At a high level:
places represent pools of available typed values together with capabilities describing how they may be used such as owned, immutably borrowed and mutably borrowed. Transitions represent API calls extracted from the crate's signatures, together with a small set of structural operations including borrow, end-borrow, projection, and re-borrow. Token flow represents ownership transfer and borrow-induced capability changes. A pushdown stack models properly nested borrow extents: creating a reference pushes a frame that binds the reference to its provenance, and ending the borrow pops the frame and returns permissions to the original owner.

To support generic-heavy APIs, we equip the model with constraint solving at firing time: a transition is enabled only if there exists an instantiation of generic types and lifetimes
that (i) unifies with the consumed token types and (ii) satisfies trait bounds and associated-type constraints in the crate environment. This separates sequencing from admissibility reasoning, and enables exploration of generic APIs without manual stubs.

From a PCPN execution trace, we deterministically synthesize a Rust snippet. When used for fuzzing, the snippet becomes a fuzz harness by adding a fixed prelude while keeping the API sequence correct by construction. Listing~\ref{lst:harness} illustrates a typical generated skeleton for Listing~\ref{lst:api}. Importantly, the generator does not need to model what the functions do internally: the signature alone determines how ownership and borrow capabilities move across call boundaries, which is exactly the information required to produce compilable code.

\begin{lstlisting}[language=Rust,caption={A compilable, signature-driven harness sketch generated from a PCPN trace.},label={lst:harness}]
fn harness(bytes: &[u8]) {
    // Construct base values (from bytes or small generators).
    let path = "tmp.db"; // tmp.db from Constant
    let mut store = match Store::open(path) { Ok(s) => s, Err(_) => return };
    // Begin a transaction: tx borrows store mutably.
    let mut tx = store.begin();
    // Choose a concrete key type K that satisfies K: KeyLike.
    // Here we pick Vec<u8> (assume the crate provides impl KeyLike for Vec<u8>).
    let k: Vec<u8> = bytes.to_vec();
    // Use tx while it is live; store cannot be used mutably here.
    tx.put(k.clone(), bytes.to_vec());
    let _v = tx.get(k);
    // Explicitly end the borrow extent and consume tx.
    let _ = tx.commit();
    // After commit, permissions return: store is usable again.
    let _tx2 = store.begin();
}
\end{lstlisting}

This paper makes the following contributions:
\begin{enumerate}
  \item[1.] We formulate signature-level Rust code generation as a typed resource-flow problem and propose PCPNs as a unifying model
  for generating compilable API sequences under Rust ownership, borrowing, lifetimes, and generics.

  \item[2.] We design a capability-and-lifetime encoding for Rust values in PCPNs, including composite types and reborrowing patterns, and show how a pushdown stack discipline captures nested borrow extents with permission return.

  \item[3.] We introduce a bounded reachability analysis that constructs a finite monomorphized reachability graph and extracts witness traces under explicit synthesis objectives.

  \item[4.] As a downstream case study, we instantiate the framework to fuzz-harness generation and evaluate on real-world crates, showing that correct-by-construction sequences substantially improve compilation success and enable deeper API exploration for stateful Rust libraries.
\end{enumerate}

\subsection{Rust Ownership, Borrowing, and Lifetimes}
\label{sec:background:rust}
% 这一小节只讲与“生成可编译的 API 调用序列”直接相关的 Rust 规则：
% ownership（移动/析构）、borrowing（& / &mut 的互斥）、lifetimes（引用不能超过被借用者）。
% Rust 主要通过所有权和借用机制来确保内存安全。每个非 `\Copy` 值在同一时间都只有一个所有者；将值传递给另一个变量（或按值传递）通常会转移所有权。当所有者超出作用域时，该值将被丢弃，其资源将被释放。
Rust enforces memory safety primarily through a static discipline of ownership and borrowing. Every non-Copy value has exactly one owner at a time; transferring a value to another variable typically moves ownership. When the owner goes out of scope, the value is dropped and its resources are released.

% 借用允许临时访问值，而无需转移所有权。Rust 区分：(i) 共享借用 `\texttt{\&T}`，允许只读访问，并且可以任意别名；以及(ii) 可变借用 `\texttt{\&mut T}`，允许修改，但必须是互斥的。核心规则通常可以概括为“别名异或可变性”：在任何时刻，要么存在多个共享借用，要么存在一个可变借用。
Borrowing allows temporary access to a value without transferring ownership.
Rust distinguishes:
(i) shared borrows \texttt{\&T}, which permit read-only access and can be aliased arbitrarily, and
(ii) mutable borrows \texttt{\&mut T}, which permit mutation but must be exclusive.
The central rule is often summarized as aliasing XOR mutability:
at any point, either many shared borrows exist, or exactly one mutable borrow exists.

% 生命周期是一个编译时区域，用于限制引用的使用时长。引用的生命周期不能超过它所引用的值，借用权限会在借用结束时归还给所有者。在现代 Rust 中，借用范围通常是推断出来的（例如，通过非词法生命周期），这意味着借用的确切结束点不一定能从语法表面看出，但对于判断后续修改是否合法仍然至关重要。
A lifetime is a compile-time region that bounds how long a reference may be used. A reference cannot outlive the value it refers to, and borrow permissions return to the owner when the borrow ends.
In modern Rust, borrow extents are often inferred via non-lexical lifetimes, meaning the precise end point of a borrow is not necessarily obvious from surface syntax, but is still crucial for deciding whether later mutations are legal.

% 对于测试用例生成，这些规则对调用序列施加了约束：生成器必须跟踪一个值是否仍然属于某个所有者、是否已被移动、当前是否处于共享借用状态或是否处于可变借用状态，并且必须确保引用的生命周期不会超过其来源。
For test-case generation, these rules impose constraints on call sequences: a generator must track whether a value is still owned, has been moved, is currently shared-borrowed, or is mutably borrowed, and it must ensure that references do not outlive their provenance.

\subsection{Generics and Trait Bounds in Signatures}
\label{sec:background:generics}
% Rust API 通常是泛型的。函数可以量化类型参数，并使用 trait 边界（可能在 `where` 子句中）对其进行约束。例如，签名 `fn f<T: Trait>(x: T) -> U` 将 `T` 的可实例化限制为实现了 `Trait` 的类型。从自动化测试生成的角度来看，这意味着生成一个函数调用需要两个步骤：(1) 为泛型参数选择具体的实例化，以及(2) 提供类型与实例化签名匹配的值。如果步骤 (1) 失败，则生成的程序将无法通过类型检查，无论运行时行为如何。
Rust APIs are frequently generic.
A function may quantify over type parameters and constrain them with trait bounds possibly in \texttt{where} clauses.
For example, a signature \texttt{fn f<T: Trait>(x: T) -> U} restricts admissible instantiations of \texttt{T} to types implementing \texttt{Trait}.
From the perspective of automated test generation, this means that producing a call is a two-step task:
(1) choose concrete instantiations for generic parameters, and
(2) supply values whose types match the instantiated signature.
If step (1) fails, the generated program will not type-check, regardless of runtime behavior.

\section{Pushdown Colored Petri Nets for Rust API Test Generation}
\label{sec:method}
% 1) 形式化定义下推着色 Petri 网（PCPN）；
% 2) 形式化给出 Rust（以函数签名为中心）到 PCPN 的映射；
% 3) 覆盖 move/borrow/mut/指针、生命周期（栈绑定）、泛型偏序、复合类型字段(重)借用、Trait/关联类型等。

\subsection{Problem Setting}
\label{sec:method:setting}
% 设$\mathcal{C}$为一个 Rust 包。我们可以从编译器前端提取出一个有限的可调用项集合$\mathcal{F}$（包含自由函数与方法），并附带它们的签名信息，具体包括：(i) 输入 / 输出类型；(ii) 引用可变性；(iii) 若存在显式生命周期参数，则包含该参数及其生存期约束；(iv) 带有 trait 约束与关联类型约束的泛型参数。
Let $\mathcal{C}$ be a Rust crate. We can extract, from the compiler front-end, a finite set of \emph{callable items}
$\mathcal{F}$ (free functions and methods) together with their signatures, including (i) input/output types, (ii) reference mutability, (iii) explicit lifetime parameters and outlives constraints when present, and (iv) generic parameters with trait bounds and associated-type constraints.

% 我们的目标是生成可编译的 Rust测试桩，这类测试桩能够构造带类型的值，并执行针对$\mathcal{F}$中各项的调用序列。我们刻意避免对函数体的内部状态进行建模；相反，我们将每一次调用视为一种原子变换，其编译期合法性由签名层面的所有权、借用、生命周期与泛型约束共同决定。
Our goal is to generate compilable Rust harnesses that construct typed values and execute sequences of calls to items in $\mathcal{F}$. We deliberately avoid modeling internal states of function bodies; instead, each call is treated as an atomic transformation whose compile-time validity is determined by the signature-level ownership/borrowing/lifetime/generic constraints.

\subsection{Petri Net Preliminaries}
\label{sec:method:pn}
\begin{definition}[Net and Petri Net]
\label{def:pn}
A net is a $3$-tuple $N=(P,T,F)$, where $P$ is a finite set of places, $T$ is a finite set of transitions,
$F\subseteq (P\times T) \cup (T\times P)$ is a finite set of arcs, and $P\cap T=\emptyset$.
A net can be viewed as a directed bipartite graph where places are represented by circles and transitions by rectangles.
The pre-set and post-set of a node $x \in\{P \cup T\}$ are defined as
$^{\bullet}x=\{y\in P\cup T\mid (y,x)\in F\}$ and $x^{\bullet}=\{y\in P\cup T\mid (x,y)\in F\}$, respectively.
In this paper, it is assumed that the weight of each directed arc is $1$.

A marking of $N=(P,T,F)$ is a mapping $M:P\rightarrow \mathbb{N}$ where $M(p)$ is the number of tokens in place $p$.
A net $N$ with an initial marking $M_{0}$ is called a Petri Net and denoted as $(N,M_{0})$.
Transition $t$ is enabled at $M$ if $\forall p\in {}^{\bullet}t: M(p)>0$, denoted as $M[t\rangle$.
A marking $M$ is a deadlock marking if any transition is disabled at $M$.
Firing an enabled transition $t$ yields a new marking $M^{\prime}$:
$M^{\prime}(p)=M(p)-1$ if $p\in {}^{\bullet}t\backslash t^{\bullet}$;
$M^{\prime}(p)=M(p)+1$ if $p\in t^{\bullet}\backslash {}^{\bullet}t$;
and otherwise $M^{\prime}(p)=M(p)$.
This is denoted as $M[t\rangle M^{\prime}$.
A marking $M_{k}$ is reachable from $M$ if $M_{k}=M$ or there exists a non-empty transition sequence
$\sigma=t_{1}t_{2}\dots t_{k}$ such that
$M[t_{1}\rangle M_{1}[t_{2}\rangle \dots M_{k-1}[t_{k}\rangle M_{k}$ (abbreviated as $M[\sigma \rangle M_{k}$).
All markings reachable from $M_{0}$ in a Petri net is written as $\mathcal{M}$.
This paper only considers bounded Petri nets, i.e.,
$\exists k\in \mathbb{N},\forall{M}\in \mathcal{M}:M(p)\leq k$.
\end{definition}

\subsection{Pushdown Colored Petri Nets (PCPN)}
\label{sec:method:pcpn}
% PCPN 是对上面 PN 的扩展：
% (i) token 带颜色(携带变量 id、泛型/生命周期实例化等)；
% (ii) 引入全局栈用于建模 borrow/lifetime 的嵌套与归还；
% (iii) 为了表达同一 place 多重输入（同型多个参数），使用标准的“加权弧/平行弧”扩展。

% 设GTy为从代码集合中提取的单态 Rust 类型集合。设TVar为泛型类型变量集合，LVar为生命周期变量集合（如'a 'static）。设Sigma{C}表示从C中提取的签名 / 类型环境（注：存储函数签名、结构体定义等全局类型信息，如fn f<'a>(x: &'a i32) -> &'a i32）。
Let $\mathsf{GTy}$ be the set of monomorphic Rust types extracted from $\mathcal{C}$. Let $\mathsf{TVar}$ be a set of generic type variables and $\mathsf{LVar}$ a set of lifetime variables. Let $\Sigma(\mathcal{C})$ denote the extracted signature/type environment.

We use the following type-expression language:
\begin{align*}
\tau \in \mathsf{Ty} ::=~&
g \mid \alpha \mid \tau_0\langle \tau_1,\dots,\tau_n\rangle \mid (\tau_1,\dots,\tau_n) \mid [\tau] \mid \\
& \mathsf{Ref}^q_{\ell}(\tau) \mid \mathsf{Assoc}(\tau,\mathsf{Tr},A) \mid \mathsf{Field}(\tau,f) \mid \mathsf{Impl}(\tau,\mathsf{Tr}),
\end{align*}
where $g\in\mathsf{GTy}$, $\alpha\in\mathsf{TVar}$, $q\in\{\mathsf{shr},\mathsf{mut}\}$,
$\ell\in\mathsf{LVar}$, $\mathsf{Tr}$ ranges over traits, $A$ over associated type names, and $f$ over fields/projection paths.

% 解决 Rust 隐式借用的生命周期追踪问题 —— 比如闭包捕获变量时生成的隐式借用、编译器自动插入的临时借用，这些借用没有用户显式声明的生命周期，需要一个专用变量承载其运行时区域标签，确保借用规则的完整性。
Let $\ell_{\bullet}\in \mathsf{LVar}$ be a reserved lifetime variable that does not appear in extracted signatures.
All internally created references are typed with $\ell_{\bullet}$ and carry the actual region label via $\lambda(\ell_{\bullet})$.

Let $\mathsf{Cap}=\{\mathsf{own},\mathsf{frz},\mathsf{blk},\mathsf{val}\}$.
We define the place set:
\[
P ~=~ \{\, p_{\kappa,\tau} \mid \kappa\in\mathsf{Cap},~ \tau\in\mathsf{Ty} \,\}.
\]
Intuitively, $p_{\mathsf{own},\tau}$ stores owned values;
% p_frz存储被共享借用冻结的所有权值（如 let r = &x后，x被冻结，不能再取可变引用）
$p_{\mathsf{frz},\tau}$ stores owned values frozen by shared borrows;
% p_blk存储被可变借用阻塞的所有权值（如 let mr = &mut x后，x被阻塞，不能再取任何引用）
$p_{\mathsf{blk},\tau}$ stores owned values blocked by a mutable borrow;
% p_val存储非所有权值（如引用&x、trait 实现证据等，本身不拥有数据）
$p_{\mathsf{val},\tau}$ stores non-owning values including references, witnesses, etc.

% 设V 为可数无穷的值标识符集合（即新鲜的 Rust 变量名，如x y temp_1）；
Let $\mathbb{V}$ be a countably infinite set of value identifiers (fresh Rust variable names),
% L为可数无穷的 区域标签集合（运行时真实的生命周期区域，如函数调用栈帧对应的区域）。
and $\mathbb{L}$ a countably infinite set of region labels.
% 设Theta为有限类型替换集合：theta是从泛型变量TVar到单态类型GTy的部分映射（如\theta(T)=i32，表示将泛型T实例化为i32）；设Lambda为有限生命周期替换集合：lambda是从生命周期变量LVar到区域标签L的部分映射（如lambda('a)=L_1，表示将生命周期'a绑定到运行时区域L_1。
Let $\Theta$ be finite type substitutions $\theta:\mathsf{TVar}\rightharpoonup \mathsf{GTy}$,
and $\Lambda$ finite lifetime substitutions $\lambda:\mathsf{LVar}\rightharpoonup \mathbb{L}$.

A token color is
\begin{equation*}
    c ~=~ \langle v,\theta,\lambda\rangle \in \mathsf{Col} := \mathbb{V}\times\Theta\times\Lambda.
\end{equation*}

% 通过 带颜色的令牌 追踪每个变量的动态信息（谁、类型是什么、生命周期绑定到哪个运行时区域），解决 Petri 网 无法区分不同变量 的问题 —— 比如两个i32变量，通过颜色中的v和lambda区分，确保借用检查时能精准匹配 变量 - 生命周期 - 区域。
A colored marking is
\begin{equation*}
    M: P \to \mathbb{N}^{(\mathsf{Col})},
\end{equation*}
i.e., each place carries a finite multiset of colored tokens.

\paragraph{Pushdown stack.}
Let $\Gamma$ be a stack alphabet:
\begin{equation*}
    \gamma ::= \mathsf{Freeze}(v)\mid \mathsf{Shr}(v_o,v_r,\ell)\mid \mathsf{Mut}(v_o,v_r,\ell),
\end{equation*}
where $\ell\in\mathbb{L}$ is a region label.
A configuration is $\mathsf{Cfg}=\langle M,S\rangle$ where $S\in \Gamma^*$ (top at the left).

We model arc multiplicities by a weight function
\begin{equation*}
    W: (P\times T)\cup (T\times P)\to \mathbb{N}
\end{equation*}
with finite support, and let $F=\{(x,y)\mid W(x,y)>0\}$.
For a transition $t$, we write $W(p,t)$ and $W(t,p)$ for input/output multiplicities.

A \emph{choice} for $t$ at marking $M$ is a multiset-valued function
\begin{equation*}
    \chi:P\to \mathbb{N}^{(\mathsf{Col})}
    \quad\text{s.t.}\quad
    \forall p\in P:\ \chi(p)\subseteq M(p)\ \wedge\ |\chi(p)| = W(p,t).
\end{equation*}

% 实例化信息绑定
Let $\iota$ range over instantiations, containing at least:
\begin{equation*}
    \iota=(\theta^{*},\lambda^{*},\nu,\mu,\dots)
\end{equation*}
where $\theta^{*}\in\Theta$ and $\lambda^{*}\in\Lambda$ are the unified substitutions, and $\nu$ and $\mu$ provide any fresh value ids / fresh region labels needed by $\mathsf{Out}$. Other resolved information such as associated-type resolution may also be stored.

Each transition $t$ carries:
\begin{align*}
    & \mathsf{Guard}(t)(\langle M,S\rangle,\chi,\iota),\qquad \\
    & \mathsf{Out}(t)(\langle M,S\rangle,\chi,\iota):P\to \mathbb{N}^{(\mathsf{Col})},\qquad \\
    & \mathsf{Act}(t)\in\{\epsilon,\mathsf{push}(\bar\gamma),\mathsf{pop}(\bar\gamma)\}.
\end{align*}

\subsection{Generics and Constraint Solving}
\label{sec:method:generics}

To keep the net bounded while still allowing generic instantiation, we fix a finite set $\mathsf{GTy}^{\dagger}\subseteq \mathsf{GTy}$ of ground Rust types considered by the generator.
Let $\widehat{\mathsf{Ty}}$ be the finite closure of $\mathsf{GTy}^{\dagger}$ under the
type constructors that we model, such as $\mathsf{Ref}^q_{\ell}(\cdot)$, tuples, arrays, and any finite set of crate-local projections forms, so that all runtime values produced by the net have types in $\widehat{\mathsf{Ty}}$.

For substitutions $\theta:\mathsf{TVar}\rightharpoonup \mathsf{GTy}^{\dagger}$ and
$\lambda:\mathsf{LVar}\rightharpoonup \mathbb{L}$, let $(\theta,\lambda)(\tau)$ denote capture-avoiding simultaneous substitution on type expressions.

An obligation is one of:
\begin{equation*}
    o ::=~ \tau:\mathsf{Tr}
\mid \mathsf{Assoc}(\tau,\mathsf{Tr},A)=\tau'
\mid \ell_1:\ell_2,
\end{equation*}
and a constraint set $\rho$ is a finite set of obligations.

We assume an entailment judgement $\Sigma(\mathcal{C})\vdash o$ meaning obligation $o$ is derivable from the crate.
We write
\begin{equation*}
    \Sigma(\mathcal{C})\vdash (\theta,\lambda)\models \rho
\quad\text{iff}\quad
\forall o\in \rho:\ \Sigma(\mathcal{C})\vdash (\theta,\lambda)(o).
\end{equation*}
This captures supertraits, where-clauses, and associated-type equalities in a uniform way
without explicit monomorphization.

Let $\mathsf{Unify}(\tau,\hat\tau)$ attempt to unify a type scheme $\tau\in\mathsf{Ty}$
with a ground type $\hat\tau\in\widehat{\mathsf{Ty}}$ and return a substitution pair $(\theta,\lambda)$, or $\bot$ if unification fails. Unification is structural on the type-expression syntax including $\mathsf{Ref}$ and projections, and is merged across multiple arguments in the standard way to obtain a single $(\theta^{*},\lambda^{*})$ when possible.

A transition corresponding to a generic API is enabled only if there exists a grounding
$(\theta^{*},\lambda^{*})$ such that:
(i) all chosen argument tokens' types unify with the signature parameter types under $(\theta^{*},\lambda^{*})$;
(ii) $\Sigma(\mathcal{C})\vdash (\theta^{*},\lambda^{*})\models \rho$.
This delays generic instantiation to firing time, avoiding up-front state explosion.


\subsection{Constructing a Rust PCPN from $\mathcal{C}$}
\label{sec:method:construct}

We construct a crate-specific PCPN
\begin{equation*}
    \mathcal{N}_{\mathcal{C}}=(P,T,W,\mathsf{Col},\Gamma,M_0,\mathsf{Guard},\mathsf{Out},\mathsf{Act}),
\end{equation*}
where $P$ is indexed by admissible ground types $\widehat{\mathsf{Ty}}$, and $M_0$ seeds a finite set of constructors producing tokens of types in $\widehat{\mathsf{Ty}}$.

We write $\mathsf{IsCopy}(\hat\tau)$ iff $\Sigma(\mathcal{C})\models \hat\tau:\mathsf{Copy}$.

For each callable item $f\in\mathcal{F}$ with signature
\begin{equation*}
    \mathsf{sig}(f)=\langle \bar\alpha;\bar\ell;\ \tau_1,\dots,\tau_n \to \tau_o;\ \rho\rangle,
\end{equation*}
we add a transition $t_f$.
The net structure of $t_f$ only specifies argument modes, while the concrete ground types of consumed/produced places are determined by a satisfiable instantiation $(\theta^{*},\lambda^{*})$ in the guard.

Let $\hat\tau_i$ denote the ground type of the chosen argument token for parameter $i$.
The mode constraints are:
\begin{itemize}
\item If parameter $i$ is by-value $\tau_i$, then the chosen token must come from
      $p_{\mathsf{own},\hat\tau_i}$.
      If $\mathsf{IsCopy}(\hat\tau_i)$, we model by-value passing as a self-loop (token reproduced).
\item If parameter $i$ is shared reference $\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)$, then the chosen token must come from
      $p_{\mathsf{val},\hat\tau_i}$ where $\hat\tau_i$ is a ground shared-ref type; and we add a self-loop since $\&T$ is Copy.
\item If parameter $i$ is mutable reference $\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)$, then the chosen token must come from
      $p_{\mathsf{val},\hat\tau_i}$ where $\hat\tau_i$ is a ground mutable-ref type.
      \emph{Rust-like default: implicit reborrow.}
      We treat this parameter as a self-loop (the reference remains usable after the call)
      \emph{unless} the reference must be moved because its lifetime escapes through outputs.
\end{itemize}

Let $\mathsf{FV}_{\ell}(\tau)$ return the set of lifetime variables appearing in $\tau$.
For a mutable-ref parameter $\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)$, define
\begin{equation*}
    \mathsf{Esc}(\ell, \tau_o) ~\equiv~ \ell\in \mathsf{FV}_{\ell}(\tau_o).
\end{equation*}
If $\mathsf{Esc}(\ell,\tau_o)$ holds, we do not self-loop that \&mut argument;
otherwise, we self-loop it to model implicit re-borrow in ordinary calls.

$\mathsf{Guard}(t_f)$:
(i) picks ground argument types $\hat\tau_1,\dots,\hat\tau_n\in\widehat{\mathsf{Ty}}$ by selecting tokens from the corresponding
typed places under the above mode constraints,
(ii) computes a unified instantiation $(\theta^{*},\lambda^{*})$ such that
$\forall i:\ \mathsf{Unify}(\tau_i,\hat\tau_i)\neq\bot$ and all unifiers are compatible, and
(iii) checks $\Sigma(\mathcal{C})\vdash (\theta^{*},\lambda^{*})\models\rho$.

$\mathsf{Out}(t_f)$ allocates fresh ids for returned values and reproduces tokens on self-looped arguments.
The grounded return type is $\hat\tau_o=(\theta^{*},\lambda^{*})(\tau_o)$, required to be in $\widehat{\mathsf{Ty}}$.
By default $\mathsf{Act}(t_f)=\epsilon$.

\subsection{Ownership and Copy as Token Flow}
\label{sec:method:ownership}

% 为了简洁，定义 color 的投影函数
Let $\pi_V(\langle v,\theta,\lambda\rangle)=v$, $\pi_\Theta(\cdot)=\theta$, and $\pi_\Lambda(\cdot)=\lambda$.
For a singleton multiset $\{c\}$, write $\mathsf{one}(\{c\})=c$.

\paragraph{(1) Move for non-Copy owned values.}
For each type $\tau$, add a transition $\mathsf{Move}_\tau$ with:
\[
W(p_{\mathsf{own},\tau},\mathsf{Move}_\tau)=1,
\]
and no outputs. Define:
\begin{align*}
\mathsf{Guard}(\mathsf{Move}_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ \neg \mathsf{IsCopy}(\tau),\\
\mathsf{Out}(\mathsf{Move}_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ \emptyset,\\
\mathsf{Act}(\mathsf{Move}_\tau)
&~=~ \epsilon.
\end{align*}

\paragraph{(2) By-value use for Copy owned values (self-loop).}
For each $\tau$, add $\mathsf{CopyUse}_\tau$:
\begin{equation*}
    W(p_{\mathsf{own},\tau},\mathsf{CopyUse}_\tau)=1,\qquad
    W(\mathsf{CopyUse}_\tau,p_{\mathsf{own},\tau})=1.
\end{equation*}
Let $c=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$. Define:
\begin{align*}
\mathsf{Guard}(\mathsf{CopyUse}_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ \mathsf{IsCopy}(\tau),\\
\mathsf{Out}(\mathsf{CopyUse}_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau}\mapsto \{c\}\,],\\
\mathsf{Act}(\mathsf{CopyUse}_\tau)
&~=~ \epsilon.
\end{align*}
% 解释：Copy 类型按值使用不 move，token 仍留在 own place。

% 用于建模 `let y = x;`，会增长 token 数，可能破坏 bounded；需要用预算/上界控制。
Fix a per-type copy budget $B_{\mathsf{copy}}(\tau)\in\mathbb{N}$.
Add $\mathsf{DupCopy}_\tau$:
\begin{equation*}
    W(p_{\mathsf{own},\tau},\mathsf{DupCopy}_\tau)=1,\qquad
    W(\mathsf{DupCopy}_\tau,p_{\mathsf{own},\tau})=2.
\end{equation*}
Let $c=\langle v,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$ and let $\nu(\mathsf{new})=v'$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(\mathsf{DupCopy}_\tau)(\langle M,S\rangle,\chi,\iota)
&\equiv \mathsf{IsCopy}(\tau) \\
&\quad \wedge\ |M(p_{\mathsf{own},\tau})| < B_{\mathsf{copy}}(\tau)\\ &\quad \wedge\ v'\notin \mathsf{Ids}(M,S),\\
\mathsf{Out}(\mathsf{DupCopy}_\tau)(\langle M,S\rangle,\chi,\iota)
&= [\,p_{\mathsf{own},\tau}\mapsto \{\langle v,\theta,\lambda\rangle,\ \langle v',\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(\mathsf{DupCopy}_\tau)
&= \epsilon.
\end{align*}
% 其中 $\mathsf{Ids}(M,S)$ 表示 marking 和 stack 中出现过的 value ids。

\paragraph{(3) Drop.}
For each place $p_{\kappa,\tau}$, add $\mathsf{Drop}_{\kappa,\tau}$:
\[
W(p_{\kappa,\tau},\mathsf{Drop}_{\kappa,\tau})=1,
\]
and no outputs. Define:
\begin{align*}
\mathsf{Guard}(\mathsf{Drop}_{\kappa,\tau})(\langle M,S\rangle,\chi,\iota) &~\equiv~ \top,\\
\mathsf{Out}(\mathsf{Drop}_{\kappa,\tau})(\langle M,S\rangle,\chi,\iota) &~=~ \emptyset,\\
\mathsf{Act}(\mathsf{Drop}_{\kappa,\tau}) &~=~ \epsilon.
\end{align*}

\paragraph{(4) First shared borrow from an owned value.}
Add $\mathsf{BorrowShrOwn}_\tau$ as $T_\tau$: $W(p_{\mathsf{own},\tau},T_\tau)$=$W(T_\tau,p_{\mathsf{frz},\tau})$=$W(T_\tau,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)})$=1.

Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$ and let $\nu(\mathsf{ref})=v_r$, $\mu(\mathsf{lab})=\ell$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_r\notin \mathsf{Ids}(M,S)\ \wedge\ \ell\notin \mathsf{Labs}(M,S),\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{frz},\tau}\mapsto \{c_o\},\\
&\qquad\ \ p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)}\mapsto \{\langle v_r,\theta,\lambda[\ell_{\bullet}\mapsto \ell]\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{push}(\mathsf{Shr}(v_o,v_r,\ell)\cdot \mathsf{Freeze}(v_o)).
\end{align*}

\paragraph{(5) Additional shared borrow while already frozen.}
Add $\mathsf{BorrowShrFrz}_\tau$ as $T_\tau$: 
$W(p_{\mathsf{frz},\tau},T_\tau)$=$W(T_\tau,p_{\mathsf{frz},\tau})$=$W(T_\tau,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)})$=1.
Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{frz},\tau}))$ and let $\nu(\mathsf{ref})=v_r$, $\mu(\mathsf{lab})=\ell$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_r\notin \mathsf{Ids}(M,S)\ \wedge\ \ell\notin \mathsf{Labs}(M,S),\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{frz},\tau}\mapsto \{c_o\},\\
&\qquad\ \ p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)}\mapsto \{\langle v_r,\theta,\lambda[\ell_{\bullet}\mapsto \ell]\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{push}(\mathsf{Shr}(v_o,v_r,\ell)).
\end{align*}

\paragraph{(6) Mutable borrow from an owned value.}
Add $\mathsf{BorrowMut}_\tau$ as $T_\tau$: $W(p_{\mathsf{own},\tau},T_\tau)$=$W(T_\tau,p_{\mathsf{blk},\tau})$=$W(T_\tau,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau)})$=1.
Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$ and let $\nu(\mathsf{ref})=v_r$, $\mu(\mathsf{lab})=\ell$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_r\notin \mathsf{Ids}(M,S)\ \wedge\ \ell\notin \mathsf{Labs}(M,S),\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{blk},\tau}\mapsto \{c_o\},\\
&\qquad\ \ p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau)}\mapsto \{\langle v_r,\theta,\lambda[\ell_{\bullet}\mapsto \ell]\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{push}(\mathsf{Mut}(v_o,v_r,\ell)).
\end{align*}

\paragraph{(7) Ending a mutable borrow (restore owner).}
Add $\mathsf{EndMut}_\tau$ as $T_\tau$:
$W(p_{\mathsf{blk},\tau},T_\tau)$=$W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau)},T_\tau)$=$W(T_\tau,p_{\mathsf{own},\tau})$=1.

Let $c_o=\langle v_o,\theta_o,\lambda_o\rangle=\mathsf{one}(\chi(p_{\mathsf{blk},\tau}))$ and
$c_r=\langle v_r,\theta_r,\lambda_r\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau)}))$.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ S = \mathsf{Mut}(v_o,v_r,\ell)\cdot S' \\
&\quad \wedge\ \theta_o=\theta_r\ \wedge\ \lambda_o=\lambda_r,\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau}\mapsto \{\langle v_o,\theta_o,\lambda_o\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{pop}(\mathsf{Mut}(v_o,v_r,\ell)).
\end{align*}

\paragraph{(8) Ending a shared borrow (non-last).}
Add $\mathsf{EndShrKeep}_\tau$ as $T_\tau$:
$W(p_{\mathsf{frz},\tau},T_\tau)$=$W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)},T_\tau)$=$W(T_\tau,p_{\mathsf{frz},\tau})$=1.

Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{frz},\tau}))$ and
$c_r=\langle v_r,\theta',\lambda'\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)}))$.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ S=\mathsf{Shr}(v_o,v_r,\ell)\cdot S' \\
&\quad \wedge\ \theta=\theta' \ \wedge\ \lambda=\lambda' \\ 
&\quad \wedge \big(S'=\epsilon\ \vee\ \mathsf{top}(S')\neq \mathsf{Freeze}(v_o)\big),\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{frz},\tau}\mapsto \{\langle v_o,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{pop}(\mathsf{Shr}(v_o,v_r,\ell)).
\end{align*}

\paragraph{(9) Ending a shared borrow (last, restore owner).}
Add $\mathsf{EndShrLast}_\tau$ as $T_\tau$:
$W(p_{\mathsf{frz},\tau},T_\tau)$=$W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)},T_\tau)$=$W(T_\tau,p_{\mathsf{own},\tau})$=1.

Let $c_o=\langle v_o,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{frz},\tau}))$ and
$c_r=\langle v_r,\theta',\lambda'\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell_{\bullet}}(\tau)}))$.
Define:
\begin{align*}
\mathsf{Guard}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~\equiv~ S=\mathsf{Shr}(v_o,v_r,\ell)\cdot \mathsf{Freeze}(v_o)\cdot S' \\
&\quad \wedge\ \theta=\theta' \ \wedge\ \lambda=\lambda',\\
\mathsf{Out}(T_\tau)(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau}\mapsto \{\langle v_o,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_\tau)
&~=~ \mathsf{pop}(\mathsf{Shr}(v_o,v_r,\ell)\cdot \mathsf{Freeze}(v_o)).
\end{align*}

\subsection{Composite Types and Field Re-Borrowing}
\label{sec:method:composite}

\paragraph{(10) Field projection by move.}
% 保守：move 出字段视为消耗整个 owner（避免 partial move 复杂性）
Let $\tau$ be a struct type with field $f:\tau_f$.
Add $\mathsf{ProjMove}_{\tau,f}$ as $T_{\tau,f}$: 
\begin{equation*}
W(p_{\mathsf{own},\tau},T_{\tau,f})=W(T_{\tau,f},p_{\mathsf{own},\tau_f})=1.
\end{equation*}

Let $c=\langle v,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$ and let $\nu(\mathsf{fld})=v_f$ be fresh.
Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_f\notin \mathsf{Ids}(M,S),\\
\mathsf{Out}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau_f}\mapsto \{\langle v_f,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_{\tau,f})
&~=~ \epsilon.
\end{align*}

\paragraph{(11) Field reborrow from a shared reference (Copy, keep parent).}
% &S -> &S.f：shared ref 是 Copy，因此 parent token 不应被消耗
Add $\mathsf{ProjShr}_{\tau,f}$ as $T_{\tau,f}$:
\begin{equation*}
    W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)},T_{\tau,f})=W(T_{\tau,f},p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)})=W(T_{\tau,f},p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau_f)})=1.
\end{equation*}

Let $c_p=\langle v_p,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)}))$
and let $\nu(\mathsf{child})=v_c$ be fresh. Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_c\notin \mathsf{Ids}(M,S),\\
\mathsf{Out}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau)}\mapsto \{c_p\},\\
&\qquad\ \ p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}_{\ell}(\tau_f)}\mapsto \{\langle v_c,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_{\tau,f})
&~=~ \epsilon.
\end{align*}

\paragraph{(12) Field reborrow from a mutable reference (reborrowing, block parent ref).}
% &mut S -> &mut S.f：子借用活着时父 &mut 不可用，因此消耗 parent ref token，并用栈 frame 恢复
Add $\mathsf{ProjMut}_{\tau,f}$ as $T_{\tau,f}$: 
\begin{equation*}
    W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)},T_{\tau,f})=W(T_{\tau,f},p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau_f)})=1.
\end{equation*}

Let $c_p=\langle v_p,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)}))$
and let $\nu(\mathsf{child})=v_c$, $\mu(\mathsf{lab})=\ell_c$ be fresh. Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ v_c\notin \mathsf{Ids}(M,S)\ \wedge\ \ell_c\notin \mathsf{Labs}(M,S),\\
\mathsf{Out}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau_f)}\mapsto
      \{\langle v_c,\theta,\lambda[\ell_{\bullet}\mapsto \ell_c]\rangle\}\,],\\
\mathsf{Act}(T_{\tau,f})
&~=~ \mathsf{push}(\mathsf{Mut}(v_p,v_c,\ell_c)).
\end{align*}

\paragraph{(13) Ending a mutable reborrow (restore parent ref).}
% pop Mut(parent,child,ell_c) 并恢复 parent &mut token
Add $\mathsf{EndProjMut}_{\tau,f}$ as $T_{\tau,f}$:
\begin{equation*}
    W(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau_f)},T_{\tau,f})=W(T_{\tau,f},p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)})=1.
\end{equation*}

Let $c_c=\langle v_c,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell_{\bullet}}(\tau_f)}))$.
Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ S=\mathsf{Mut}(v_p,v_c,\ell_c)\cdot S',\\
\mathsf{Out}(T_{\tau,f})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}_{\ell}(\tau)}\mapsto \{\langle v_p,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(T_{\tau,f})
&~=~ \mathsf{pop}(\mathsf{Mut}(v_p,v_c,\ell_c)).
\end{align*}

\subsection{Traits and Associated Types}
\label{sec:method:traits}

\paragraph{(14) Trait witness production.}
% 生成一个逻辑见证 token；值本身保留（self-loop）
If $\Sigma(\mathcal{C})\models \tau:\mathsf{Tr}$, add $\mathsf{Impl}_{\tau,\mathsf{Tr}}$:
\begin{equation*}
W(p_{\mathsf{own},\tau},\mathsf{Impl}_{\tau,\mathsf{Tr}})=W(\mathsf{Impl}_{\tau,\mathsf{Tr}},p_{\mathsf{own},\tau})=1
\end{equation*}

$W(\mathsf{Impl}_{\tau,\mathsf{Tr}},p_{\mathsf{val},\mathsf{Impl}(\tau,\mathsf{Tr})})=1.$

Let $c=\langle v,\theta,\lambda\rangle=\mathsf{one}(\chi(p_{\mathsf{own},\tau}))$. Define:
\begin{align*}
\mathsf{Guard}(\mathsf{Impl}_{\tau,\mathsf{Tr}})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ \Sigma(\mathcal{C})\models \tau:\mathsf{Tr},\\
\mathsf{Out}(\mathsf{Impl}_{\tau,\mathsf{Tr}})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{own},\tau}\mapsto \{c\},\\
&\quad p_{\mathsf{val},\mathsf{Impl}(\tau,\mathsf{Tr})}\mapsto \{\langle v,\theta,\lambda\rangle\}\,],\\
\mathsf{Act}(\mathsf{Impl}_{\tau,\mathsf{Tr}})
&~=~ \epsilon.
\end{align*}

\paragraph{(15) Associated type resolution as retyping.}
% 仅改变静态类型标签：Assoc(...) -> tau'
If $\Sigma(\mathcal{C}) \models $ \\
$\mathsf{Assoc}(\tau,\mathsf{Tr},A)=\tau'$, add $\mathsf{AssocCast}_{\tau,\mathsf{Tr},A}$ as $T_{\tau,\mathsf{Tr},A}$:
\begin{equation*}
    W(p_{\mathsf{val},\mathsf{Assoc}(\tau,\mathsf{Tr},A)},T_{\tau,\mathsf{Tr},A})=
W(T_{\tau,\mathsf{Tr},A},p_{\mathsf{val},\tau'})=1.
\end{equation*}

Let $c=\mathsf{one}(\chi(p_{\mathsf{val},\mathsf{Assoc}(\tau,\mathsf{Tr},A)}))$. Define:
\begin{align*}
\mathsf{Guard}(T_{\tau,\mathsf{Tr},A})(\langle M,S\rangle,\chi,\iota)
&~\equiv~ \Sigma(\mathcal{C})\models \mathsf{Assoc}(\tau,\mathsf{Tr},A)=\tau',\\
\mathsf{Out}(T_{\tau,\mathsf{Tr},A})(\langle M,S\rangle,\chi,\iota)
&~=~ [\,p_{\mathsf{val},\tau'}\mapsto \{c\}\,],\\
\mathsf{Act}(T_{\tau,\mathsf{Tr},A})
&~=~ \epsilon.
\end{align*}

\subsection{Soundness Intuition}
\label{sec:method:soundness}
% 根据构造方式，Nc中的任意变迁序列均对应一个满足以下条件的测试套：(i) 被移动的自有值不会被重复使用（它们会从库所$p_{\mathsf{own},\tau}$中被消耗）；(ii) 可变性别名的产生被严格禁止（在$\mathsf{Mut}$帧的作用下，自有值的所属权会被转移至库所$p_{\mathsf{blk},\tau}$）；(iii) 共享别名场景下的突变操作被严格禁止（在$\mathsf{Freeze}/\mathsf{Shr}$帧的作用下，自有值的所属权会被托管至库所$p_{\mathsf{frz},\tau}$）；(iv) 生命周期具备合法的嵌套结构，并通过栈弹出变迁完成生命周期的终结。泛型实例化过程由可满足性关系$\Vdash$关联类型解析机制共同约束。因此，在该保守式结构化借用规则的约束下，所生成的测试套能够被 Rust 编译器正常接受。
By construction, any transition sequence in $\mathcal{N}_{\mathcal{C}}$ corresponds to a harness in which:
(i) moved owned values are not reused, they are consumed from $p_{\mathsf{own},\tau}$,
(ii) mutable aliasing is prevented. owners are moved to $p_{\mathsf{blk},\tau}$ under $\mathsf{Mut}$ frames,
(iii) mutation under shared aliasing is prevented. owners reside in $p_{\mathsf{frz},\tau}$ under $\mathsf{Freeze}/\mathsf{Shr}$ frames. and
(iv) lifetimes are properly nested and ended through stack-pop transitions.
Generic instantiation is guarded by the satisfiability relation $\Vdash$ and associated-type resolution.
Therefore, generated harnesses are expected to be accepted by the Rust compiler under this conservative, structured borrowing discipline.

\section{Bounded Reachability Analysis and Trusted Snippet Synthesis}
\label{sec:analysis}
% 本节构建分析与合成层，将面向特定包的下推着色 Petri 网（PCPN）$\mathcal{N}_{\mathcal{C}}$转化为构造正确的 Rust 代码片段。此时，该 Petri 网模型已完成对所有权传递、借用能力与生命周期的签名级建模。后续核心任务为在网中搜索满足用户目标的执行路径，并生成具备自包含特性的程序片段。
This section develops the analysis and synthesis layer that turns a crate-specific PTPN $\mathcal{N}_{\mathcal{C}}$ into correct-by-construction Rust snippets. At this point, the Petri-net already captures the signature-level movement of ownership, borrowing capabilities, and lifetimes. What remains is to search the net for executions that satisfy a user objective and to emit a self-contained program fragment.

\subsection{Synthesis objectives as reachability analysis}
\label{sec:analysis:objectives}
% 采用 Petri 网语义的核心优势在于各类合成任务均可被统一表示为有界可达性查询
A key benefit of using a Petri-net semantics is that a broad class of synthesis tasks can be expressed uniformly as bounded reachability analysis.

A configuration is a pair $\mathsf{Cfg}=\langle M,S\rangle$ where $M$ is a colored marking and $S$ is a pushdown stack word. Intuitively, $M$ describes the currently available symbolic resources, while $S$ records open structured scopes induced by borrows and reborrows that must eventually be discharged.

We express user intents as a predicate $\Phi$ over canonical configurations.
Common instances include:
\begin{itemize}
\item Type goal: produce a value of type $\hat\tau$ by requiring that $M(p_{\mathsf{own},\hat\tau})\neq \emptyset$ or $M(p_{\mathsf{val},\hat\tau})\neq \emptyset$ for reference-like results.
\item API goal: reach a specific API call by requiring that the witness sequence contains a designated transition.
\item Coverage goal: maximize or hit a threshold of transition coverage over $T_{\mathsf{api}}\subseteq T$.
\item Well-formed snippet goal: require that the configuration is closable, for example, its stack can be discharged to empty.
\end{itemize}

% (i) 如何在显式边界约束下构建有限可达性图；(ii) 如何提取满足谓词的变迁触发证据序列；(iii) 如何通过栈消解操作确定性地闭合所有未完成作用域；(iv) 如何生成自包含 Rust 代码片段，并基于模型证明其可编译性。
The rest of the section shows how to build a finite reachability graph under explicit bounds, extract a witness firing sequence satisfying $\Phi$, deterministically close open scopes by discharging the stack, and emit a self-contained Rust snippet whose compilation is justified by the model.

\subsection{Operational semantics of PCPN executions}
\label{sec:analysis:semantics}

We use the firing semantics from Section~\ref{sec:method}.
Let $\mathsf{Cfg}=\langle M,S\rangle$. A transition firing is witnessed by an instantiated firing rule $\varphi=\langle t,\chi,\iota\rangle$, where $\chi$ selects the consumed tokens and $\iota$ records the concrete instantiation including type substitution, lifetime substitution, and any resolved associated-type information. We write $\mathsf{tr}(\varphi)=t$.

We use multiset sum/difference operators $\oplus$ and $\ominus$ pointwise on places.

\begin{definition}[Enabledness and firing]
\label{def:pcpn:firing}
Let $\mathsf{Cfg}=\langle M,S\rangle$.
A transition $t$ is enabled at $\mathsf{Cfg}$ if there exist
$\chi$ and $\iota$ such that $\mathsf{Guard}(t)(\mathsf{Cfg},\chi,\iota)$ holds and, if $t$ performs a pop action,
its pop-pattern matches the stack top.
A firing rule is $\varphi=\langle t,\chi,\iota\rangle$.

Given $\varphi$, firing yields $\mathsf{Cfg}'=\langle M',S'\rangle$ where:
\begin{itemize}
\item Consume: $M^{-}(p)=M(p)\ominus \chi(p)$ for all $p$.
\item Produce: let $\pi=\mathsf{Out}(t)(\mathsf{Cfg},\chi,\iota)$ with $|\pi(p)|=W(t,p)$, then $M'=M^{-}\oplus \pi$.
\item Stack: $S'$ is obtained by applying $\mathsf{Act}(t)$ (push/pop/$\epsilon$) to $S$.
\end{itemize}
We denote this step as $\mathsf{Cfg}[\varphi\rangle \mathsf{Cfg}'$.
\end{definition}

A firing sequence is a finite sequence $\sigma=\varphi_1\cdots\varphi_n$ such that
$\mathsf{Cfg}_0[\sigma\rangle \mathsf{Cfg}_n$ where $\mathsf{Cfg}_0=\langle M_0,\epsilon\rangle$.

\subsection{Guard solving: unification, lifetimes, traits, and capabilities}
\label{sec:analysis:guard}

For each transition $t\in T$, we structure its guard as:
\begin{align*}
\mathsf{Guard}(t)(\mathsf{Cfg},\chi,\iota)\ \equiv\
&G_t^{\mathsf{uni}}(\chi,\iota)\ \wedge\
G_t^{\mathsf{life}}(\chi,\mathsf{Cfg}.S,\iota) \\
&\wedge\
G_t^{\mathsf{trait}}(\chi,\iota)\ \wedge\
G_t^{\mathsf{cap}}(\chi,\mathsf{Cfg}.S,\iota).
\end{align*}
Not all conjuncts are non-trivial for every transition:
e.g., pure drop/move transitions may have trivial $G_t^{\mathsf{trait}}$,
while borrow-ending transitions heavily rely on $G_t^{\mathsf{life}}$ and $G_t^{\mathsf{cap}}$.

\paragraph{Type unification constraint $G_t^{\mathsf{uni}}$.}
It enforces a consistent substitution for generic type variables induced by
matching signature parameter schemes against the consumed token types in $\chi$,
as well as any syntactic type equalities induced by where-clauses.

\paragraph{Lifetime/stack constraint $G_t^{\mathsf{life}}$.}
It enforces consistency of lifetime substitution and compatibility with the pushdown discipline.
If $t$ performs a pop action, the required pattern must match the current stack top,
enforcing properly nested borrow extents.

\paragraph{Trait and associated-type constraint $G_t^{\mathsf{trait}}$.}
It checks trait bounds and associated-type equalities under the current type/lifetime instantiation,
using the extracted crate environment $\Sigma(\mathcal{C})$.

\paragraph{Capability constraint $G_t^{\mathsf{cap}}$.}
It enforces Rust ownership and borrowing discipline at the level of places and stack frames,
including freezing/blocking effects and admissibility of consuming resources under mutable operations.

\begin{algorithm}[t]
\caption{$\mathsf{SolveGuard}(t,\mathsf{Cfg},\chi)$}
\label{alg:solve-guard}
\begin{algorithmic}[1]
\REQUIRE transition $t$, configuration $\mathsf{Cfg}=\langle M,S\rangle$, choice $\chi$
\ENSURE finite set of admissible instantiations $\mathsf{Inst}(t,\mathsf{Cfg},\chi)$
\STATE $\Theta \gets \mathsf{UnifyTypes}(t,\chi)$ \COMMENT{finite: candidates restricted as in Section~\ref{sec:method:generics}}
\STATE $\Lambda \gets \mathsf{UnifyLifetimes}(t,\chi,S)$
\STATE $\mathsf{Inst}\gets \emptyset$
\FORALL{$(\theta,\lambda)\in \Theta\times \Lambda$}
  \IF{$\mathsf{CheckTraitAssoc}(t,\theta,\lambda)$ \AND $\mathsf{CheckCap}(t,\mathsf{Cfg},\chi,\theta,\lambda)$}
     \STATE $\mathsf{Inst}\gets \mathsf{Inst}\cup\{(\theta,\lambda,\dots)\}$
  \ENDIF
\ENDFOR
\RETURN $\mathsf{Inst}$
\end{algorithmic}
\end{algorithm}

\subsection{Finiteness: budgets, monomorphization, and canonicalization}
\label{sec:analysis:finiteness}
% 下推着色 Petri 网融合了有色令牌、新鲜标识符与下推栈结构，其原始状态空间可能是无限的。通过以下三种手段构造有限的状态探索域：对库所令牌数量与栈深度设置显式预算；限制泛型与生命周期的实例化范围（有界单态化选择）；基于新的名称规范化对配置进行等价划分。有限性的保证是构建可达性图并提取证据序列的关键前提。
The PCPN $\mathcal{N}_{\mathcal{C}}$ combines colored tokens, fresh identifiers, and a pushdown stack, so its raw state space can be infinite. 

We obtain a finite exploration domain by:
(i) explicit budgets on place multiplicities and stack depth,
(ii) finite instantiation domains for generics and lifetimes,
and (iii) quotienting configurations by canonicalization of fresh names.
This finiteness is essential for building a finite reachability graph from which we can extract witness sequences.

% 设B:P->N为库所预算函数，对任意库所p，要求任意变迁触发前均满足|M(p)|<B(p)（避免令牌数量超过预算）。设D\in\mathbb{N}\为栈深度预算，对任意执行压栈操作的变迁，要求触发前栈深度满足|S|<D。上述预算约束为令牌数量与借用嵌套深度设定了明确的有限上界。
Let $B:P\to\mathbb{N}$ be a place budget. For any transition that would increase $|M(p)|$ for some $p$, we require $|M(p)|<B(p)$ before firing. Let $D\in\mathbb{N}$ be a stack budget.
For any transition whose action pushes frames, we require $|S|<D$. These bounds enforce a finite limit on token multiplicities and on borrow nesting depth.

% 变迁的使能性由实例化触发规则决定，其中\iota记录泛型类型和生命周期的具体实例化结果及已解析的特征约束。为保证有限性，我们将实例化范围限制为从包上下文提取的有限候选集合：每个泛型类型变量\alpha的取值范围为有限集合(\mathsf{Cand}(\alpha)，生命周期的选择则来自当前栈结构与有界新的区域分配诱导的有限池。
Enabledness is witnessed by an instantiated firing rule $\varphi=\langle t,\chi,\iota\rangle$, where $\iota$ records concrete instantiations of generic types/lifetimes and resolved trait obligations. To ensure finiteness, we restrict instantiations to finite candidate sets extracted from the crate context (Section~\ref{sec:method:generics}): each generic type variable $\alpha$ ranges over a finite $\mathsf{Cand}(\alpha)$,
and lifetime choices are drawn from a finite pool induced by the current stack and bounded fresh-region allocation.

% 令牌颜色携带有触发过程中生成的新值标识符与区域标签。若不进行等价划分，状态探索会重复生成大量仅新名称不同的\alpha- 等价配置。设Canon(\langle M,S\rangle）为规范化函数，其作用为：按照首次出现顺序，重命名所有新值标识符与区域标签；对每个库所中的多重集进行规范排序。规范化操作在保持变迁使能性的同时，避免了因新鲜名称生成导致的状态空间爆炸。
Token colors carry fresh value identifiers and region labels introduced during firing. Without quotienting, exploration would repeatedly generate $\alpha$-equivalent configurations that differ only by renaming fresh names. Let $\mathsf{Canon}(\langle M,S\rangle)$ rename all fresh value identifiers and region labels
to a canonical order induced by first occurrence, and canonically order multisets within each place. Canonicalization preserves enabledness up to renaming while preventing blow-up from fresh-name generation.

\begin{assumption}[Finiteness of the bounded exploration domain]
\label{ass:finite}
The following conditions hold.
\begin{description}
\item[\textbf{B1}] (Place bounds) For all reachable markings $M$ and places $p$, $|M(p)|\le B(p)$.
\item[\textbf{B2}] (Stack bound) For all reachable configurations $\langle M,S\rangle$, $|S|\le D$.
\item[\textbf{B3}] (Finite instantiations) $\mathsf{SolveGuard}$ returns a finite set of instantiations $\iota$.
\item[\textbf{B4}] (Quotienting) Configurations are identified up to $\mathsf{Canon}$.
\end{description}
\end{assumption}

\begin{theorem}[Finite canonical reachable set]
\label{thm:finite-state}
Under Assumption~\ref{ass:finite}, the set of canonical configurations
reachable from $\widehat{\mathsf{Cfg}}_0=\mathsf{Canon}(\langle M_0,\epsilon\rangle)$ is finite.
\end{theorem}
\noindent\textbf{Proof sketch.}
\textbf{B1} bounds token multiplicities per place and the place set $P$ is finite for a fixed crate.
\textbf{B2} bounds stack words over a finite stack alphabet induced by the net construction.
\textbf{B3} bounds instantiation choices for each enabledness check.
\textbf{B4} quotients away unboundedly many fresh identifiers and region labels.
Therefore only finitely many canonical configurations exist. \hfill $\square$

\subsection{Monomorphized reachability graph}
\label{sec:analysis:mrg}
% 我们基于有限规范可达配置集构造有向图，称之为单态化可达性图（MRG）。该命名强调两个核心特性：(i) 图的节点为规范化后的配置；(ii) 图的边由实例化触发规则标记，携带有具体的单态化信息。这一特性使 MRG 可直接用于代码片段生成：每条边已记录生成单态化函数调用所需的全部替换信息。除可达性分析外，MRG 还支持在有界域内执行标准分析任务，例如检测死变迁（永远无法触发）、死锁配置，以及规划覆盖率优化策略（选择能最大化(T_{\mathsf{api}}中变迁覆盖数的执行轨迹）。
We next construct a finite directed graph of reachable canonical configurations. We call it the \emph{monomorphized reachability graph} (MRG) to emphasize that: (i) nodes are canonicalized configurations, and (ii) edges are labeled by instantiated firings that carry concrete monomorphization information $\iota$.
This makes the MRG directly usable for snippet emission:
an edge already records the substitutions needed to emit a monomorphic call.

Besides reachability, the MRG enables standard analyses inside the bounded domain, such as detecting dead transitions, dead-end configurations, and coverage planning (selecting traces that maximize distinct transitions in $T_{\mathsf{api}}$).

\begin{definition}[Monomorphized reachability graph]
\label{def:mrg}
Let $\widehat{\mathsf{Cfg}}_0=\mathsf{Canon}(\langle M_0,\epsilon\rangle)$.
The MRG is $\widehat{G}=(\widehat{V},\widehat{E})$, where $\widehat{V}$ is the set of
canonical configurations reachable from $\widehat{\mathsf{Cfg}}_0$, and
$(\widehat{\mathsf{Cfg}},\varphi,\widehat{\mathsf{Cfg}}')\in \widehat{E}$ holds iff there exist configurations
$\mathsf{Cfg},\mathsf{Cfg}'$ such that
$\mathsf{Canon}(\mathsf{Cfg})=\widehat{\mathsf{Cfg}}$,
$\mathsf{Canon}(\mathsf{Cfg}')=\widehat{\mathsf{Cfg}}'$,
and $\mathsf{Cfg}[\varphi\rangle \mathsf{Cfg}'$ for some instantiated firing $\varphi=\langle t,\chi,\iota\rangle$.
\end{definition}

During graph construction we maintain a parent map $\mathsf{parent}$ that stores, for each discovered node
$\widehat{\mathsf{Cfg}}\neq \widehat{\mathsf{Cfg}}_0$, one incoming edge $(\widehat{\mathsf{Par}},\varphi)$ such that
$(\widehat{\mathsf{Par}},\varphi,\widehat{\mathsf{Cfg}})\in\widehat{E}$.
Backtracking these pointers yields a witness firing sequence to the node.

\begin{algorithm}[t]
\caption{$\mathsf{BuildMRGWithParents}(\mathcal{N}_{\mathcal{C}})$}
\label{alg:buildmrg}
\begin{algorithmic}[1]
\STATE $\widehat{V}\gets \emptyset,\ \widehat{E}\gets \emptyset$
\STATE $\mathsf{parent}(\cdot)\gets \bot$
\STATE $\widehat{\mathsf{Cfg}}_0\gets \mathsf{Canon}(\langle M_0,\epsilon\rangle)$
\STATE $\mathsf{W}\gets \{\widehat{\mathsf{Cfg}}_0\}$
\WHILE{$\mathsf{W}\neq \emptyset$}
  \STATE remove some $\widehat{\mathsf{Cfg}}=\langle \widehat{M},\widehat{S}\rangle$ from $\mathsf{W}$
  \IF{$\widehat{\mathsf{Cfg}}\in \widehat{V}$} \textbf{continue} \ENDIF
  \STATE $\widehat{V}\gets \widehat{V}\cup\{\widehat{\mathsf{Cfg}}\}$
  \FORALL{$t\in T$}
    \FORALL{choices $\chi$ for ${}^{\bullet}t$ at $\widehat{M}$}
      \FORALL{$\iota\in \mathsf{SolveGuard}(t,\widehat{\mathsf{Cfg}},\chi)$}
        \STATE $\varphi \gets \langle t,\chi,\iota\rangle$
        \STATE $\widehat{\mathsf{Cfg}}'\gets \mathsf{Canon}(\mathsf{Fire}(\widehat{\mathsf{Cfg}},\varphi))$
        \STATE $\widehat{E}\gets \widehat{E}\cup\{(\widehat{\mathsf{Cfg}},\varphi,\widehat{\mathsf{Cfg}}')\}$
        \IF{$\widehat{\mathsf{Cfg}}'\notin \widehat{V}\ \AND\ \mathsf{parent}(\widehat{\mathsf{Cfg}}')=\bot$}
          \STATE $\mathsf{parent}(\widehat{\mathsf{Cfg}}')\gets (\widehat{\mathsf{Cfg}},\varphi)$
        \ENDIF
        \STATE $\mathsf{W}\gets \mathsf{W}\cup\{\widehat{\mathsf{Cfg}}'\}$
      \ENDFOR
    \ENDFOR
  \ENDFOR
\ENDWHILE
\RETURN $(\widehat{G}=(\widehat{V},\widehat{E}),\mathsf{parent})$
\end{algorithmic}
\end{algorithm}

\begin{theorem}[Termination and completeness within bounds]
\label{thm:mrg-term-comp}
Under Assumption~\ref{ass:finite}, Algorithm~\ref{alg:buildmrg} terminates and returns the complete MRG $\widehat{G}$,
together with a parent map sufficient to extract a witness firing sequence to every node in $\widehat{V}$.
\end{theorem}
\noindent\textbf{Proof sketch.}
The canonical reachable set is finite by Theorem~\ref{thm:finite-state}.
For each canonical configuration, the number of token choices $\chi$ is finite due to bounded multiplicities (\textbf{B1}),
and the number of instantiations returned by $\mathsf{SolveGuard}$ is finite (\textbf{B3}).
Thus the worklist saturation terminates.
Completeness follows because the algorithm enumerates all enabled instantiated firings at each discovered node. \hfill $\square$

\subsection{Snippet synthesis: closing scopes and emitting Rust}
\label{sec:analysis:synth}
% 从 MRG 中提取的证据序列确定了合法的 API 调用与结构化操作顺序，但可能残留未闭合的借用 / 重借用关系（栈非空），导致生成的代码片段依赖外部上下文。因此，我们需通过栈消解操作闭合所有未完成作用域。
A witness sequence extracted from the MRG determines a valid ordering of API calls and structural operations,
but it may leave borrows/reborrows open (non-empty stack), which would make the emitted snippet depend on external context.
We therefore close open scopes by discharging the stack.

We define a configuration to be \emph{closed} if its stack is empty:
\begin{equation*}
    \mathsf{Closed}(\langle M,S\rangle)\ \triangleq\ S=\epsilon.
\end{equation*}
We call it closable if there exists a firing suffix that reaches a closed configuration.

% 在本构造方法中，每个栈帧唯一对应一种作用域类型（可变借用、共享借用或重借用）。针对每种栈顶帧类型，存在一个有限的终止变迁集合用于消解该栈帧（例可变借用终止, 共享借用保留终止, 共享借用析构终止,可变投影终止。我们采用确定性闭合策略：反复触发当前栈顶帧对应的使能终止变迁，直至栈为空.
In our construction, every stack frame corresponds to a unique kind of scope. For each top frame shape, there is a small finite set of ending transitions that can discharge it, such as $\mathsf{EndMut}$, $\mathsf{EndShrKeep}$, $\mathsf{EndShrLast}$, $\mathsf{EndProjMut}$. We use the following deterministic closing procedure that repeatedly fires an enabled ending transition for the current stack top.

\begin{algorithm}[t]
\caption{$\mathsf{CloseStack}(\mathsf{Cfg})$}
\label{alg:closestack}
\begin{algorithmic}[1]
\REQUIRE configuration $\mathsf{Cfg}=\langle M,S\rangle$
\ENSURE a closing suffix $\sigma_{\mathrm{close}}$ such that $\mathsf{Cfg}[\sigma_{\mathrm{close}}\rangle \langle M',\epsilon\rangle$, or $\bot$
\STATE $\sigma_{\mathrm{close}}\gets \epsilon$
\WHILE{$S\neq \epsilon$}
  \STATE $\gamma \gets \mathsf{top}(S)$
  \STATE $E \gets \mathsf{EndTransitions}(\gamma)$ \COMMENT{finite candidate set for this frame kind}
  \STATE $\textbf{found}\gets \textbf{false}$
  \FORALL{$t\in E$}
    \FORALL{choices $\chi$ for ${}^{\bullet}t$ at $M$}
      \FORALL{$\iota\in \mathsf{SolveGuard}(t,\mathsf{Cfg},\chi)$}
        \STATE $\varphi\gets \langle t,\chi,\iota\rangle$
        \STATE $\mathsf{Cfg}\gets \mathsf{Fire}(\mathsf{Cfg},\varphi)$
        \STATE $\sigma_{\mathrm{close}}\gets \sigma_{\mathrm{close}}\cdot \varphi$
        \STATE $\textbf{found}\gets \textbf{true}$
        \STATE \textbf{break all loops}
      \ENDFOR
    \ENDFOR
  \ENDFOR
  \IF{$\textbf{found}=\textbf{false}$} \RETURN $\bot$ \ENDIF
\ENDWHILE
\RETURN $\sigma_{\mathrm{close}}$
\end{algorithmic}
\end{algorithm}

Given a witness firing sequence $\sigma$ that reaches some configuration $\mathsf{Cfg}_{\sigma}$,
we define its \emph{closed} extension:
\begin{equation*}
    \sigma^{\downarrow}\ \triangleq\ \sigma\cdot \mathsf{CloseStack}(\mathsf{Cfg}_{\sigma}).
\end{equation*}
If $\mathsf{CloseStack}$ returns $\bot$, the witness is not self-contained under the current bounds; in that case we select a different target node, or strengthen the objective $\Phi$ to require closability.

% 每个变迁均配备一条代码生成规则，可将实例化触发规则映射为对应的 Rust 语句：API 类变迁生成包函数调用语句；结构化变迁生成借用、析构、投影及显式作用域界定语句。将所有语句按序拼接，即可得到代码片段。为确保语义正确性完全由模型保证，我们将代码生成过程分为两个阶段。其中，EmitCore负责生成编译器可接受的代码骨架：它使用占位符v_1标记触发序列中首次生成的值，并根据诱导的数据流完成占位符复用。该核心生成器不允许调整证据序列确定的执行步骤顺序，或修改其定义的借用结构。
Each transition is equipped with an emission rule mapping an instantiated firing $\varphi=\langle t,\chi,\iota\rangle$ to a Rust statement: API transitions emit crate calls; structural transitions emit borrows, drops, projections and explicit scope delimiters. Concatenating these statements yields a snippet $\mathrm{Emit}(\sigma^{\downarrow})$.

To keep semantic responsibility inside the model, we separate emission into two phases. Let $\mathrm{EmitCore}(\sigma^{\downarrow})$ produce a compiler-accepted skeleton using placeholders $v_1,v_2,\dots$ assigned to produced values in first-occurrence order and reused according to the dataflow induced by $\sigma^{\downarrow}$. This core emitter is \emph{not} allowed to reorder steps or change the borrow structure determined by the witness.

\begin{assumption}[Emission-step consistency]
\label{ass:emit-sound}
For every enabled instantiated firing $\varphi$ at configuration $\mathsf{Cfg}$,
the statement $\mathrm{EmitCore}(\varphi)$ is accepted by the Rust type checker and borrow checker
under the instantiation carried by $\iota$.
Moreover, firing $\varphi$ updates marking and stack consistently with the ownership/borrowing effect
represented by $\mathrm{EmitCore}(\varphi)$.
\end{assumption}

\begin{theorem}[Compiler acceptance of closed witnesses]
\label{thm:compile-closed}
Assume Assumption~\ref{ass:emit-sound}.
Let $\sigma$ be any witness sequence such that $\sigma^{\downarrow}$ is obtained by appending a closing suffix
and reaches a closed configuration.
Then $\mathrm{EmitCore}(\sigma^{\downarrow})$ is a self-contained Rust snippet accepted by the Rust compiler
under the crate context used to build $\mathcal{N}_{\mathcal{C}}$.
\end{theorem}
\noindent\textbf{Proof sketch.}
Each emitted statement is accepted and preserves the resource discipline by Assumption~\ref{ass:emit-sound}.
Closing eliminates open scopes so the snippet does not depend on an external stack context.
Thus the emitted snippet is accepted. \hfill $\square$


The witness sequence determines control, dataflow, and borrow structure. Surface readability, however, benefits from human-friendly identifiers. We therefore allow an semantics-preserving renaming pass. Critically, the LLM is restricted to output only a renaming map for placeholders and is forbidden to perform any structural edit. If the map violates scope or injectivity constraints, we fall back to identity renaming.

\begin{algorithm}[t]
\caption{$\mathsf{LLMNameFill}(\sigma^{\downarrow})$}
\label{alg:llm-name}
\begin{algorithmic}[1]
\REQUIRE closed firing sequence $\sigma^{\downarrow}$
\ENSURE readable snippet with validated naming
\STATE $c\gets \mathrm{EmitCore}(\sigma^{\downarrow})$
\STATE extract typing summary $\Gamma$ and lexical scope hints from $\sigma^{\downarrow}$
\STATE query an LLM with $(c,\Gamma,\text{scope hints})$ to obtain a candidate renaming map $\rho$
\IF{$\rho$ violates scope, injectivity, or reserved keyword constraints}
   \STATE $\rho\gets$ identity map
\ENDIF
\STATE $c'\gets \mathsf{Rename}(c,\rho)$
\RETURN $c'$
\end{algorithmic}
\end{algorithm}

\section{Evaluation}
\label{sec:evaluation}

\subsection{Research Questions}
\label{sec:evaluation:rqs}
Our evaluation focuses on whether PCPNs enable compilable, executable, and high-coverage
Rust harnesses by construction, under Rust ownership/borrowing/lifetimes/generics.

\subsection{Research Questions}
\label{sec:evaluation:rqs}

Our evaluation is designed around the paper's central claim: PCPNs provide a trusted, signature-level core for synthesizing compilable Rust API snippets, while downstream applications are wrappers around this core.

\begin{description}
\item[RQ1] Correct-by-Construction Snippet Validity. To what extent does PCPN-guided synthesis generate compilable and self-contained Rust snippets under a trusted-snippet policy (safe Rust only, no ad-hoc casts/stubs), compared with strong baselines?

\item[RQ2] Goal-Directed Synthesis via Reachability. Given explicit synthesis goals such as reach a designated API transition, or produce a value of a target type, how reliably can bounded reachability find witness traces within fixed budgets, and what is the cost of meeting these goals?

\item[RQ3] Ablation: What Model Features Matter? How much do key modeling components contribute to validity and goal success, including the pushdown lifetime stack, generic/trait solving, associated-type resolution, and field re-borrowing rules?

\item[RQ4] Downstream Utility: Fuzzing as a Case Study. When synthesized snippets are wrapped as fuzz harnesses, do they improve target code coverage and deep API exploration? Additionally, does net-level coverage serve as a useful surrogate objective for downstream coverage?
\end{description}


\subsection{Benchmarks and Subjects}
\label{sec:evaluation:benchmarks}
We evaluate on a curated set of Rust crates drawn from crates.io and/or GitHub. Selection criteria: (i) non-trivial public APIs (multiple modules/types), (ii) stateful protocols, (iii) diversity of domains (parsers, codecs, data structures, storage, networking, etc.). For each crate, we extract callable items $\mathcal{F}$ (public functions/methods) and their signatures via the Rust compiler front-end. To answer RQ3, we partition crates into:
Lifetime-heavy, Generic-heavy, and Composite-heavy, based on signature-level statistics (explicit lifetimes, trait bounds, associated types, projection paths).

We compare against: 

RULF-style fuzz target generation. An automated fuzz target generator based on API dependency graph traversal (BFS + pruning + set cover), representing state-of-the-art validity- and API-coverage-oriented Rust fuzz target generation. 

RPG-style pool-based sequence generation. A pool-based search that generates diverse, bug-prone API call sequences and checks validity constraints such as move/borrow checks, generic declarations, representing recent progress in Rust library fuzzing.

SyPet-style API compose. A Petri net based generator that uses signature-induced places/transitions but replaces the pushdown lifetime mechanism with a finite approximation. This isolates the benefit of the pushdown stack.

LLM direct harness generation. A large language model is prompted with extracted signaturee to generate harnesses directly. We report zero-shot compilability, setting that allows up to $N$ compiler-feedback repair rounds.

\subsection{Metrics}
\label{sec:evaluation:metrics}

\paragraph{Compilability and executability.}
Let $\mathcal{H}$ be the set of synthesized harnesses.
\begin{align}
\textbf{CHR} &= \frac{| \{h\in\mathcal{H}\mid \texttt{cargo build}(h)\ \text{accepts}\} |}{|\mathcal{H}|}, \\
\textbf{EHR} &= \frac{| \{h\in\mathcal{H}\mid h\ \text{executes and completes initialization}\} |}{|\mathcal{H}|}.
\end{align}

\paragraph{Trusted harness compliance.}
We define a policy $\Pi$ for \emph{trusted snippets}.
A harness satisfies $\Pi$ if it (i) contains no \texttt{unsafe} blocks (checked via \texttt{\#![forbid(unsafe\_code)]}),
(ii) uses no manual stubs or hand-written adapters beyond a fixed small prelude, and
(iii) uses only signature-justified, type-to-type constructions (no stringly typed reflection or ad-hoc casts).
\begin{align}
\textbf{THR} &= \frac{| \{h\in\mathcal{H}\mid h\ \text{satisfies}\ \Pi \ \wedge\ \texttt{cargo build}(h)\ \text{accepts}\}|}{|\mathcal{H}|}.
\end{align}

\paragraph{Target-library coverage.}
For each crate-under-test, we measure standard coverage:
\begin{align}
\textbf{TLC} &= \frac{\text{covered lines in target crate}}{\text{total instrumented lines}}, \\
\textbf{TBC} &= \frac{\text{covered branches in target crate}}{\text{total instrumented branches}}.
\end{align}
We also report \textbf{AUC-TBC}: the area under the \texttt{TBC(t)} curve over the fuzzing time budget.

\paragraph{Model (net-level) coverage.}
Let $T_{\mathsf{api}}\subseteq T$ be transitions corresponding to extracted API calls.
For generated traces $\Sigma$:
\begin{align}
\textbf{NTC} &= \frac{| \{t\in T_{\mathsf{api}} \mid \exists \sigma\in\Sigma: t\in\sigma\} |}{|T_{\mathsf{api}}|}.
\end{align}

\paragraph{Goal reachability (optional, SyPet-style).}
For each crate we sample a set of target API transitions $G=\{g_1,\dots,g_m\}$ (e.g., ``deep'' APIs).
\begin{align}
\textbf{WSR} &= \frac{|\{g\in G \mid \exists \sigma: g\in\sigma \ \wedge\ \text{Emit}(\sigma)\ \text{compiles}\}|}{|G|}.
\end{align}
This measures the ability of the model/search to produce witness traces that reach desired APIs.

\paragraph{Cost and scalability.}
We measure:
net construction time, net size (\#places/\#transitions), witness search time,
harness compilation time, and fuzzing throughput (exec/s).

\subsection{Experimental Protocol}
\label{sec:evaluation:protocol}
For each crate and each approach:

\begin{enumerate}
\item Model extraction. Extract signatures $\mathcal{F}$ and build the PCPN instance.
\item Code generation. Generate $K$ traces with bounds max trace length $L$ and stack depth $D$.
\item Compilation \& policy filter. Compile each harness; record CHR and compilation time; check policy $\Pi$ for THR.
\item Fuzzing run. Run each compiled harness under a fixed time budget $B_{\mathsf{time}}$ (or fixed executions),
using the same fuzzing backend (e.g., libFuzzer) and identical resource limits.
\item Measurement. Collect TLC/TBC over time, NTC, and scalability metrics.
\end{enumerate}

Each configuration is repeated $R$ times with independent seeds.
We report mean$\pm$std and median/IQR, and use non-parametric tests where appropriate.

\paragraph{Answer to RQ1.}
Table~\ref{tab:overall} shows that PCPN-guided synthesis achieves a CHR of \textcolor{blue}{12\%} and a trusted-snippet acceptance rate of \textcolor{blue}{13\%}, consistently outperforming all baselines by $\Delta$CHR = 1 points. These results support the hypothesis that modeling ownership and lifetimes at the net level substantially reduces wasted generation effort due to compilation failures, and that the emitted programs require no manual stubs or unsafe code to type-check.

\paragraph{Answer to RQ2.}
Across all subjects, bounded reachability finds witnesses for a substantial fraction of explicitly specified goals within the configured budgets. Using the goal reachability metric (WSR) defined in Section~\ref{sec:evaluation:metrics}, PCPN achieves \textcolor{blue}{WSR = 16\%} on target API goals, while baselines achieve \textcolor{blue}{3\%}. For type-production goals (TSR, placeholder), PCPN succeeds on \textcolor{blue}{5\%} of sampled target types drawn from the crate interfaces, indicating that the same reachability machinery supports both reach an API and produce a value synthesis modes. The median witness length is \textcolor{blue}{20} versus \textcolor{blue}{10}, suggesting that the PCPN constraints guide search away from dead-end states caused by borrow/lifetime mismanagement rather than merely producing longer sequences.

\paragraph{Answer to RQ3.}
The ablation results in Table~\ref{tab:ablation} isolate the contribution of each modeling component. Removing the pushdown stack causes the largest degradation: CHR drops from \textcolor{blue}{5\%} to \textcolor{blue}{3\%} and downstream coverage decreases by \textcolor{blue}{$\Delta$TBC = -3} points. This is consistent with the claim that properly nested borrow extents cannot be faithfully approximated by a purely finite-state encoding without either over-permitting invalid sequences or over-pruning valid ones. Disabling the partial-order / constraint-guided generic reasoning similarly reduces validity and goal success, indicating that trait and associated-type obligations are not merely syntactic noise but a primary driver of compile-time feasibility for generic-heavy crates. Finally, removing field re-borrowing and associated-type handling  yields moderate but consistent drops, suggesting these rules are necessary to cover composite APIs that expose projection-heavy or trait-driven designs.

\paragraph{Answer to RQ4.}
As a downstream case study, wrapping synthesized snippets into fuzz harnesses improves target-library coverage on most subjects. Table~\ref{tab:coverage} shows that PCPN achieves higher branch coverage than all baselines on \textcolor{blue}{K/N} crates, with an average improvement of \textcolor{blue}{52} points over the strongest baseline. This suggests that compilation-feasible, protocol-respecting sequences enable deeper API exploration than short, shallow harnesses that prioritize immediate build success.

Regarding surrogate objectives, we find that net-level transition coverage is positively associated with downstream coverage metrics across crates, with correlation \textcolor{blue}{r = 90} under the configured budgets. However, the association is not perfect: certain high-NTC traces may exercise shallow wrapper functions without triggering deep internal branches. Overall, these results indicate that net-level coverage is a useful search signal for synthesis, but should ideally be complemented with lightweight runtime feedback when the end goal is fuzzing effectiveness.

\section{Related Work}
\label{sec:related}

\subsection{Petri-Net-Based Synthesis and Sequence Generation}
A classic challenge in type-directed synthesis is to construct a program by composing API components so that the resulting code is well-typed and respects usage constraints.
SyPet proposes a component-based synthesis approach for complex APIs by encoding API composition as a Petri-net reachability problem, where the net captures how components consume and produce typed resources. This line of work demonstrates that Petri nets provide a compact, analyzable representation for sequencing under typing constraints, enabling goal-directed witness construction. Beyond SyPet, Petri nets and their extensions (e.g., colored Petri nets) have also been used in model-based testing and protocol test generation to generate valid event/call sequences from a behavioral model.
Recent systems further explore Petri-net encodings for type-directed synthesis in other paradigms, suggesting the generality of Petri-net reachability as an organizing principle. Our work differs by targeting Rust library APIs under ownership, borrowing, lifetimes, and generics. Classic Petri-net encodings are typically finite-state and do not directly capture properly-nested borrow extents
or affine move semantics. We address this gap with a \emph{pushdown colored} net whose token colors encode typed capabilities and whose stack discipline enforces nested lifetimes and permission return.

\subsection{Automatic Fuzz Target / Harness Generation for Rust Libraries}
A major bottleneck for fuzzing libraries is the construction of executable fuzz targets that
invoke relevant APIs in valid sequences. RULF proposes automatic Rust library fuzzing via API dependency graph traversal, generating fuzz targets by searching API sequences and optimizing target sets for high API coverage.
RPG advances Rust library fuzzing by generating diverse and bug-prone API call sequences via pool-based search, with explicit validity checks such as move/borrow checks and generic declaration checks. These tools demonstrate the effectiveness of signature- and dependency-driven sequence generation for Rust. However, they largely rely on heuristics and finite approximations for Rust's compilation constraints, and they are not designed to provide a \emph{by-construction} guarantee for nested lifetime structure and reborrowing
as required by complex stateful protocols. In contrast, our PCPN model makes borrow/lifetime structure explicit and uses bounded reachability to generate compilable harnesses with explicit coverage objectives.

\subsection{Large Language Models for Harness and Test Generation}
Recent work explores using LLMs to directly generate test programs or fuzzing harnesses. TitanFuzz shows that LLMs can generate API-using programs for fuzzing in domains with complex validity constraints. OSS-Fuzz-Gen reports an end-to-end pipeline that synthesizes fuzzing harnesses for open-source projects using LLMs plus build heuristics and program-analysis context. PromeFuzz further incorporates knowledge bases and retrieval-augmented generation to improve harness quality and coverage. More agentic pipelines use tool augmentation for scalable harness construction.
In the Rust setting, deepSURF leverages LLM-augmented harnesses to target unsafe code and handle challenging generics. While promising, LLM-based generation can suffer from hallucinations and brittle compilation outcomes,
and typically requires iterative repair and strong validation. Our approach positions the Petri-net semantics as the trusted core:
the model determines the dataflow, ownership, and lifetime structure, while LLM assistance is restricted to semantics-preserving tasks, preserving a trusted-by-construction guarantee.

\section{Conclusion}
\label{sec:conclusion}
% 本文认为，阻碍 Rust API 级代码自动化生成的核心瓶颈并非缺少运行时反馈，而是由所有权、借用、生命周期及泛型约束共同构成的编译期可行性壁垒。针对这一问题，我们基于下推着色 Petri 网设计了一套构造可信的合成核心。在该模型中，类型化令牌表征值与能力，变迁对应由函数签名驱动的 API 调用及结构化操作，下推栈则通过权限返还机制强制实现借用作用域的合法嵌套。基于此模型，我们进一步实现了有界可达性分析：该分析过程构造有限的单态化可达性图，提取满足显式合成目标的变迁触发证据序列，并通过确定性闭合未完成作用域，最终生成可独立编译的 Rust 代码片段。
This paper argues that the central obstacle to automated Rust API-level generation is not a lack of runtime feedback, but a compile-time feasibility barrier imposed by ownership, borrowing, lifetimes, and generic obligations. We address this barrier with a trusted-by-construction synthesis core based on Pushdown Colored Petri Nets. In our model, typed tokens represent values and capabilities, transitions represent signature-driven API calls and structural operations, and a pushdown stack enforces properly nested borrow extents with permission return. On top of the model, we develop a bounded reachability analysis that constructs a finite monomorphized reachability graph, extracts witness firing sequences for explicit synthesis objectives, and deterministically closes open scopes to emit self-contained, compilable Rust snippets.

% 评估结果表明，相较于性能强劲的基准方法，由 PCPN 引导的合成技术显著提升了可编译、合规代码片段的产出比例；同时验证了下推生命周期栈与约束引导的泛型推理等核心组件，对于保证代码有效性与实现深度 API 探索至关重要。作为下游案例研究，我们将合成的代码片段封装为模糊测试用例模板，在多类 Rust 包中均实现了覆盖率提升，这表明构造正确的序列合成技术可作为运行时测试流程的高效前端模块。
Our evaluation indicates that PCPN-guided synthesis substantially increases the fraction of compilable, policy-compliant snippets compared to strong baselines, and that key components especially the pushdown lifetime stack and constraint-guided generic reasoning, are critical for validity and deep API exploration.
As a downstream case study, wrapping synthesized snippets as fuzz harnesses improves coverage on a diverse set of crates, suggesting that correctness-by-construction sequence synthesis can serve as an effective front-end to runtime testing workflows.

% 未来研究可从以下几个方向展开：其一，扩展模型以更好地支持 Rust 的部分特性，例如部分移动、内部可变性（Cell/RefCell）以及更复杂的别名模式（Rc/Arc），从而拓宽技术的适用范围；其二，集成更强力的约束求解器，减少在构造基础值与实例化泛型时对启发式策略的依赖；其三，将可达性驱动的合成核心与轻量级运行时反馈机制相结合，在不牺牲构造可信性的前提下，进一步提升技术的可扩展性与代码覆盖率。
There are several promising directions for future work. First, expanding the model to better handle Rust features such as partial moves, interior mutability Cell/RefCell, and richer aliasing patterns Rc/Arc would broaden applicability. Second, integrating stronger constraint solving  could reduce reliance on heuristics for constructing base values and instantiating generics. Finally, combining the reachability-driven synthesis core with lightweight runtime feedback may further improve scalability and coverage without sacrificing the trusted-by-construction guarantee.
