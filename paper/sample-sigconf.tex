\begin{abstract}
Rust libraries increasingly expose stateful, type-directed APIs whose effective testing requires generating long, well-typed call sequences rather than mutating unstructured byte inputs. However, Rust’s affine ownership, borrow exclusivity, and lifetime constraints impose a compilation barrier: naively generated sequences frequently fail to type-check due to move-after-use, illegal aliasing under \texttt{\&mut}, or unsatisfied generic bounds, preventing any runtime feedback. This paper addresses signature-level test generation for Rust crates by modeling API invocations as a resource-flow problem over typed values and permissions.

We propose a \emph{Pushdown Colored Petri Net} (PCPN) model that is constructed from a crate’s extracted function signatures. Places denote Rust type expressions and related compile-time obligations (including generic placeholders, trait satisfaction, associated types, and field projections), while transitions capture signature-driven function calls and a small set of structural transformations (e.g., borrowing, borrow termination, projection and reborrowing of fields). Token flow represents ownership transfer and borrow-induced capability changes, and a pushdown stack enforces properly nested borrow extents, binding produced references to their provenance and restoring permissions when borrows end. To support generic-heavy APIs, we define a partial-order-based satisfaction relation that checks whether a concrete instantiation meets trait bounds and associated-type constraints during transition enabling. Composite types are handled via formal projection and (re)borrow transitions that preserve Rust’s aliasing restrictions at the level of generated harness code.

On top of the model, we formulate generation as a reachability problem with explicit coverage objectives over transitions (and, optionally, over signature items) and develop a bounded reachability analysis that searches for high-coverage firing traces under guard constraints.
Each trace is deterministically translated into a Rust harness, inserting \texttt{drop} or scope boundaries to realize stack-pop events, yielding programs that are compilable under the model’s conservative borrowing discipline. We implement a prototype tool that integrates with the Rust toolchain for signature extraction and produces harnesses compatible with standard coverage-guided fuzzing workflows.

% % Rust 在工业界和学术界被广泛视为一种既高性能又高可靠的系统级编程语言,其所有权、借用与生命周期机制从根本上避免了诸如悬垂指针、数据竞争和部分内存错误等传统漏洞.这些特性也使得 Rust 成为安全关键软件和基础设施软件的重要候选.然而,当前面向 Rust 的测试用例生成(尤其是模糊测试 / fuzzing)技术,与 Rust 类型系统所蕴含的丰富语义之间仍存在明显鸿沟：多数工具要么仅将程序视作“黑盒”并对输入字节流进行变异,要么在生成 API 调用序列时采用与 Rust 不匹配的简化模型,无法系统地覆盖所有权转移、借用检查、生命周期与高级泛型等语言特性.
% Rust has quickly become a prominent systems programming language in both industry and academia. It combines high performance with strong safety guarantees, eliminating many traditional bugs such as dangling pointers, data races, and a large class of memory errors through its ownership, borrowing, and lifetime system. These features also make Rust an attractive target for safety- and security-critical software. At the same time, they pose new challenges for \emph{automated test generation}, especially in the context of fuzzing: compared to C/C++ or garbage-collected languages, generating a set of Rust test cases that are not only high-coverage but also type-correct and borrow-safe requires respecting ownership transfer, borrowing rules, lifetimes, and advanced generics already at generation time.

% % 以典型的基于输入的模糊测试为例,现有工具通常假定目标函数已经通过编译、类型检查和借用检查,测试阶段只需要不断变异输入字节流以触发更多路径.这类方法在 C/C++ 等语言中行之有效,但在 Rust 环境下有两个局限：其一,它们几乎完全忽略了复杂类型结构、泛型约束以及 API 级的调用协议；其二,它们无法为“生成新的 Rust 程序或 API 调用序列”这一类结构化测试生成问题提供支持.对于需要自动构造高阶测试驱动(如自动生成调用链、构造复杂数据结构、组合多模块 API)的场景,仅对原始字节流进行变异不足以覆盖 Rust 代码中蕴含的大量语义路径.
% Existing fuzzing and test generation techniques for Rust typically fall into two extremes. One line of work treats the target as a black box function and mutates input byte streams or structured values, assuming that the function signatures are given and already well-typed. Another line of work generates code or API call sequences from simplified models that largely ignore Rust's ownership and borrowing semantics. The former is effective when the goal is to stress-test a fixed interface with relatively simple input types, but it struggles to cover complex control-flow patterns, rich data structures, and intricate generic constraints that arise from more structured code. The latter can, in principle, explore a richer space of code-level behaviors, yet often produces a large number of \emph{ill-typed} or \emph{borrow-invalid} programs that are rejected by the Rust compiler before any dynamic execution takes place.

As a consequence, the generator produces numerous programs that are statically rejected by the Rust compiler, providing no useful feedback to the testing process while consuming generation and compilation budget.

In addition, reasoning is inherently \emph{path-sensitive}: different control-flow paths may create and end borrows at different program points, and the lifetime of a reference is constrained by the points where it may be used along all feasible paths.

Most existing sequence generators and Petri-net models treat type names as simple labels and do not model this partial order, making it difficult to automatically check whether a proposed instantiation of a generic API is type-correct.

% To meet these requirements, we propose a new model for Rust test generation based on \emph{pushdown coloured Petri nets}. In our model, the program's control structure is represented by a Petri net whose places and transitions correspond to program locations and operations, respectively. Tokens carry rich colours that encode ownership, borrowing, lifetimes, and type information; the pushdown stack represents the call stack and lexical scopes. Informally, executing a Rust program corresponds to firing transitions in the Petri net, consuming and producing tokens whose colours reflect the evolving ownership and borrow state. Loops and recursion are captured by cycles in the net together with the pushdown stack, and their effect is summarized in a way that mirrors the compiler's borrow checking.

% Crucially, our model is not an independent semantics for Rust. % Based on this model, a test generator can search for sequences of transitions (code fragments) that correspond to well-typed, borrow-safe Rust programs. At the same time, the model is expressive enough to account for flow- and path-sensitive phenomena such as move assignments, temporary borrows, reborrowing, and the termination of borrows at points that are not syntactically explicit in the source code.

\section{Introduction}
% 本节用“Rust + 测试用例生成/模糊测试”的视角来写动机、挑战与贡献,先讲 Rust 特性与挑战,再讲现有方法不足,最后提出一个“模型(Petri 网)+ 工程实现 + 评估/验证 + 工具集成”的贡献闭环.
Rust has become a mainstream choice for building reliable systems software and performance-critical libraries.

% 强调 Rust 的工程价值：高性能 + 强类型系统 + 内存安全,适合作为被测目标(尤其是库/组件),也适合作为生成测试 harness 的语言本身.
Its combination of low-level performance and a rich type system eliminates broad classes of memory-safety bugs, while still exposing fine-grained control over resources.
As a result, an increasing amount of security and correctness-critical functionality is delivered as reusable Rust crates.
For such libraries, automated test generation especially coverage-guided fuzzing, is an indispensable technique for discovering panics, logic bugs, and unexpected corner cases that escape conventional unit tests.

Despite its success, \emph{effective} test-case generation for Rust libraries remains substantially harder than for many other languages.
% 引出核心矛盾：要想覆盖更高,往往需要“长序列 + 状态交互 + 结构化输入”；而 Rust 的所有权/借用/生命周期/泛型让“生成可编译、可执行”的序列本身就很难.
Modern fuzzers excel at mutating byte arrays and driving a single entry function, but library APIs are often \emph{stateful} and \emph{type-directed}:
coverage typically depends on constructing a web of interacting objects and invoking a long sequence of API calls that respects resource protocols.
In Rust, such protocols are enforced not only by runtime checks, but also by compile-time rules on ownership, borrowing, and lifetimes.
Consequently, a large fraction of na\"{i}vely generated call sequences never reach execution: they fail to compile due to move errors, aliasing violations, or unsatisfied generic constraints.
This compilation barrier turns into a coverage barrier.

\paragraph{A motivating example.}
% 给一个Rust 的例子：复合类型 + 借用 + 生命周期 + move(consume self)+ 泛型约束(trait bound).
Consider a simplified crate API in Listing~\ref{lst:api}, which resembles many real-world libraries (parsers, databases, protocol stacks, etc.).
It features (i) composite types (\texttt{Result}, \texttt{Option}, \texttt{Vec}), (ii) lifetime-carrying handles that borrow from a parent object, and (iii) a generic entry point constrained by trait bounds.

% \usepackage{listings}
% \usepackage{courier}
% \lstset{basicstyle=\ttfamily\small,columns=fullflexible,breaklines=true}
\begin{lstlisting}[language=Rust,caption={A small Rust API exhibiting ownership transfer, borrowing, lifetimes, and generics.},label={lst:api}]
pub struct Store { /* ... */ }

pub struct Tx<'s> { store: &'s mut Store /* ... */ }

pub trait KeyLike {
    fn as_bytes(&self) -> &[u8];
}

impl Store {
    pub fn open(path: &str) -> Result<Self, ()> { /* ... */ }

    // Borrowing & lifetime: a transaction borrows the store mutably.
    pub fn begin<'s>(&'s mut self) -> Tx<'s> { /* ... */ }
}

impl<'s> Tx<'s> {
    pub fn put<K: KeyLike>(&mut self, k: K, v: Vec<u8>) { /* ... */ }

    // Immutable borrow from &self, returns a reference tied to 's.
    pub fn get<'a, K: KeyLike>(&'a self, k: K) -> Option<&'a [u8]> { /* ... */ }

    // Ownership consumption: committing consumes the Tx value.
    pub fn commit(self) -> Result<(), ()> { /* ... */ }
}
\end{lstlisting}

A test generator that ignores Rust's ownership discipline might attempt to interleave calls in ways that are invalid in Rust.
For instance, after creating a transaction \texttt{tx} that holds \texttt{\&mut store}, a generator may try to call \texttt{store.begin()} again, or mutate \texttt{store} directly, while \texttt{tx} is still live---both are rejected by the borrow checker.
Similarly, \texttt{commit(self)} consumes \texttt{tx}, so any subsequent use of \texttt{tx} triggers a move error.
Finally, generic parameters such as \texttt{K: KeyLike} require the generator to produce a concrete type that satisfies the bound, rather than an arbitrary byte blob.

These issues are not merely syntactic.
% 强调：Rust 的类型系统把“协议”放在编译期；如果生成器不理解,就会大量产生不可编译样本,反馈式 fuzzing 也无从谈起.
In Rust, \emph{well-typedness} encodes a large portion of the API protocol.
Therefore, achieving high coverage for Rust libraries hinges on the ability to generate \emph{compilable} and \emph{type-correct} call sequences that respect ownership, borrowing, lifetimes, and generic constraints.

\paragraph{Why existing test generation falls short.}
% 这里对应两类“不足”：(1) 现有 fuzzing / 其他测试生成；(2) 现有 Petri 网生成序列在 Rust 上的问题.
A wide range of automated testing techniques exist, including coverage-guided fuzzing, property-based testing, grammar-based generation, symbolic execution, and model-based testing.
However, when applied to Rust library APIs, they commonly face at least one of the following limitations:

% 列举“当前模糊测试/测试生成不满足的特性”.不绑定某个具体工具版本,避免过度事实陈述.
\begin{enumerate}
  \item \textbf{Byte-oriented inputs without typed object construction.}
  Many fuzzing setups operate on byte arrays and rely on a hand-written harness to translate bytes into structured objects.
  For stateful APIs, the harness must additionally manage long call sequences, intermediate values, and resource protocols.
  This manual effort is error-prone and often becomes the dominant bottleneck.

  \item \textbf{Limited support for Rust-specific resource constraints.}
  Generators that treat values as freely copyable struggle with Rust's affine ownership:
  values can be moved, partially moved, borrowed immutably, or borrowed mutably, with strict aliasing rules.
  Moreover, Rust lifetimes are typically inferred (e.g., non-lexical lifetimes), which means that the precise end of a borrow is not explicitly written in the source code---yet it is essential for deciding when subsequent mutations become legal.

  \item \textbf{Weak handling of composite and generic APIs.}
  Real crates expose APIs over nested algebraic data types, slices, smart pointers, and rich trait-bound generics.
  Without a principled mechanism for instantiating generics (and ensuring that candidate types satisfy trait bounds), generators either (i) avoid such functions, or (ii) produce ill-typed code that never compiles.

  \item \textbf{Poor scalability when internal state is over-modeled.}
  Some analysis-driven approaches attempt to model internal program state to drive generation.
  While powerful, these approaches can be expensive and brittle for large crates, and they are often unnecessary when the goal is \emph{API-level} test generation rather than full program verification.
\end{enumerate}

A parallel line of work explores Petri nets (and related automata) for generating event sequences and testing protocols.
% 指出“现在 Petri 网生成代码序列不满足 Rust 的特性”：传统 Petri 网多用于协议/并发/事件序列,token 通常缺少“类型 + 借用 + 生命周期 + move”的语义.
Classic Petri-net encodings, however, are typically \emph{untyped} or only weakly typed: tokens represent abstract resources, but not Rust values with ownership and borrow capabilities.
As a result, straightforward Petri-net-driven sequence generation tends to violate Rust's compile-time rules in three recurring ways:
(i) it fails to enforce \emph{affinity} (a moved token must not be reused),
(ii) it cannot express \emph{borrow exclusivity} (no mutable aliasing, no mutation under an outstanding immutable borrow),
and (iii) it lacks a notion of \emph{scope- and lifetime-bounded availability} (references must not outlive the borrowed owner, and borrow permissions must eventually return).
Furthermore, generic constraints require a solver-like step (selecting admissible instantiations), which traditional Petri-net models do not provide.

\paragraph{Our perspective: signature-driven generation without heavy internal modeling.}
% 特别强调的差异点：我们有 Rust 代码库,函数签名给了输入/输出类型；我们不需要像 PCG 那样对函数体内部状态做“过度建模”.
This paper targets a common and practically important setting:
we are given a Rust codebase and we aim to generate test harnesses that invoke its public (or selected) functions and methods.
Crucially, we assume that \emph{function signatures}---including input/output types, reference mutability, lifetimes, and generic bounds---are available from the compiler toolchain.
We do \emph{not} attempt to precisely model the internal control flow or memory state of function bodies.
Instead, we treat each function as an atomic transition whose \emph{type signature} specifies how values (and their capabilities) can flow across call boundaries.
This signature-level stance makes the approach lightweight and scalable, while still capturing the constraints that dominate compilation success for generated tests.

\paragraph{Approach overview: Pushdown Colored Petri Nets for Rust API fuzzing.}
% 核心技术：下推着色 Petri 网.要把“类型、所有权、借用、生命周期、泛型”映射到 Petri 网的 token/颜色/栈结构上.
We introduce a new model for Rust test-case generation based on \emph{Pushdown Colored Petri Nets} (PCPNs).
At a high level:
\begin{itemize}
  \item \textbf{Places} represent pools of available \emph{typed values} together with \emph{capabilities} describing how they may be used (owned, immutably borrowed, mutably borrowed).
  \item \textbf{Transitions} represent \emph{function calls} extracted from the target crate's signatures.
  A transition consumes tokens matching the function's input types/capabilities and produces tokens matching its output types/capabilities.
  \item \textbf{Colors} encode Rust types (including composites) and capability metadata (e.g., \texttt{owned T}, \texttt{\&T}, \texttt{\&mut T}).
  \item \textbf{A pushdown stack} models \emph{nested scopes} and \emph{borrow extents}.
  When a reference is created, the model records (pushes) the borrowed provenance; when the reference becomes dead (e.g., explicitly \texttt{drop}'d or out of scope in the generated harness), the model pops the corresponding frame and returns permissions to the original owner.
\end{itemize}

This pushdown structure is essential for Rust-style lifetime reasoning:
borrow relationships can nest (reborrowing), and long API sequences may create and discharge borrows repeatedly.
A finite-state abstraction often either becomes overly conservative (blocking legal programs) or overly permissive (generating ill-typed programs).
PCPNs provide a compact, compositional way to represent potentially unbounded, properly nested borrow patterns.

\paragraph{Generics via a satisfiability partial order.}
% 对应你“泛型用偏序关系验证类型是否满足泛型参数要求”的想法.这里要给清晰定义,但不必把实现细节写死.
To support generic APIs, we equip the model with a \emph{type satisfaction} relation written as a partial order $\preceq$.
Intuitively, $A \preceq \mathcal{R}$ means that the concrete type $A$ satisfies the generic requirement $\mathcal{R}$ (e.g., a set of trait bounds, lifetime constraints, and where-clauses).
During transition firing, generic parameters are instantiated by selecting concrete types that are minimal (or otherwise prioritized) under $\preceq$ while remaining consistent with the surrounding token colors.
This design cleanly separates:
(i) the Petri-net execution semantics for sequencing, and
(ii) the constraint reasoning needed to produce admissible generic instantiations.
In practice, the partial order enables both sound pruning (reject impossible instantiations) and guided exploration (prefer simpler types early, diversify later).

\paragraph{From model to executable fuzz harnesses.}
% 给一个“生成出来的 harness 长什么样”的代码例子,并点出我们不需要建模函数体内部状态.
From a PCPN execution trace, we synthesize a Rust harness that composes API calls and feeds the resulting program to a standard coverage-guided fuzzer.
Listing~\ref{lst:harness} illustrates a typical generated skeleton for the API in Listing~\ref{lst:api}.
The harness is compilable by construction: every use respects move semantics, borrow exclusivity, and lifetime boundaries.

\begin{lstlisting}[language=Rust,caption={A compilable, signature-driven harness sketch generated from a PCPN trace.},label={lst:harness}]
fn harness(bytes: &[u8]) {
    // Construct base values (from bytes or small generators).
    let path = "tmp.db"; // tmp.db from Constant
    let mut store = match Store::open(path) { Ok(s) => s, Err(_) => return };

    // Begin a transaction: tx borrows store mutably.
    let mut tx = store.begin();

    // Choose a concrete key type K that satisfies K: KeyLike.
    // Here we pick Vec<u8> (assume the crate provides impl KeyLike for Vec<u8>).
    let k: Vec<u8> = bytes.to_vec();

    // Use tx while it is live; store cannot be used mutably here.
    tx.put(k.clone(), bytes.to_vec());
    let _v = tx.get(k);

    // Explicitly end the borrow extent and consume tx.
    let _ = tx.commit();

    // After commit, permissions return: store is usable again.
    let _tx2 = store.begin();
}
\end{lstlisting}

\noindent
Importantly, the generator does not need to model what \texttt{Store::open}, \texttt{Tx::put}, or \texttt{Tx::get} do internally.
% 区别于 PCG,我们不追踪“函数体内每个程序点”的能力变化；只在“函数边界”做资源流动建模,足以让生成序列可编译并能跑起来,从而提升覆盖率探索效率.
The signature alone determines how ownership and borrow capabilities move across call boundaries, which is exactly the information required to produce executable tests.

\section{Background}
\label{sec:background}
% 本章给出最小但足够的背景：Rust 的所有权/借用/生命周期(理解“为什么生成的序列很难编译”),

\subsection{Rust Ownership, Borrowing, and Lifetimes}
\label{sec:background:rust}
% 这一小节只讲与“生成可编译的 API 调用序列”直接相关的 Rust 规则：
% ownership(移动/析构)、borrowing(& / &mut 的互斥)、lifetimes(引用不能超过被借用者).

Rust enforces \emph{memory safety} primarily through a static discipline of \emph{ownership} and \emph{borrowing}.
Every non-\texttt{Copy} value has exactly one owner at a time; transferring a value to another variable (or passing it by value) typically \emph{moves} ownership.
When the owner goes out of scope, the value is dropped and its resources are released.

Borrowing allows temporary access to a value without transferring ownership.
Rust distinguishes:
(i) shared borrows \texttt{\&T}, which permit read-only access and can be aliased arbitrarily, and
(ii) mutable borrows \texttt{\&mut T}, which permit mutation but must be exclusive.
The central rule is often summarized as \emph{aliasing XOR mutability}:
at any point, either many shared borrows exist, or exactly one mutable borrow exists.

A \emph{lifetime} is a compile-time region that bounds how long a reference may be used.
A reference cannot outlive the value it refers to, and borrow permissions return to the owner when the borrow ends.
In modern Rust, borrow extents are often \emph{inferred} (e.g., via non-lexical lifetimes), meaning the precise end point of a borrow is not necessarily obvious from surface syntax, but is still crucial for deciding whether later mutations are legal.

% 下面给一个最小示例：move + 借用互斥 + 生命周期结束后权限“归还”.
\begin{lstlisting}[language=Rust,caption={Ownership moves, borrow exclusivity, and the end of a borrow extent.},label={lst:background:rust}]
fn demo() {
    let mut v: Vec<u8> = vec![1,2,3];

    // Shared borrow: many aliases are allowed, but no mutation through v while r is live.
    let r: &Vec<u8> = &v;
    // v.push(4); // ERROR (conceptually): cannot mutate while shared borrow exists.

    // Borrow ends here (extent), permissions return to v.
    drop(r);
    v.push(4); // OK

    // Move: ownership transfer; v becomes unusable after move.
    let w = v;
    // v.push(5); // ERROR: use of moved value.
    let _ = w.len();
}
\end{lstlisting}

For test-case generation, these rules impose constraints on \emph{call sequences}:
a generator must track whether a value is still owned, has been moved, is currently shared-borrowed, or is mutably borrowed, and it must ensure that references do not outlive their provenance.

\subsection{Generics and Trait Bounds in Signatures}
\label{sec:background:generics}
% 我们只需要“签名级别”的泛型信息：类型参数、trait bound、where 子句.
% 重点：生成器不仅要挑一个“类型”,还要保证它满足 bound(比如 T: Read + Clone).

Rust APIs are frequently generic.
A function may quantify over type parameters and constrain them with trait bounds (possibly in \texttt{where} clauses).
For example, a signature \texttt{fn f<T: Trait>(x: T) -> U} restricts admissible instantiations of \texttt{T} to types implementing \texttt{Trait}.
From the perspective of automated test generation, this means that producing a call is a two-step task:
(1) choose concrete instantiations for generic parameters, and
(2) supply values whose types match the instantiated signature.
If step (1) fails, the generated program will not type-check, regardless of runtime behavior.

\section{Method: Pushdown Colored Petri Nets for Rust API Test Generation}
\label{sec:method}
% 1) 形式化定义下推着色 Petri 网(PCPN)；
% 2) 形式化给出 Rust(以函数签名为中心)到 PCPN 的映射；
% 3) 覆盖 move/borrow/mut/指针、生命周期(栈绑定)、泛型偏序、复合类型字段(重)借用、Trait/关联类型等.

\subsection{Problem Setting}
\label{sec:method:setting}
% 明确问题设定：我们有一个 Rust 代码库及其函数签名；我们的建模是“签名级别”,不对函数体内部状态过度建模.
Let $\mathcal{C}$ be a Rust crate. We assume we can extract, from the compiler front-end, a finite set of \emph{callable items}
$\mathcal{F}$ (free functions and methods) together with their signatures, including
(i) input/output types, (ii) reference mutability, (iii) explicit lifetime parameters and outlives constraints when present, and
(iv) generic parameters with trait bounds and associated-type constraints.

Our goal is to generate compilable Rust \emph{harnesses} that construct typed values and execute sequences of calls to items in $\mathcal{F}$.
We deliberately avoid modeling internal states of function bodies; instead, each call is treated as an atomic transformation whose
compile-time validity is determined by the signature-level ownership/borrowing/lifetime/generic constraints.

\subsection{Petri Net Preliminaries}
\label{sec:method:pn}
\begin{definition}[Net and Petri Net]
\label{def:pn}
A net is a $3$-tuple $N=(P,T,F)$, where $P$ is a finite set of places, $T$ is a finite set of transitions,
$F\subseteq (P\times T) \cup (T\times P)$ is a finite set of arcs, and $P\cap T=\emptyset$.
A net can be viewed as a directed bipartite graph where places are represented by circles and transitions by rectangles.
The pre-set and post-set of a node $x \in\{P \cup T\}$ are defined as
$^{\bullet}x=\{y\in P\cup T\mid (y,x)\in F\}$ and $x^{\bullet}=\{y\in P\cup T\mid (x,y)\in F\}$, respectively.
In this paper, it is assumed that the weight of each directed arc is $1$.

A marking of $N=(P,T,F)$ is a mapping $M:P\rightarrow \mathbb{N}$ where $M(p)$ is the number of tokens in place $p$.
A net $N$ with an initial marking $M_{0}$ is called a Petri Net and denoted as $(N,M_{0})$.
Transition $t$ is enabled at $M$ if $\forall p\in {}^{\bullet}t: M(p)>0$, denoted as $M[t\rangle$.
A marking $M$ is a deadlock marking if any transition is disabled at $M$.
Firing an enabled transition $t$ yields a new marking $M^{\prime}$:
$M^{\prime}(p)=M(p)-1$ if $p\in {}^{\bullet}t\backslash t^{\bullet}$;
$M^{\prime}(p)=M(p)+1$ if $p\in t^{\bullet}\backslash {}^{\bullet}t$;
and otherwise $M^{\prime}(p)=M(p)$.
This is denoted as $M[t\rangle M^{\prime}$.
A marking $M_{k}$ is reachable from $M$ if $M_{k}=M$ or there exists a non-empty transition sequence
$\sigma=t_{1}t_{2}\dots t_{k}$ such that
$M[t_{1}\rangle M_{1}[t_{2}\rangle \dots M_{k-1}[t_{k}\rangle M_{k}$ (abbreviated as $M[\sigma \rangle M_{k}$).
All markings reachable from $M_{0}$ in a Petri net is written as $\mathcal{M}$.
This paper only considers bounded Petri nets, i.e.,
$\exists k\in \mathbb{N},\forall{M}\in \mathcal{M}:M(p)\leq k$.
\end{definition}

\subsection{Pushdown Colored Petri Nets}
\label{sec:method:pcpn}
% 在 PN 基础上扩展：
% (i) token 带“颜色”(用于携带 Rust 变量身份、泛型实例化信息、生命周期标签等)；
% (ii) 引入一个全局栈(pushdown store),用于建模生命周期/借用范围的嵌套与归还.

\paragraph{Type expressions and capability labels.}
% places 需要对应 Rust “类型/占位符/trait/关联类型/字段访问器”等,因此先定义一个“类型表达式语言”.
Let $\mathsf{Ty}$ be the set of Rust type expressions extracted from $\mathcal{C}$, extended with:
\begin{align*}
\tau \in \mathsf{Ty} ::=~&
g \mid \alpha \mid \tau_1\langle \tau_2,\dots,\tau_n\rangle \mid (\tau_1,\dots,\tau_n) \mid [\tau] \mid \\
& \mathsf{Ref}^q(\tau) \mid \mathsf{Assoc}(\tau,\mathsf{Tr},A) \mid \mathsf{Field}(\tau,f) \mid \mathsf{Impl}(\tau,\mathsf{Tr})
\end{align*}
where:
$g$ ranges over ground types (e.g., \texttt{u8}, \texttt{Vec<u8>}, user structs/enums),
$\alpha$ ranges over generic type variables,
$q\in\{\mathsf{shr},\mathsf{mut}\}$ distinguishes shared/mutable references,
$\mathsf{Tr}$ ranges over traits,
$A$ ranges over associated type names,
and $f$ ranges over field names / projection paths.
We use $\mathsf{Impl}(\tau,\mathsf{Tr})$ as a \emph{witness-like} type expression meaning ``$\tau$ satisfies trait $\mathsf{Tr}$''.

% 为了表达“同一类型在不同借用状态下可用性不同”,我们引入 capability 标签并将其编码进 place 的标签中.
Let $\mathsf{Cap}=\{\mathsf{own},\mathsf{frz},\mathsf{blk},\mathsf{val}\}$ denote capability modes:
\begin{itemize}
  \item $\mathsf{own}$: an owned value that can be moved and mutably accessed;
  \item $\mathsf{frz}$: an owned value that is \emph{frozen} due to outstanding shared borrows (read-only access allowed);
  \item $\mathsf{blk}$: an owned value that is \emph{blocked} due to an outstanding mutable borrow (no direct access);
  \item $\mathsf{val}$: a non-owning value form (e.g., a reference value itself, trait witness token, etc.).
\end{itemize}

\paragraph{Places.}
% 库所 = “capability + 类型表达式”.这满足你说的 places 表示类型/trait/关联类型/字段访问器等.
We define the set of places as
\[
P ~=~ \{\, p_{\kappa,\tau} \mid \kappa\in\mathsf{Cap},~ \tau\in\mathsf{Ty} \,\},
\]
where $p_{\mathsf{own},\tau}$ stores owned values of type $\tau$,
$p_{\mathsf{frz},\tau}$ stores frozen owned values,
$p_{\mathsf{blk},\tau}$ stores blocked owned values,
and $p_{\mathsf{val},\tau}$ stores non-owning values (e.g., references $\mathsf{Ref}^q(\tau)$, witnesses $\mathsf{Impl}(\tau,\mathsf{Tr})$, etc.).

\paragraph{Colors (token payload).}
% token 的“颜色”需要携带：
% (i) 变量身份(生成 harness 时用于命名)；
% (ii) 泛型实例化(类型替换)；
% (iii) 生命周期/借用标签(与栈绑定).
Let $\mathbb{V}$ be a countably infinite set of value identifiers (fresh Rust variable names),
and let $\mathbb{L}$ be a countably infinite set of lifetime labels (borrow-region identifiers).
Let $\Theta$ be the set of finite type substitutions $\theta:\mathsf{Var}\rightharpoonup \mathsf{Ty}_g$ mapping generic type variables to ground types.
Let $\Lambda$ be the set of finite lifetime substitutions $\lambda:\mathsf{LVar}\rightharpoonup \mathbb{L}$ mapping lifetime variables (e.g., \texttt{'a}) to region labels.

A token color is a triple
\[
c ~=~ \langle v, \theta, \lambda \rangle \in \mathcal{C} := \mathbb{V}\times\Theta\times\Lambda.
\]
Intuitively, $v$ names the runtime value in the generated harness,
while $(\theta,\lambda)$ record how generic parameters and lifetime parameters are instantiated for that value.

\paragraph{Markings as multisets.}
% 你给的 PN marking 是 $P\to\mathbb{N}$(无颜色),这里扩展为“每个 place 上是一个颜色 multiset”.
A \emph{colored marking} is a mapping
\[
M: P \to \mathbb{N}^{(\mathcal{C})},
\]
where $\mathbb{N}^{(\mathcal{C})}$ denotes finite multisets over $\mathcal{C}$.
We write $c\in M(p)$ if the multiplicity of $c$ at place $p$ is positive.
We still assume every arc has weight $1$, i.e., each firing consumes/produces at most one token per incident place.

\paragraph{Pushdown stack.}
% 栈用于建模“借用范围的嵌套 + 归还”.栈符号记录“谁借了谁、借用种类、绑定的 lifetime label”等.
Let $\Gamma$ be a stack alphabet with symbols of the form:
\[
\gamma ::= \mathsf{Freeze}(v) \mid \mathsf{Shr}(v_o,v_r,\ell) \mid \mathsf{Mut}(v_o,v_r,\ell),
\]
where $v_o$ is the owner value id, $v_r$ is the borrower value id (reference or borrow-carrying composite),
and $\ell\in\mathbb{L}$ is the lifetime label.
A configuration is a pair $\langle M, S\rangle$ where $M$ is a colored marking and $S\in \Gamma^*$ is a stack.

\paragraph{Transitions and firing semantics.}
% 每个 transition 除了 pre/post places,还带：
% (i) 颜色约束(guard)：用于统一泛型/生命周期替换并检查 trait 等；
% (ii) 输出构造函数：从输入 token 生成输出 token(含 fresh id)；
% (iii) 栈动作：push/pop(可为空).
A transition $t\in T$ is equipped with:
(i) a guard predicate $G_t$ over selected input colors and the current stack,
(ii) an output function $\mathsf{Out}_t$ that constructs produced colors (possibly allocating fresh $v$ or $\ell$),
and (iii) a stack action $\mathsf{Act}_t \in \{\mathsf{push}(\bar\gamma),~\mathsf{pop}(\bar\gamma),~\epsilon\}$,
where $\bar\gamma$ is a finite sequence of stack symbols.

We say $t$ is enabled at configuration $\langle M,S\rangle$, written $\langle M,S\rangle[t\rangle$, if:
\begin{enumerate}
  \item for every $p\in {}^{\bullet}t$ there exists a color $c_p$ with $c_p\in M(p)$ (choose one token per input place);
  \item $G_t(\{c_p\}_{p\in{}^{\bullet}t},S)$ holds; and
  \item if $\mathsf{Act}_t=\mathsf{pop}(\bar\gamma)$, then $\bar\gamma$ is a prefix of $S$ (top-matching, LIFO).
\end{enumerate}

Firing consumes one selected token from each $p\in {}^{\bullet}t$, produces one token into each $p\in t^{\bullet}$
with colors computed by $\mathsf{Out}_t$, and updates the stack according to $\mathsf{Act}_t$.
We write
\[
\langle M,S\rangle[t\rangle \langle M',S'\rangle.
\]
Reachability is defined as in Definition~\ref{def:pn}, now over configurations.

\subsection{Constructing a Rust PCPN}
\label{sec:method:construct}
% 本节给出“如何从 Rust 代码库构造网”：places 来自类型表达式；transitions 来自函数签名 + 结构性转换(借用、字段投影、trait/assoc).
Let $\Sigma(\mathcal{C})$ denote the extracted signature environment.
We build a PCPN
\[
\mathcal{N}_{\mathcal{C}} = (P,T,F,\mathcal{C},\Gamma,M_0)
\]
where $P$ is as defined above and $M_0$ seeds the net with a small set of primitive constructors
(e.g., integers, byte vectors, empty containers) as initial tokens.

\paragraph{Transitions from function signatures.}
% 每个函数/方法签名生成一个 transition.输入输出 place 由参数类型/返回类型确定；
% guard 检查泛型/trait/assoc/生命周期约束；Out 生成返回值 token(fresh 变量名).
For each item $f\in\mathcal{F}$ with signature
\[
\mathsf{sig}(f)=\langle \bar\alpha;~ \bar{\ellvar};~ \tau_1,\dots,\tau_n \to \tau_o;~ \rho \rangle,
\]
we create a transition $t_f$ with:
\[
{}^{\bullet}t_f=\{p_{\mathsf{val},\tau_1},\dots,p_{\mathsf{val},\tau_n}\},\quad
t_f^{\bullet}=\begin{cases}
\{p_{\mathsf{val},\tau_o}\} & \text{if }\tau_o\neq ()\\
\emptyset & \text{otherwise}
\end{cases}
\]
(extensions for tuples are discussed in Section~\ref{sec:method:composite}).

The guard $G_{t_f}$ enforces:
(i) consistent instantiation of type variables and lifetime variables across parameters,
(ii) satisfaction of trait bounds and associated-type constraints in $\rho$ (Section~\ref{sec:method:generics}),
and (iii) borrow-state feasibility (e.g., no use of a blocked owner).

The output function $\mathsf{Out}_{t_f}$ allocates a fresh value id $v_o$ for the return value (if any),
and produces a color $\langle v_o,\theta',\lambda'\rangle$ where $(\theta',\lambda')$ is the unified instantiation inferred from the inputs.

\paragraph{Structural transitions.}
% 除函数调用外,我们加入结构性 transition：
% 1) move/borrow/drop 等“语句级”资源变化；
% 2) 字段投影/重借用；
% 3) trait witness 与关联类型重写(类型层面的转换,辅助泛型/调用匹配).
In addition to $t_f$, we include:
\begin{itemize}
  \item \textbf{Ownership transitions} (move, drop, borrow) in Section~\ref{sec:method:ownership};
  \item \textbf{Lifetime/stack transitions} (ending borrows, reborrowing) in Section~\ref{sec:method:lifetime};
  \item \textbf{Composite transitions} (field projection and reborrowing) in Section~\ref{sec:method:composite};
  \item \textbf{Trait/associated-type transitions} in Section~\ref{sec:method:traits}.
\end{itemize}


\subsection{Mapping Rust Ownership to Token Flow}
\label{sec:method:ownership}
% move/borrow/mut/指针等所有权关系如何映射；
% 关键：token 在不同 capability place 间流动,模拟“可用/冻结/阻塞”.

\paragraph{Move (by-value use).}
% move 语义：被 move 的 owned token 被消耗；除非函数返回它,否则不会再出现.
For a by-value use of an owned value of type $\tau$, we model the move by consuming the token at $p_{\mathsf{own},\tau}$.
Formally, we add a transition $\mathsf{Move}_\tau$:
\[
{}^{\bullet}\mathsf{Move}_\tau=\{p_{\mathsf{own},\tau}\},\qquad
\mathsf{Move}_\tau^{\bullet}=\emptyset.
\]
Its firing corresponds to emitting a Rust statement that consumes $v$ (e.g., passing $v$ to a function by value, or assigning it elsewhere).
If an API call returns the moved value, that return is represented by $t_f$ producing a new token (fresh id) in the corresponding place.

\paragraph{Drop (end of scope).}
% drop 表示“显式结束一个值的生存期”；对携带借用的复合值,drop 也会触发借用归还(由栈规则保证).
For any place $p_{\kappa,\tau}$ we add a transition $\mathsf{Drop}_{\kappa,\tau}$ that removes one token from that place.
If the dropped value carries an active borrow, the corresponding borrow-ending transitions (Section~\ref{sec:method:lifetime})
must be fired (or will become fireable) to restore the owner.

\paragraph{Shared borrow from an owned value.}
% 从 owned 值创建 \&T：owner 进入 frozen 状态；产生一个 reference token；栈 push 两个符号：
% Freeze(owner)(表示 owner 从 own->frz)以及 Shr(owner,ref,lbl)(绑定生命周期).
We introduce a transition $\mathsf{BorrowShr}_\tau$ that creates a shared reference:
\[
{}^{\bullet}\mathsf{BorrowShr}_\tau=\{p_{\mathsf{own},\tau}\},\qquad
\mathsf{BorrowShr}_\tau^{\bullet}=\{p_{\mathsf{frz},\tau},~p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}(\tau)}\}.
\]
Let the input token color be $\langle v_o,\theta,\lambda\rangle$.
$\mathsf{Out}_{\mathsf{BorrowShr}_\tau}$ allocates a fresh reference id $v_r$ and a fresh lifetime label $\ell$,
and produces:
\[
\langle v_o,\theta,\lambda\rangle \text{ at } p_{\mathsf{frz},\tau},
\quad
\langle v_r,\theta,\lambda[\ellvar\mapsto \ell]\rangle \text{ at } p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}(\tau)},
\]
where $\ellvar$ is an internal lifetime variable for the created reference.
The stack action is:
\[
\mathsf{Act}_{\mathsf{BorrowShr}_\tau}=\mathsf{push}\big(\mathsf{Freeze}(v_o)\cdot \mathsf{Shr}(v_o,v_r,\ell)\big).
\]
This corresponds to Rust code \texttt{let v\_r = \&v\_o;}.

\paragraph{Mutable borrow from an owned value.}
% 从 owned 值创建 \&mut T：owner 进入 blocked；产生 \&mut token；栈 push Mut(owner,ref,lbl).
Similarly, mutable borrow is:
\[
{}^{\bullet}\mathsf{BorrowMut}_\tau=\{p_{\mathsf{own},\tau}\},\qquad
\mathsf{BorrowMut}_\tau^{\bullet}=\{p_{\mathsf{blk},\tau},~p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}(\tau)}\},
\]
with output producing a fresh $v_r$ and $\ell$, and
\[
\mathsf{Act}_{\mathsf{BorrowMut}_\tau}=\mathsf{push}\big(\mathsf{Mut}(v_o,v_r,\ell)\big),
\]
corresponding to Rust code \texttt{let v\_r = \&mut v\_o;}.

\paragraph{Pointers.}
% 指针类型(如 Box<T>, Rc<T>)在本模型中首先作为普通 owned 类型处理；
% 对其“解引用借用”可作为结构性转换(类似字段投影)加入 transition：&Box<T> -> &T 等.
Smart pointers (e.g., \texttt{Box<T>}, \texttt{Rc<T>}) are treated as regular owned types in $\mathsf{Ty}$.
When a pointer type supports deref coercions, we add structural transitions
(e.g., $p_{\mathsf{val},\mathsf{Ref}^{q}(\mathsf{Box}(\tau))}\to p_{\mathsf{val},\mathsf{Ref}^{q}(\tau)}$)
analogous to field projection (Section~\ref{sec:method:composite}).
Unsafe raw pointers are out of scope and can be treated as uninterpreted ground types if needed.

\subsection{Lifetimes via Stack Binding}
\label{sec:method:lifetime}
% 通过函数/借用展示生命周期；
% 核心：栈符号 Shr/Mut 把“owner token”和“borrower token”绑定到同一个 lifetime label；
% 结束生命周期 = pop 符号 + token 在 frz/blk/own 之间回流.

\paragraph{Ending a mutable borrow.}
% pop Mut(owner,ref,lbl) 时,需要同时消耗 ref token 和 blk(owner) token,
% 并把 owner 还原到 own(owner).
For each $\tau$, we add a transition $\mathsf{EndMut}_\tau$:
\[
{}^{\bullet}\mathsf{EndMut}_\tau=\{p_{\mathsf{blk},\tau},~p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}(\tau)}\},\qquad
\mathsf{EndMut}_\tau^{\bullet}=\{p_{\mathsf{own},\tau}\}.
\]
Its guard requires that the top of the stack matches $\mathsf{Mut}(v_o,v_r,\ell)$ and that the consumed colors
carry $v_o$ at $p_{\mathsf{blk},\tau}$ and $v_r$ at the reference place.
The stack action pops that symbol.
This corresponds to Rust code \texttt{drop(v\_r);} and then regaining access to \texttt{v\_o}.

\paragraph{Ending a shared borrow (nested accounting).}
% 共享借用允许多个并发别名.我们用“Freeze(owner)”作为外层标记：
% 第一次 shared borrow 时 push Freeze(owner) + Shr(owner,ref,lbl)；
% 后续 shared borrow 只 push Shr；
% 当 Shr 弹出后,如果栈顶是 Freeze(owner),说明这是最后一个 shared borrow,继续 pop Freeze 并把 owner frz->own.
We add two transitions:
\begin{align*}
{}^{\bullet}\mathsf{EndShrKeep}_\tau&=\{p_{\mathsf{frz},\tau},~p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}(\tau)}\},&
(\mathsf{EndShrKeep}_\tau)^{\bullet}&=\{p_{\mathsf{frz},\tau}\},\\
{}^{\bullet}\mathsf{EndShrLast}_\tau&=\{p_{\mathsf{frz},\tau},~p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}(\tau)}\},&
(\mathsf{EndShrLast}_\tau)^{\bullet}&=\{p_{\mathsf{own},\tau}\}.
\end{align*}
Both pop $\mathsf{Shr}(v_o,v_r,\ell)$.
Additionally, $\mathsf{EndShrLast}_\tau$ requires that after popping $\mathsf{Shr}(v_o,v_r,\ell)$,
the new stack top is $\mathsf{Freeze}(v_o)$; it then also pops $\mathsf{Freeze}(v_o)$.
This enforces a structured (LIFO) discipline for shared-borrow extents in generated harnesses.

\paragraph{Lifetime propagation through function calls.}
% 展示“两个 token 如何通过栈/标签绑定生命周期”的典型方式：
% 对签名里显式出现的 'a,我们用 lifetime substitution \lambda 将 'a 映射到某个栈标签 \ell；
% 共享 'a 的输入/输出 token 通过统一 \lambda 达成“同寿命”.
Consider a function signature
\[
f:\ \langle \bar\alpha;~ 'a;~ \mathsf{Ref}^{\mathsf{shr}}_{'a}(\tau)\to \mathsf{Ref}^{\mathsf{shr}}_{'a}(\tau')\rangle.
\]
In our net, $t_f$ consumes one token from $p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}(\tau)}$ and produces one token at
$p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}(\tau')}$.
The guard enforces that the input color $\langle v,\theta,\lambda\rangle$ has $\lambda('a)=\ell$,
and the output color reuses the same $\lambda$ so that the return reference is labeled by the same lifetime $\ell$.
No stack operation is performed by $t_f$; the lifetime is governed by the borrow-creating/ending transitions.

\paragraph{Borrow-carrying composite returns.}
% Rust 模式：函数/方法返回一个“携带借用”的 owned 值(例如 Tx<'s> 存储 &mut Store).
% 这时借用的范围由返回值本身决定：直到该值被 drop/consume.
If a call returns an owned value whose type contains lifetimes (e.g., $\tau_o = \mathsf{Tx}<'s>$),
$\mathsf{Out}_{t_f}$ produces an owned token whose lifetime substitution maps $'s$ to a fresh label $\ell$,
and $t_f$ pushes the corresponding $\mathsf{Mut}$ or $\mathsf{Shr}$ frame that blocks/freezes the owner.
A consuming call (e.g., \texttt{commit(self)}) ends the borrow by firing the corresponding end-borrow transition,
popping the frame and restoring the owner token.

\subsection{Generic Instantiation via a Partial Order}
\label{sec:method:generics}
% 泛型参数的偏序关系如何定义；
% 给出一个形式化的“满足 trait bound/关联类型约束”的判定,并用于 transition guard.

\paragraph{Type facts and requirements.}
% 把“类型满足哪些 trait、关联类型等于什么”视为一组 facts；
% 用集合包含关系形成偏序(poset).
Let $\mathsf{Fact}$ be a universe of type facts of the form:
\[
\mathsf{impl}(\tau,\mathsf{Tr})\quad\text{and}\quad
\mathsf{assoc}(\tau,\mathsf{Tr},A)=\tau'.
\]
We assume the crate environment provides a (possibly compiler-oracle-backed) predicate
$\mathsf{Entails}_{\Sigma(\mathcal{C})}(F)$ that returns whether a fact $F$ holds (e.g., a trait is implemented),
and a resolver that returns associated-type equalities when they are determined by impls.

Define the \emph{feature set} of a ground type $\tau$ as
\begin{equation}
\begin{split}
\mathsf{Feat}(\tau) &= \{\, \mathsf{impl}(\tau,\mathsf{Tr}) \mid \Sigma(\mathcal{C})\models \tau:\mathsf{Tr}\,\} \\
     &\cup  \{\, \mathsf{assoc}(\tau,\mathsf{Tr},A)=\tau' \mid \Sigma(\mathcal{C})\models \langle \tau \text{ as }\mathsf{Tr}\rangle::A=\tau' \,\} 
\end{split}
\end{equation}
	
A generic requirement $\rho$ (from where-clauses) is mapped to a set of required facts $\mathsf{Req}(\rho)\subseteq \mathsf{Fact}$.

\paragraph{Partial order.}
% 偏序定义：用 fact 集合的包含关系(子集)作为“信息强弱/约束强弱”；
% 一个具体类型能满足 requirement,当且仅当它的 Feat superset 覆盖 requirement.
We define a partial order on fact-sets by subset:
\[
X \preceq Y \iff X\subseteq Y.
\]
We then say a ground type $\tau$ \emph{satisfies} a requirement $\rho$, written $\tau \Vdash \rho$, iff
\[
\mathsf{Req}(\rho) ~\preceq~ \mathsf{Feat}(\tau).
\]
This yields a monotone, checkable criterion: the more features a type has, the more requirements it can satisfy.

\paragraph{Instantiation as guard solving.}
% 当 transition 对应的函数有泛型参数 \alpha 时,guard 需要找一个 substitution \theta,
% 使得每个 \alpha 的约束都被满足,并且多个参数之间的类型变量统一一致.
For a signature with generic variables $\bar\alpha$ and requirement $\rho$,
the guard $G_{t_f}$ is satisfiable if there exists a substitution $\theta$ such that:
\begin{enumerate}
  \item $\theta$ assigns a ground type to each $\alpha\in\bar\alpha$;
  \item for each bound $\alpha:\mathsf{Tr}$ in $\rho$, we have $\theta(\alpha)\Vdash (\alpha:\mathsf{Tr})$;
  \item for each associated-type constraint $\langle \alpha \text{ as }\mathsf{Tr}\rangle::A=\tau'$ in $\rho$,
        the resolver yields $\mathsf{assoc}(\theta(\alpha),\mathsf{Tr},A)=\theta(\tau')$ (after applying $\theta$);
  \item substitutions carried by input tokens are consistent with $\theta$ on shared variables.
\end{enumerate}
When multiple substitutions are possible, the generator may explore them in an order guided by $\preceq$
(e.g., prefer smaller $\mathsf{Feat}(\tau)$ first, or prefer types drawn from a finite candidate pool).

\subsection{Composite Types and Field Re-Borrowing}
\label{sec:method:composite}
% 复合类型如何重借用字段(field reborrow)；
% places 里有 Field(\tau,f)；transitions 表示投影与(重)借用；
% 同时用栈约束保证“父借用与子借用”的关系正确.

\paragraph{Field projection (move).}
% 从 owned 结构体中 move 出字段在 Rust 中可能导致 partial move；
% 为了保证生成代码可编译且模型可控,我们采用保守规则：move 字段时消耗整个 owner(不再使用剩余字段).
Let $\tau$ be a struct type with field $f:\tau_f$.
We add a transition $\mathsf{ProjMove}_{\tau,f}$:
\[
{}^{\bullet}\mathsf{ProjMove}_{\tau,f}=\{p_{\mathsf{own},\tau}\},\qquad
\mathsf{ProjMove}_{\tau,f}^{\bullet}=\{p_{\mathsf{own},\tau_f}\}.
\]
Its firing corresponds to Rust code \texttt{let y = x.f;} while treating \texttt{x} as consumed.

\paragraph{Field reborrow from a shared reference.}
% &S -> &S.f 不引入可变性,且允许别名；我们将其建模为“从 ref token 生成 ref token”,并保持父 ref 仍可用(用 self-loop 或线性都可).
We add $\mathsf{ProjShr}_{\tau,f}$:
\[
{}^{\bullet}\mathsf{ProjShr}_{\tau,f}=\{p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}(\tau)}\},\qquad
\mathsf{ProjShr}_{\tau,f}^{\bullet}=\{p_{\mathsf{val},\mathsf{Ref}^{\mathsf{shr}}(\tau_f)}\}.
\]
The output reuses the same lifetime substitution $\lambda$ (same region label), reflecting that
a shared reborrow can have the same or shorter lifetime; we conservatively reuse the same label.

\paragraph{Field reborrow from a mutable reference (reborrowing).}
% &mut S -> &mut S.f 是典型 reborrow：子借用存活期间父 &mut S 不能被使用；
% 用栈实现：push Mut(parent,child,lbl) 并把父 ref 暂时视为冻结/阻塞,直到 child drop.
We model mutable field reborrow with a transition $\mathsf{ProjMut}_{\tau,f}$:
\[
{}^{\bullet}\mathsf{ProjMut}_{\tau,f}=\{p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}(\tau)}\},\qquad
\mathsf{ProjMut}_{\tau,f}^{\bullet}=\{p_{\mathsf{val},\mathsf{Ref}^{\mathsf{mut}}(\tau_f)}\}.
\]
Let the input color be $\langle v_p,\theta,\lambda\rangle$.
We allocate a fresh child-ref id $v_c$ and label $\ell_c$, push $\mathsf{Mut}(v_p,v_c,\ell_c)$,
and produce the child reference token carrying $\ell_c$.
To ensure the parent reference is unusable during $\ell_c$, we optionally place the parent into a separate ``frozen-ref'' place;
equivalently, one can enforce linear use by consuming the parent and only restoring it when ending the child borrow.

\paragraph{Tuples and common algebraic data types.}
% tuple/Option/Result 等可以用“构造/析构 transition”处理；
% 这里给最简约接口(你后续可按需要扩展).
For tuples, we add projections $\mathsf{Proj}_i:(\tau_1,\dots,\tau_n)\to \tau_i$ and constructors.
For sum types (e.g., $\mathsf{Option}(\tau)$), we add constructors $\mathsf{Some}:\tau\to\mathsf{Option}(\tau)$ and $\mathsf{None}:\to\mathsf{Option}(\tau)$,
and optionally destructors that branch (nondeterministic transitions), if pattern-matching is desired in harness synthesis.

\subsection{Traits and Associated Types}
\label{sec:method:traits}
% traits/关联类型在你的设想里也要成为 places/transition；
% 这里给出两类：trait witness 生成；associated type 的“重写/转换”(辅助把投影类型对齐到具体类型).

\paragraph{Trait witness production.}
% 当我们拥有一个值(或一个类型实例)时,可以产生一个“实现 trait 的见证 token”,用于驱动后续关联类型解析或 trait 方法选择.
If $\Sigma(\mathcal{C})\models \tau:\mathsf{Tr}$, we include a transition $\mathsf{Impl}_{\tau,\mathsf{Tr}}$:
\[
{}^{\bullet}\mathsf{Impl}_{\tau,\mathsf{Tr}}=\{p_{\mathsf{own},\tau}\},\qquad
\mathsf{Impl}_{\tau,\mathsf{Tr}}^{\bullet}=\{p_{\mathsf{own},\tau},~p_{\mathsf{val},\mathsf{Impl}(\tau,\mathsf{Tr})}\}.
\]
The self-loop keeps the value; the produced witness token is purely logical (may correspond to type-ascription / method availability).

\paragraph{Associated type resolution as retyping.}
% 关联类型是类型层面的等式：<T as Tr>::A = U.
% 我们用一个“retag”式 transition 把 token 从投影类型 place 转到具体类型 place(值不变,仅改变其静态类型标签).
If the environment yields $\Sigma(\mathcal{C})\models \mathsf{Assoc}(\tau,\mathsf{Tr},A)=\tau'$, we add:
\[
{}^{\bullet}\mathsf{AssocCast}_{\tau,\mathsf{Tr},A}=\{p_{\mathsf{val},\mathsf{Assoc}(\tau,\mathsf{Tr},A)}\},\qquad
\mathsf{AssocCast}_{\tau,\mathsf{Tr},A}^{\bullet}=\{p_{\mathsf{val},\tau'}\}.
\]
The output preserves the same value id $v$ and substitutions $(\theta,\lambda)$, reflecting a type-level refinement.

\subsection{End-to-End Example Mapping}
\label{sec:method:example}
% 给一个小例子把上述规则串起来,突出：
% 1) move/borrow 的 token 流动；
% 2) 函数返回“携带借用”的值；
% 3) get 返回引用导致 tx 冻结,直到 drop.
% 复用 introduction 的 Store/Tx 例子,但只展示“映射思想”,不需要画图.

\begin{lstlisting}[language=Rust,caption={A Rust snippet whose compilation constraints are captured by PCPN token flow.},label={lst:method:demo}]
fn demo(bytes: &[u8]) {
    let mut store = Store::open("tmp").unwrap();
    let mut tx = store.begin();      // tx carries &mut store
    tx.put(bytes.to_vec(), bytes.to_vec());

    let r = tx.get(bytes.to_vec());  // returns Option<&[u8]>, freezes tx while r is live
    drop(r);                         // ends shared borrow; tx becomes mutable again

    tx.put(bytes.to_vec(), bytes.to_vec());
    tx.commit().unwrap();            // ends the &mut store borrow carried by tx
}
\end{lstlisting}

% 对该片段的 PCPN 解释：
% - begin: 消耗 p_{own,Store} 的 token,产生 p_{blk,Store} + p_{own,Tx<'s>},并 push Mut(store,tx,ell)；
% - put: 对 Tx token 做 self-loop(方法可反复调用),同时消耗输入 Vec<u8> token；
% - get: 将 Tx 从 own -> frz,并产生 Ref^{shr}([u8]) token,同时 push Freeze(tx)+Shr(tx,r,ell2)；
% - drop(r): pop Shr；若 last 则 pop Freeze 并把 Tx frz -> own；
% - commit(self): 消耗 Tx token,并与 blk(Store) 配对,pop Mut,恢复 Store 到 own.

\subsection{Soundness Intuition (Compilation-Safety)}
\label{sec:method:soundness}
% 避免做过强声明.我们给“直觉性正确性”：若按规则生成序列,则不会出现明显 borrow/move 编译错误.
By construction, any transition sequence in $\mathcal{N}_{\mathcal{C}}$ corresponds to a harness in which:
(i) moved owned values are not reused (they are consumed from $p_{\mathsf{own},\tau}$),
(ii) mutable aliasing is prevented (owners are moved to $p_{\mathsf{blk},\tau}$ under $\mathsf{Mut}$ frames),
(iii) mutation under shared aliasing is prevented (owners reside in $p_{\mathsf{frz},\tau}$ under $\mathsf{Freeze}/\mathsf{Shr}$ frames), and
(iv) lifetimes are properly nested and ended through stack-pop transitions.
Generic instantiation is guarded by the satisfiability relation $\Vdash$ and associated-type resolution.
Therefore, generated harnesses are expected to be accepted by the Rust compiler under this conservative, structured borrowing discipline.

\section{Formal Analysis for Sequence Generation and Coverage}
\label{sec:analysis}
% (i) 以严格的发生规则 + guard 约束定义“什么样的 firing 序列对应可生成的 Rust 代码序列”；
% (ii) 基于可达性分析(reachability analysis)构造可达图,并在此基础上定义覆盖率指标与覆盖保证.
% 本章讨论的是模型层面的可生成覆盖率(generability coverage),它衡量“我们能生成多少类可编译调用序列”；
% runtime 覆盖(basic-block/edge coverage)留在实验章节用 fuzzer/coverage 工具测量.

\subsection{Reachability Graph vs. Coverability Tree}
\label{sec:analysis:rg-vs-ct}
% - 可达图(reachability graph)：对 bounded net 给出精确可达状态空间,能抽取具体见证序列；
% - 覆盖树/可覆盖树(coverability tree, Karp--Miller)：为 unbounded net 给出过近似,节点可能出现 $\omega$,不对应具体 token 数；
% 我们的目标是“生成具体代码序列”,必须输出具体 firing trace,因此需要精确的可达性对象.
% 因此主分析采用(规范化后的)可达图；同时在构造可达图时维护一个见证树(witness tree)
% 以便高效抽取 root-to-node 的具体 trace 生成代码片段.见证树是可达图的一棵生成树(spanning tree),不是 Karp--Miller 覆盖树.

In Petri-net analysis, \emph{reachability graphs} provide an exact state-space representation for bounded systems,
while \emph{coverability trees} (e.g., Karp--Miller) provide an over-approximation for unbounded systems and may introduce $\omega$ markings.
Since our goal is to synthesize \emph{concrete} firing sequences that translate to \emph{concrete} Rust code snippets,
we base our analysis on a (canonical) reachability graph.
For trace extraction, we additionally maintain a \emph{witness tree} as a by-product of reachability saturation,
so that each discovered configuration comes with a unique root-to-node witness trace.

\subsection{Configurations and Firing Semantics}
\label{sec:analysis:semantics}
% 发生语义是本章核心：严格定义“启用/发生/后继配置”.
% 沿用 PN 结构($N=(P,T,F)$,arc weight=1),并扩展：
% - place 上是带颜色 token 的多重集；
% - 全局栈 $S$ 表示生命周期/借用范围；
% - guard + 实例化 $\iota$ 是启用条件的一部分；
% - 为代码生成,需要记录每次 firing 的“证据”(choice + instantiation),我们称为 certificate.

Let $\mathcal{N}_{\mathcal{C}}=(P,T,F,\mathcal{C},\Gamma,M_0)$ be the crate-specific net from Section~\ref{sec:method}.
A \emph{colored marking} is a mapping $M:P\rightarrow \mathbb{N}^{(\mathcal{C})}$ (finite multisets of colors).
A \emph{configuration} is a pair $\mathsf{Cfg}=\langle M,S\rangle$ where $S\in \Gamma^*$ is the stack.

For a transition $t\in T$, let ${}^{\bullet}t$ and $t^{\bullet}$ be its preset and postset (Definition~\ref{def:pn}).
Since all arc weights are $1$, one firing consumes/produces at most one token per incident place.

\paragraph{Token selection (choice).}
A \emph{choice} for $t$ at marking $M$ is a function
$\chi:{}^{\bullet}t\rightarrow \mathcal{C}$ such that $\chi(p)\in M(p)$ for every $p\in {}^{\bullet}t$.
We write $\chi\subseteq M$ to denote that $\chi picks$ one available token from each input place.

\paragraph{Instantiation and certificates.}
Each transition $t$ is equipped with a guard predicate $G_t$ and an output constructor $\mathsf{Out}_t$ (Section~\ref{sec:method:pcpn}).
The instantiation component $\iota$ records the unified substitutions needed by $G_t$ (e.g., type/lifetime substitutions, plus any resolved associated types).

\begin{definition}[Firing Certificate]
\label{def:cert}
A \emph{firing certificate} for $t$ at configuration $\langle M,S\rangle$ is a triple
$\kappa=\langle t,\chi,\iota\rangle$ such that:
(i) $\chi\subseteq M$, and (ii) $G_t(\chi,S,\iota)$ holds.
\end{definition}

\begin{definition}[Enabledness and Firing]
\label{def:firing}
Let $\mathsf{Cfg}=\langle M,S\rangle$ and $t\in T$.
We say $t$ is \emph{enabled} at $\mathsf{Cfg}$, written $\mathsf{Cfg}[t\rangle$, if there exists a certificate
$\kappa=\langle t,\chi,\iota\rangle$ (Definition~\ref{def:cert}).

Given a certificate $\kappa=\langle t,\chi,\iota\rangle$, firing $t$ yields $\mathsf{Cfg}'=\langle M',S'\rangle$, written
$\mathsf{Cfg}[\kappa\rangle \mathsf{Cfg}'$, where:
\begin{itemize}
  \item (\textbf{Consume}) For each $p\in {}^{\bullet}t$, remove one occurrence of $\chi(p)$ from $M(p)$.
  \item (\textbf{Produce}) Let $\mathsf{Out}_t(\chi,S,\iota)$ return a function $\pi:t^{\bullet}\rightarrow \mathcal{C}$ giving one produced color per output place.
        For each $q\in t^{\bullet}$, add one occurrence of $\pi(q)$ to $M(q)$.
  \item (\textbf{Stack}) Update $S$ to $S'$ according to $t$'s stack action (push/pop/$\epsilon$), provided the pop pattern matches the stack top.
\end{itemize}
All other places remain unchanged (as in Definition~\ref{def:pn}).
\end{definition}

% 到此“发生规则”严格完成：token 多重集更新 + 栈更新 + guard/certificate 作为启用前提.
% 下一小节把 guard 约束细化成可形式化求解的子系统.

\subsection{Guard System (Type/Lifetime/Trait/Capability Constraints)}
\label{sec:analysis:guards}
% guard 必须是“论文可读的形式化对象”,因此拆分为 4 类约束,并给统一求解(产生一组 instantiations).
% Trait/assoc satisfiability 复用第三章的偏序判定 $\Vdash$.

For each $t\in T$, we write the guard as a conjunction:
\[
G_t(\chi,S,\iota)\ \equiv\
G_t^{\mathsf{uni}}(\chi,\iota)\ \wedge\
G_t^{\mathsf{life}}(\chi,S,\iota)\ \wedge\
G_t^{\mathsf{trait}}(\chi,\iota)\ \wedge\
G_t^{\mathsf{cap}}(\chi,S,\iota).
\]

\paragraph{(1) Unification constraint $G_t^{\mathsf{uni}}$.}
It enforces consistent instantiation of generic type variables across all consumed tokens and signature-imposed equalities.
The instantiation $\iota$ contains a type substitution $\theta$; $G_t^{\mathsf{uni}}$ requires that:
(i) $\theta$ agrees with the substitutions carried inside chosen token colors, and
(ii) all signature equalities hold after applying $\theta$.

\paragraph{(2) Lifetime/stack constraint $G_t^{\mathsf{life}}$.}
It enforces lifetime substitution $\lambda$ (in $\iota$) consistency and stack compatibility:
(i) if a signature uses the same lifetime variable across multiple parameters, their chosen tokens must agree on $\lambda$, and
(ii) if $t$ pops frames, the required frame pattern must match the stack top (LIFO).

\paragraph{(3) Trait/associated-type constraint $G_t^{\mathsf{trait}}$.}
It checks all bounds in the where-clause $\rho$ of $t$ under $\theta$:
for each requirement $\rho_i$, require $\theta(\tau)\Vdash \rho_i$ and that associated-type equalities resolve consistently.

\paragraph{(4) Capability constraint $G_t^{\mathsf{cap}}$.}
It enforces Rust-compatibility of the consumed/produced places:
e.g., mutating transitions must consume from $\mathsf{own}$ (or from $\&mut$ reference places),
and borrow-creating transitions must place owners into $\mathsf{frz}$ or $\mathsf{blk}$ while the borrow is live.

\begin{algorithm}[t]
\caption{$\mathsf{SolveGuard}(t,\chi,S)$}
\label{alg:solve-guard}
% 返回所有可行 instantiations,从而得到所有可行 firing certificates.
% 有限性来自：bounded token/stack + 有限泛型候选集(下一节写).
\begin{algorithmic}[1]
\REQUIRE transition $t$, choice $\chi$, stack $S$
\ENSURE finite set of admissible instantiations $\mathsf{Inst}(t,\chi,S)$
\STATE $\Theta \gets \mathsf{UnifyTypes}(t,\chi)$ \hfill % 解类型方程,得到可能的 $\theta$
\STATE $\Lambda \gets \mathsf{UnifyLifetimes}(t,\chi,S)$ \hfill % 统一 lifetime,并检查 pop 与栈顶匹配
\STATE $\mathsf{Inst}\gets \emptyset$
\FORALL{$(\theta,\lambda)\in \Theta\times \Lambda$}
  \IF{$\mathsf{CheckTraitAssoc}(t,\theta)$ \AND $\mathsf{CheckCap}(t,\chi,S,\theta,\lambda)$}
     \STATE $\mathsf{Inst}\gets \mathsf{Inst}\cup\{(\theta,\lambda)\}$
  \ENDIF
\ENDFOR
\RETURN $\mathsf{Inst}$
\end{algorithmic}
\end{algorithm}

% 到这里 guard 系统已足够严格：它既是启用条件,也是后续可达图构造算法中的核心子过程.

\subsection{Canonical Reachability Graph and Witness Tree}
\label{sec:analysis:reachability}
% (1) 可达图(reachability graph)作为标准分析对象,给出精确“可生成空间”；
% (2) 见证树(witness tree)作为抽取具体 trace 的结构(注意：这不是 Karp--Miller 覆盖树).

\paragraph{Boundedness assumptions (for finiteness).}
% 为了得到“可证明的覆盖保证”,必须把分析域变成有限.
% 这与第二章/第三章设定一致：bounded Petri net + 有界栈深 + 有限泛型候选集 + 规范化重命名.
Assume:
(B1) the underlying net is $k$-bounded on token counts (Definition~\ref{def:pn});
(B2) stack depth is bounded by $D$;
(B3) each generic variable ranges over a finite candidate set $\mathsf{Cand}(\alpha)$, so $\mathsf{SolveGuard}$ returns a finite set;
(B4) fresh value ids/lifetime labels are quotiented by $\alpha$-renaming via a canonicalization function $\mathsf{Canon}(\cdot)$.

\begin{definition}[Annotated Canonical Reachability Graph]
\label{def:crg}
The (canonical) reachability graph is $\widehat{G}=(\widehat{V},\widehat{E})$ where:
$\widehat{V}$ is the set of canonical configurations reachable from $\mathsf{Canon}(\langle M_0,\epsilon\rangle)$, and
\[
(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\in \widehat{E}
\quad\text{iff}\quad
\exists \mathsf{Cfg},\mathsf{Cfg}':\ 
\mathsf{Canon}(\mathsf{Cfg})=\widehat{\mathsf{Cfg}},\
\mathsf{Canon}(\mathsf{Cfg}')=\widehat{\mathsf{Cfg}}',\
\mathsf{Cfg}[\kappa\rangle \mathsf{Cfg}'.
\]
\end{definition}

\begin{definition}[Witness Tree]
\label{def:wit-tree}
A witness tree $\mathcal{T}$ is a rooted tree over $\widehat{V}$ that stores, for each discovered node
$\widehat{\mathsf{Cfg}}\neq \widehat{\mathsf{Cfg}}_0$, one parent edge
$(\widehat{\mathsf{Par}},\kappa,\widehat{\mathsf{Cfg}})\in \widehat{E}$.
Thus each node has a unique root-to-node witness trace.
\end{definition}

\begin{algorithm}[t]
\caption{$\mathsf{SaturateWithWitness}(\mathcal{N}_{\mathcal{C}})$}
\label{alg:saturate-wit}
% 饱和(saturation)算法：计算完整 $\widehat{G}$,同时维护 witness tree 的 parent 指针.
\begin{algorithmic}[1]
\STATE $\widehat{V}\gets \emptyset,\ \widehat{E}\gets \emptyset$
\STATE $\mathsf{parent}(\cdot)\gets \bot$ \hfill % witness tree 的父边(首次发现时记录)
\STATE $\widehat{\mathsf{Cfg}}_0\gets \mathsf{Canon}(\langle M_0,\epsilon\rangle)$
\STATE $\mathsf{W}\gets \{\widehat{\mathsf{Cfg}}_0\}$
\WHILE{$\mathsf{W}\neq \emptyset$}
  \STATE remove some $\widehat{\mathsf{Cfg}}=\langle \widehat{M},\widehat{S}\rangle$ from $\mathsf{W}$
  \IF{$\widehat{\mathsf{Cfg}}\in \widehat{V}$} \textbf{continue} \ENDIF
  \STATE $\widehat{V}\gets \widehat{V}\cup\{\widehat{\mathsf{Cfg}}\}$
  \FORALL{$t\in T$}
    \FORALL{choices $\chi\subseteq \widehat{M}$ for ${}^{\bullet}t$}
      \FORALL{$\iota\in \mathsf{SolveGuard}(t,\chi,\widehat{S})$}
        \STATE $\kappa \gets \langle t,\chi,\iota\rangle$
        \STATE $\widehat{\mathsf{Cfg}}'\gets \mathsf{Canon}(\mathsf{Fire}(\widehat{\mathsf{Cfg}},\kappa))$
        \STATE $\widehat{E}\gets \widehat{E}\cup\{(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\}$
        \IF{$\widehat{\mathsf{Cfg}}'\notin \widehat{V}$ \AND $\mathsf{parent}(\widehat{\mathsf{Cfg}}')=\bot$}
          \STATE $\mathsf{parent}(\widehat{\mathsf{Cfg}}')\gets (\widehat{\mathsf{Cfg}},\kappa)$
        \ENDIF
        \STATE $\mathsf{W}\gets \mathsf{W}\cup\{\widehat{\mathsf{Cfg}}'\}$
      \ENDFOR
    \ENDFOR
  \ENDFOR
\ENDWHILE
\RETURN $(\widehat{V},\widehat{E},\mathsf{parent})$
\end{algorithmic}
\end{algorithm}

\begin{theorem}[Termination and Completeness]
\label{thm:term-comp}
Under (B1--B4), Algorithm~\ref{alg:saturate-wit} terminates and returns the complete canonical reachability graph $\widehat{G}$.
Moreover, the stored $\mathsf{parent}$ edges form a witness tree $\mathcal{T}$ spanning $\widehat{V}$.
\end{theorem}
\noindent\textbf{Proof sketch.}
(B1--B4) imply finitely many canonical configurations and finitely many admissible certificates per configuration.
Thus saturation terminates. Completeness follows from enumerating all enabled certificates. \hfill $\square$

% 到这里,“可达图 vs 覆盖树”的关系清楚了：
% - 我们用可达图 $\widehat{G}$ 做精确分析(覆盖保证基于它)；
% - 我们用 witness tree $\mathcal{T}$ 抽取具体 trace 生成代码(每个节点有唯一见证路径).

\subsection{Coverage Metrics for Generated \emph{Snippet Sets}}
\label{sec:analysis:coverage-metrics}
% 重要修正：论文里覆盖率通常针对“一组生成程序/片段”,而不是单条 trace.
% 因此我们定义 coverage over a set of traces $\Sigma$(每条 trace 生成一个代码片段).
% 另外,分母应使用“可达目标集合”,否则包含不可达 API 会导致覆盖率永远达不到 1.

Let $T_{\mathrm{API}}\subseteq T$ be transitions corresponding to crate functions/methods.
A \emph{trace} is a sequence of certificates $\sigma=\kappa_1\cdots \kappa_n$ such that
$\langle M_0,\epsilon\rangle[\sigma\rangle \langle M_n,S_n\rangle$.
Let $\mathrm{Emit}(\sigma)$ be the Rust snippet synthesized from $\sigma$ by the mapping in Section~\ref{sec:method}
(API transitions emit calls; structural transitions emit borrows/projections/drops).

\paragraph{Reachable objective sets.}
Define the reachable API-transition set:
\[
T_{\mathrm{API}}^{\mathrm{reach}}=\{\, t\in T_{\mathrm{API}}\mid \exists (\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\in \widehat{E}:\ \kappa.\mathsf{tr}=t \,\}.
\]
Similarly, for type-forms, let $P_{\mathrm{form}}\subseteq P$ be the places representing type forms (owned/ref/assoc/field, etc.), and define:
\[
P_{\mathrm{form}}^{\mathrm{reach}}=\{\, p\in P_{\mathrm{form}}\mid \exists \widehat{\mathsf{Cfg}}=\langle \widehat{M},\widehat{S}\rangle\in \widehat{V}:\ \widehat{M}(p)\neq \emptyset \,\}.
\]
For borrow-shapes, let $\mathcal{B}^{\mathrm{reach}}=\{\, \mathsf{Shape}(\widehat{S}) \mid \langle \widehat{M},\widehat{S}\rangle\in \widehat{V}\,\}$.

\paragraph{Coverage of a trace set.}
Let $\Sigma$ be a finite set of traces (equivalently, a set of generated snippets $\{\mathrm{Emit}(\sigma)\mid \sigma\in\Sigma\}$).
Define:
\[
\mathsf{API}(\Sigma)=\bigcup_{\sigma\in\Sigma}\{\, \kappa.\mathsf{tr}\mid \kappa\in\sigma,\ \kappa.\mathsf{tr}\in T_{\mathrm{API}}\,\},
\]
\[
\mathsf{Vis}_P(\Sigma)=\bigcup_{\sigma\in\Sigma}\bigcup_{i=0}^{|\sigma|}\{\, p\in P_{\mathrm{form}}\mid M_i(p)\neq \emptyset \,\},
\quad
\mathsf{Vis}_B(\Sigma)=\bigcup_{\sigma\in\Sigma}\bigcup_{i=0}^{|\sigma|}\{\mathsf{Shape}(S_i)\}.
\]
Then the three coverage ratios are:
\[
\mathrm{Cov}_{\mathrm{API}}(\Sigma)=\frac{|\mathsf{API}(\Sigma)|}{|T_{\mathrm{API}}^{\mathrm{reach}}|},
\quad
\mathrm{Cov}_{\mathrm{Type}}(\Sigma)=\frac{|\mathsf{Vis}_P(\Sigma)\cap P_{\mathrm{form}}^{\mathrm{reach}}|}{|P_{\mathrm{form}}^{\mathrm{reach}}|},
\quad
\mathrm{Cov}_{\mathrm{Borrow}}(\Sigma)=\frac{|\mathsf{Vis}_B(\Sigma)|}{|\mathcal{B}^{\mathrm{reach}}|}.
\]
We combine them as:
\[
\mathrm{GCov}(\Sigma)=w_1\cdot \mathrm{Cov}_{\mathrm{API}}(\Sigma)+w_2\cdot \mathrm{Cov}_{\mathrm{Type}}(\Sigma)+w_3\cdot \mathrm{Cov}_{\mathrm{Borrow}}(\Sigma),
\quad w_i\ge 0.
\]

% 这样定义的好处：
% - 分母是“可达目标”,因此覆盖率 1 是可达成的；
% - 以 trace set 为对象,符合“生成多个 harness/snippet 来覆盖更多 API”的现实与论文表述习惯.

\subsection{Coverage-Guaranteed Snippet Generation from $\widehat{G}$}
\label{sec:analysis:guarantee}
% 本节给出“如何保证覆盖率”的学术化结论：
% - 先饱和得到 $\widehat{G}$；
% - 再为每个可达目标(API transition / type form / borrow shape)构造一个 witness trace；
% - 这些 witness traces 生成的 snippet 集合 $\Sigma$ 即保证覆盖率=1(对可达目标).

\paragraph{Witness extraction.}
Using the witness-tree parent map, each reachable node $\widehat{\mathsf{Cfg}}$ has a unique witness trace
$\mathsf{WitNode}(\widehat{\mathsf{Cfg}})$ obtained by backtracking $\mathsf{parent}$ pointers to the root.
For each edge $(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')\in \widehat{E}$, define a witness trace:
\[
\mathsf{WitEdge}(\widehat{\mathsf{Cfg}},\kappa,\widehat{\mathsf{Cfg}}')=
\mathsf{WitNode}(\widehat{\mathsf{Cfg}})\cdot \kappa.
\]
% 这条 trace 对应一段可生成代码序列：先到达 source 配置,再执行该 edge 的那一步调用/转换.

\begin{theorem}[Full Model-Level Coverage within Bounds]
\label{thm:full-coverage}
Let $(\widehat{V},\widehat{E},\mathsf{parent})$ be the result of Algorithm~\ref{alg:saturate-wit}.
Define the snippet set
\[
\Sigma_{\mathrm{API}}=\{\ \mathsf{WitEdge}(e)\mid e\in \widehat{E},\ e.\kappa.\mathsf{tr}\in T_{\mathrm{API}}\ \}.
\]
Then $\mathrm{Cov}_{\mathrm{API}}(\Sigma_{\mathrm{API}})=1$.
Moreover, by adding witnesses for first occurrences of each $p\in P_{\mathrm{form}}^{\mathrm{reach}}$
and each $b\in \mathcal{B}^{\mathrm{reach}}$, we obtain a finite set $\Sigma^\star$ such that
\[
\mathrm{Cov}_{\mathrm{API}}(\Sigma^\star)=\mathrm{Cov}_{\mathrm{Type}}(\Sigma^\star)=\mathrm{Cov}_{\mathrm{Borrow}}(\Sigma^\star)=1.
\]
\end{theorem}
\noindent\textbf{Proof sketch.}
For any $t\in T_{\mathrm{API}}^{\mathrm{reach}}$, by definition there exists an edge $e\in \widehat{E}$ whose certificate transition is $t$.
$\mathsf{WitEdge}(e)$ contains $t$, hence the union over $\Sigma_{\mathrm{API}}$ covers all reachable API transitions.
The place/shape statements follow by choosing, for each reachable objective, a node/edge where it appears and taking its witness. \hfill $\square$

% 上面给的是“保证覆盖率=1(对可达目标)”的构造性结论.
% 实际生成时你通常不想为每条边都生成一个 snippet(太多).
% 因此下一段给一个标准的“少量 snippet 达到高覆盖”的最大覆盖抽取(可选写).

\subsection{Budgeted High-Coverage Selection (Optional)}
\label{sec:analysis:budgeted}
% 把“从很多 witness 里选少量 snippet”形式化成 maximum coverage / set cover,并给贪心算法与近似保证.

Let $\mathcal{S}$ be a pool of candidate witness traces (e.g., a subset of $\{\mathsf{WitEdge}(e)\mid e\in\widehat{E}\}$).
Each trace $\sigma$ covers objectives:
\[
\mathsf{Obj}(\sigma)=
\big(\mathsf{API}(\{\sigma\})\big)\ \cup\
\big(\mathsf{Vis}_P(\{\sigma\})\cap P_{\mathrm{form}}^{\mathrm{reach}}\big)\ \cup\
\mathsf{Vis}_B(\{\sigma\}).
\]
Given a budget $m$, select $\Sigma\subseteq \mathcal{S}$ with $|\Sigma|\le m$ maximizing $\left|\bigcup_{\sigma\in \Sigma}\mathsf{Obj}(\sigma)\right|$.

\begin{algorithm}[t]
\caption{$\mathsf{GreedyCover}(\mathcal{S},m)$}
\label{alg:greedycover}
% 标准贪心最大覆盖；若采用统一权重,则有 $(1-1/e)$ 近似保证(可作为论文的理论点).
\begin{algorithmic}[1]
\STATE $\Sigma\gets \emptyset,\ \mathsf{U}\gets$ all objectives initially uncovered
\FOR{$i=1$ to $m$}
  \STATE choose $\sigma^\star\in \mathcal{S}\setminus \Sigma$ maximizing $|\mathsf{Obj}(\sigma^\star)\cap \mathsf{U}|$
  \STATE $\Sigma\gets \Sigma\cup\{\sigma^\star\}$
  \STATE $\mathsf{U}\gets \mathsf{U}\setminus \mathsf{Obj}(\sigma^\star)$
\ENDFOR
\RETURN $\Sigma$
\end{algorithmic}
\end{algorithm}

\section{Evaluation}
\label{sec:evaluation}
% (1) 我们的 PCPN 生成器确实能生成“可编译/可执行”的 Rust harness(Rust 约束被满足)；
% (2) 生成的 harness 能在真实 crate 上带来更高的覆盖率(更有效的测试)；
% (3) PCPN 中关键设计(栈生命周期、泛型偏序、字段(重)借用等)对效果是必要的；
% (4) 方法具有可扩展性(能在真实代码规模上运行)；
% 并且(可选)讨论“模型覆盖率”与“代码覆盖率”的相关性,从而把 Petri 网层面的覆盖最大化变成学术论证闭环.

\subsection{Research Questions}
\label{sec:evaluation:rqs}
% RQs 要直接对应论文贡献点(PCPN + Rust 映射 + 生成 harness + 覆盖率提升 + 特性支持).
% 建议 4--6 个问题,避免过散.
We structure the evaluation around the following research questions:

\begin{description}
  \item[\textbf{RQ1 (Compilability \& Executability).}]
  % 证明“生成的 Rust 代码能过编译/能跑”,这是 Rust 场景最大的门槛.
  Does our PCPN-guided generator produce compilable and executable harnesses at a higher rate than baselines that do not model
  ownership/borrowing/lifetimes/generics precisely?

  \item[\textbf{RQ2 (Target Code Coverage).}]
  % 核心有效性：在相同 fuzzing 预算下,目标 crate 的覆盖率是否更高.
  Under the same fuzzing budget, do PCPN-generated harnesses achieve higher \emph{target-library} coverage
  (line/branch/function coverage of the crate-under-test) than competing approaches?

  \item[\textbf{RQ3 (Bug-Finding Effectiveness).}]
  % 覆盖率是手段,最终是发现 bug(panic、crash、UB、logic error).
  Does our approach find more unique failures (e.g., panics, assertion violations, sanitizer crashes) and/or find them faster
  (time-to-first-crash) than baselines?

  \item[\textbf{RQ4 (Feature Support and Stress Cases).}]
  % 点题“我们支持 Rust 特性”：复合类型、借用/生命周期、泛型、trait/关联类型、字段重借用.
  How robust is the approach on crates that heavily use (i) borrow-carrying types and explicit lifetimes in signatures,
  (ii) generic APIs with trait bounds and associated types, and (iii) composite types requiring field projection/reborrowing?

  \item[\textbf{RQ5 (Ablation: Which Components Matter?).}]
  % 证明“不是随便一个搜索/生成就行”,而是 PCPN 的关键部件带来收益.
  What is the contribution of each major component (pushdown lifetime stack, generic partial-order instantiation, field (re)borrowing rules,
  POR/heuristics if any) to compilability and coverage?

  \item[\textbf{RQ6 (Model Coverage $\rightarrow$ Code Coverage).}]
  % 学术闭环：我们在 Petri 网层面追求 transition coverage/marking diversity,这是否与目标代码覆盖相关？
  To what extent does increasing \emph{net-level coverage} (e.g., transition coverage over API transitions) correlate with
  increased \emph{target-code} coverage, and can it serve as a reliable surrogate objective?
\end{description}

\subsection{Benchmarks and Subjects}
\label{sec:evaluation:benchmarks}
% 基准集要“可解释 + 可复现”.建议两类：真实 crates + 人工微基准(覆盖特性边界).
\paragraph{Real-world crates.}
We evaluate on a curated set of Rust crates drawn from crates.io.
We select crates that (i) expose non-trivial public APIs (multiple modules/types),
(ii) include stateful protocols (resources/handles/transactions/builders), and
(iii) collectively cover diverse domains (parsers, codecs, data structures, storage, networking, etc.).
% 这里给出选择标准即可,具体名单放表格；不要在正文里硬编百分比.
For each crate, we extract callable items $\mathcal{F}$ (public functions/methods) and their signatures via the compiler front-end.

\paragraph{Feature-stratified subsets.}
% 为了回答 RQ4,建议把 crate 按特性分组(用静态特征统计)：
We partition crates into groups based on signature-level features:
\begin{itemize}
  \item \textbf{Lifetime-heavy}: explicit lifetimes and borrow-carrying return types in signatures;
  \item \textbf{Generic-heavy}: frequent type parameters with trait bounds / where-clauses / associated types;
  \item \textbf{Composite-heavy}: deep nested ADTs and frequent field projections (struct/tuple) in APIs.
\end{itemize}

\paragraph{Microbenchmarks (optional but recommended).}
% 用小程序专门“单点打击”每个 Rust 约束(move, &/&mut, reborrow, assoc type).
We additionally design a suite of synthetic microbenchmarks, each isolating one feature:
\emph{move-only}, \emph{shared-borrow}, \emph{mutable-borrow}, \emph{borrow-carrying return},
\emph{generic bound}, \emph{associated type constraint}, and \emph{field reborrowing}.
This suite enables controlled ablations and avoids confounding factors from large codebases.

\subsection{Baselines}
\label{sec:evaluation:baselines}
% 基线要能回答“没有我们这个模型会怎样”.建议至少 3 个层次：
% (B0) 传统 fuzz(字节输入 + 手写/半手写 harness)；
% (B1) 只用签名做 typed 生成但忽略 borrow/lifetime(常见失败点)；
% (B2) Petri 网但不含 pushdown/不含泛型求解(对齐你论文讨论).
We compare against the following baselines:

\begin{description}
  \item[\textbf{B0: Byte-oriented fuzzing harness}.]
  A conventional setup where the fuzzer mutates a byte array and a harness decodes it into inputs for a fixed entry point.
  % 如果没有统一 harness,可在每个 crate 选择一个典型入口函数；或用社区已有 harness(如果有).

  \item[\textbf{B1: Signature-driven random sequences (no borrow/lifetime model).}]
  A type-directed generator that samples call sequences using signatures, but treats values as freely duplicable or ends borrows heuristically
  without a formal stack discipline.
  % 这是最能体现 Rust 约束“为什么需要 PCPN”的对照组.

  \item[\textbf{B2: Petri net without pushdown (finite-state approximation).}]
  A Petri-net-based generator that uses places/transitions derived from signatures but replaces the pushdown lifetime mechanism with a finite-state
  approximation (e.g., lexical-only or bounded non-nested borrow tracking).
  % 用于证明“pushdown 栈”的价值(RQ5).

  \item[\textbf{B3 (optional): Property-based testing}.]
  A property-based generator (e.g., arbitrary structured values) for selected APIs, without stateful call-sequence generation.
  % 可选,主要对比“有结构输入但无序列协议”的情况.
\end{description}

\subsection{Metrics}
\label{sec:evaluation:metrics}
% 指标必须能一一对应 RQ.建议至少：编译率、可执行率、目标覆盖、模型覆盖、崩溃数、时间指标、成本指标.
\paragraph{Compilability and executability.}
Let $\mathcal{H}$ be the set of synthesized harnesses.
We define:
\begin{align}
\textbf{CHR} &= \frac{|\{h\in\mathcal{H} \mid \texttt{rustc}(h)~\text{accepts}\}|}{|\mathcal{H}|},
\label{eq:chr}\\
\textbf{EHR} &= \frac{|\{h\in\mathcal{H} \mid h~\text{executes and completes initialization}\}|}{|\mathcal{H}|}.
\label{eq:ehr}
\end{align}
% CHR=编译通过率；EHR=能跑起来(至少跑过初始化/跑若干轮 fuzz iteration)的比例.

\paragraph{Target-library coverage.}
% 这是论文核心“有效性”指标：覆盖率针对目标 crate,而不是 harness 本身.
For each crate-under-test, we measure standard code coverage (e.g., line and branch coverage):
\begin{align}
\textbf{TLC} &= \frac{\text{covered lines in target crate}}{\text{total instrumented lines in target crate}},
\label{eq:tlc}\\
\textbf{TBC} &= \frac{\text{covered branches in target crate}}{\text{total instrumented branches in target crate}}.
\label{eq:tbc}
\end{align}

\paragraph{Model (net-level) coverage.}
% 把 Petri 网的 coverage 作为“生成过程的学术指标”,并用于 RQ6 的相关性分析.
Let $T_{\mathsf{api}}\subseteq T$ be transitions corresponding to extracted API calls.
For a trace multiset $\Sigma$ (generated during exploration/fuzzing), define:
\begin{align}
\textbf{NTC} &= \frac{|\{t\in T_{\mathsf{api}} \mid \exists \sigma\in\Sigma: t\in\sigma\}|}{|T_{\mathsf{api}}|}.
\label{eq:ntc}
\end{align}
% NTC=网层面 API transition 覆盖率(至少被触发一次的比例).

\paragraph{Bug-finding.}
% 用“唯一崩溃数 + 首次崩溃时间”最常见.
We record:
(i) the number of unique failures (deduplicated by stack trace hash or sanitizer report signature),
and (ii) time-to-first-crash (TTFC).

\paragraph{Cost and scalability.}
We measure:
net construction time, exploration time per trace, harness compilation time, and fuzzing throughput (exec/s).

\subsection{Experimental Protocol}
\label{sec:evaluation:protocol}
% 协议要保证“公平比较”,典型做法：固定 budget,重复多次取均值/方差.
For each crate and each approach (ours and baselines), we perform the following steps:

\begin{enumerate}
  \item \textbf{Model extraction (ours / B2).} Extract signatures $\mathcal{F}$ and build the Petri-net instance.
  \item \textbf{Harness generation.} Generate $K$ harnesses (or $K$ traces), each bounded by max trace length $L$ and stack depth $D$.
  \item \textbf{Compilation filter.} Compile each harness; record CHR and compile-time.
  \item \textbf{Fuzzing run.} Run each compiled harness for a fixed time budget $B_{\mathsf{time}}$ (or fixed total executions),
        using the same fuzzer backend and identical resource limits.
  \item \textbf{Measurement.} Collect target coverage (TLC/TBC), net-level coverage (NTC), and failures.
\end{enumerate}

\paragraph{Repetitions and statistics.}
Each configuration is repeated $R$ times with independent random seeds.
We report mean and standard deviation (or median and IQR) and use non-parametric tests when appropriate.
% 这里可写 “Wilcoxon/Mann-Whitney” 之类,但如果你不想承诺统计检验,也可以只写置信区间.

\subsection{Ablation and Sensitivity Studies}
\label{sec:evaluation:ablation}
% 为回答 RQ5/RQ4,做消融与敏感性：
\paragraph{Ablations.}
We evaluate the following variants:
\begin{itemize}
  \item \textbf{NoStack}: remove the pushdown lifetime stack and replace with a finite approximation;
  \item \textbf{NoPO}: replace the partial-order-based generic instantiation with random instantiation from the same candidate pool;
  \item \textbf{NoFieldReborrow}: disable field projection/reborrow transitions;
  \item \textbf{NoAssoc}: disable associated-type rewriting transitions.
\end{itemize}
We compare each variant against the full system in CHR/EHR, TLC/TBC, and TTFC.

\paragraph{Sensitivity.}
We vary $L$ (trace length bound), $D$ (stack depth bound), and $|\mathsf{Cand}(\alpha)|$ (generic candidate pool size)
to quantify trade-offs between exploration power, compilation success, and runtime coverage.
