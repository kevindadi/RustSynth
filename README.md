# SyPetype - Pushdown CPN Safe Rust Synthesizer

[中文](#中文说明) | [English](#english)

---

## 中文说明

SyPetype 是一个基于 **Pushdown Colored Petri Net (Pushdown CPN)** 的 Safe Rust 代码片段合成器。它从 rustdoc JSON 文档中解析公开 API 签名，构建 Pushdown CPN 模型，通过有界可达性搜索找到可触发序列，最终输出可编译的 Safe Rust 代码。

### 核心特性

- **9-Place 模型**：对每个基础类型 T，区分 {T, &T, &mut T} × {own, frz, blk} = 9 个 place
- **Pushdown 栈**：记录 outstanding borrows，实现 LIFO 借用/归还语义
- **类型统一**：支持泛型函数的 unification 和 completion
- **规范化**：状态空间 canonicalization 避免无限状态爆炸
- **0-ary Producer**：自动识别无参数的 const fn 作为值源

### 安装

```bash
# 需要 Rust nightly 工具链（用于生成 rustdoc JSON）
rustup toolchain install nightly

# 克隆并构建
git clone https://github.com/your-repo/sypetype.git
cd sypetype
cargo build --release
```

### 快速开始

#### 1. 生成 rustdoc JSON

```bash
cd examples/toy_api
cargo +nightly rustdoc -Z unstable-options --output-format json --lib
# 输出文件：target/doc/toy_api.json
```

#### 2. 运行合成器

```bash
cd ../..
cargo run --release -- synth \
    --doc-json examples/toy_api/target/doc/toy_api.json \
    --task examples/toy_api/task.toml \
    --out synthesized.rs
```

#### 3. 验证生成的代码

```bash
# 检查生成的代码是否可编译
rustc --edition 2021 synthesized.rs --crate-type lib
```

### 任务配置文件

任务配置使用 TOML 格式：

```toml
# task.toml
[inputs]
doc_json = "target/doc/toy_api.json"

[search]
stack_depth = 8           # 最大栈深度
default_place_bound = 2   # 每个 place 的默认 token 上界
max_steps = 100           # 最大搜索步数

[search.place_bounds]
"own_i32" = 3             # 覆盖特定 place 的上界

[filter]
allow = [                 # 只允许使用这些函数
  "make_counter",
  "Counter::new",
  "Counter::inc",
  "Counter::get",
]

[goal]
want = "own i32"          # 目标：获得一个 owned i32
count = 1
```

### 示例输出

```rust
//! Generated by SyPetype PCPN Synthesizer

fn main() {
    let counter_0 = make_counter(); // const_Counter
    let mut_counter_1 = &mut counter_0; // borrow_mut(Counter)
    counter_0.inc(); // Counter::inc
    drop(mut_counter_1); // end_mut(Counter)
    let i32_2 = counter_0.get(); // Counter::get
}
```

## English

SyPetype is a **Pushdown Colored Petri Net (Pushdown CPN)** based Safe Rust code snippet synthesizer. It parses public API signatures from rustdoc JSON, builds a Pushdown CPN model, searches for valid firing sequences via bounded reachability, and outputs compilable Safe Rust code.

### Key Features

- **9-Place Model**: For each base type T, distinguishes {T, &T, &mut T} × {own, frz, blk} = 9 places
- **Pushdown Stack**: Tracks outstanding borrows with LIFO borrow/return semantics
- **Type Unification**: Supports generic function unification and completion
- **Canonicalization**: State space canonicalization to prevent infinite state explosion
- **0-ary Producer**: Automatically identifies parameterless const fn as value sources

## Architecture

```
rustdoc JSON
     │
     ▼
┌─────────────────┐
│  API Graph      │  (Bipartite: Functions ↔ Types)
│  extract.rs     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  PCPN Model     │  (9 places per type, Pushdown stack)
│  pcpn.rs        │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Simulator      │  (BFS + Canonicalization)
│  simulator.rs   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Code Emitter   │  (Witness → Rust code)
│  emitter.rs     │
└─────────────────┘
```

## Module Overview

| Module | Description |
|--------|-------------|
| `types.rs` | 9-place type definitions (TypeForm, Capability, Token, etc.) |
| `config.rs` | Task configuration parsing (TOML) |
| `unify.rs` | Type unification and completion |
| `pcpn.rs` | Pushdown CPN model construction |
| `simulator.rs` | Bounded reachability search with canonicalization |
| `emitter.rs` | Witness to Rust code translation |
| `extract.rs` | rustdoc JSON → API Graph |

## Supported Rust Constructs

- Primitive types (i32, u64, bool, etc.)
- User-defined structs
- Generics with bounds (Copy, Clone)
- Shared references (&T)
- Mutable references (&mut T)
- Methods (self, &self, &mut self)
- Free functions
- const fn (as 0-ary producers)

## Limitations

- No lifetime elision inference (uses heuristics)
- No associated types / trait impl analysis
- No async/await support
- Simplified outlives checking via stack order
